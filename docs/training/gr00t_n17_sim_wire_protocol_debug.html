GR00T-N1.7 自训 ckpt × LeIsaac sim eval —— 4 层 wire 协议根因复盘

Root-cause: 4-layer wire protocol mismatch between self-trained GR00T-N1.7 server (--use-sim-policy-wrapper) and LeIsaac N1.5 client.

日期：2026-05-23 · 模型：自训 checkpoint-1200 (cold-start Cosmos-Reason2-2B + Gr00tN1d7 DiT)

TL;DR ckpt 健康 (2.2B params / 0 NaN)，直接 zmq probe 出 action range -28°~+21° 看似合理 —— 但 sim 机械臂完全不动。表象一个，根因 四层 wire 协议错配，缠了大半天才剥完。最后 client 跑通：arm 会动 (oranges 0/3 是因为 1200 step ≪ hi-space 6000 step 收敛点)。

Four stacked bugs in client <-> server payload schema: missing observation envelope, missing T axis + wrong dtype, msgpack-numpy bytes-key decode skipped, mnp.decode requires bytes keys not str.

1. 症状与初步证据 — Symptom & first evidence

训练 1200/6000 step (≈ 20% 进度) 拉本机做 sim eval：

  • ckpt 文件健康：Gr00tN1d7.from_pretrained() 加载 OK，2.2B params，0 NaN/Inf，698 weight keys 完整 match model architecture
  • 用 dummy obs 直接 zmq probe server：出非零 action，arm shape (1, 40, 5) range -28°~+21°，gripper (1, 40, 1) range 24~32 —— 看起来完全合理
  • 但跑 LeIsaac sim：机械臂完全不动，eval 报 KeyError traceback

第一直觉以为是训练量不足学不会任务，但 probe 出的 action 数值在 SO-101 关节合理范围，且服务端 NaN-free —— 模型实际在出动作，问题在 client 没把动作正确送到 sim env。

3. 四层根因 — Four stacked bugs

Bug 1 data envelope 缺 observation 包装

Server 启动用 --use-sim-policy-wrapper 时，Gr00tSimPolicyWrapper.get_action(observation, options=None) 走的是 kwargs 解包 路径。server 框架收到 zmq 请求后等价于：

policy.get_action(**data)   # data 是 request["data"]

LeIsaac 原版 N1.5 client 默认发：

request = {"endpoint": "get_action", "data": flat_obs}

其中 flat_obs 含 video.front, state.single_arm 等键。Python 把 video.front 当 kwarg 名 —— 不是合法 identifier：

"BasePolicy.get_action() got an unexpected keyword argument 'video.front'"

N1.5 raw server 直接接 flat obs 是合法的，但启用 sim_policy_wrapper 后必须多一层 envelope。

Bug 2 obs shape 缺 T 轴 + dtype 错

包了 observation 后，Gr00tSimPolicyWrapper.check_observation 强校验：

  • video: (B, T, H, W, 3) uint8 5D —— LeIsaac 给的是 (B, H, W, 3) 4D 缺 T 轴
  • state: (B, T, D) float32 3D —— LeIsaac 给的是 (B, D) 2D 且 dtype 是 float64
  • T = len(modality_configs["video"].delta_indices)，SO-101 PickOrange 配置下 T = 1

修复：obs 内每个 video/state ndarray 用 v[:, None, ...] 在 batch 后插入 T=1 轴，state 强转 float32。

Bug 3 msgpack-numpy bytes-key 漏解码

N1.7 server 返回 action chunk 时，每个 ndarray 用 msgpack_numpy 标准编码：

{b'nd': True, b'type': '<f4', b'kind': b'', b'shape': [1,40,5], b'data': <bytes>}

注意 keys 是 bytes。LeIsaac 的 MsgSerializer._decode_custom 当时只查 obj.get("nd")（str key），bytes key 直接漏过：

if obj.get("nd") is True:   # ✗ 不命中 bytes key
    return mnp.decode(obj)
return obj                  # ← 原 dict 透传给 client

结果 client 拿到的 action_chunk[0]["action.single_arm"] 是 嵌套 dict 而不是 ndarray，下一步 arm.ndim 触发 AttributeError: 'dict' object has no attribute 'ndim'。

Bug 4 反向修 key 又踩坑 —— mnp.decode 要 bytes keys

诊断 Bug 3 时我直觉地把 bytes keys 转 str 再喂给 mnp.decode：

x = {k.decode(): v for k, v in x.items()}   # ✗ 反方向了
return _mnp.decode(x)                        # 返回 dict 不是 ndarray

实测 mnp.decode 内部 hard-coded 用 bytes keys 查 b'nd'。正确做法是 保留 bytes keys，或反向把 str → bytes。验证：

>>> mnp.decode({b'nd':True, b'data':..., b'type':'<f4', b'shape':[1,40,5]})
ndarray (1, 40, 5)                        # ✓
>>> mnp.decode({'nd':True, 'data':..., 'type':'<f4', 'shape':[1,40,5]})
dict (returned unchanged)                  # ✗

4. 修复 patch — Minimal fix

所有改动集中在 LeIsaac/source/leisaac/leisaac/policy/service_policy_clients.py 的 Gr00tServicePolicyClient.get_action：

def get_action(self, observation_dict):
    # 1) flat obs with proper shape
    obs_dict = {f"video.{k}": observation_dict[k].cpu().numpy().astype(np.uint8)
                for k in self.camera_keys}
    joint_pos = convert_leisaac_action_to_lerobot(observation_dict["joint_pos"])
    obs_dict["state.single_arm"] = joint_pos[:, 0:5].astype(np.float32)
    obs_dict["state.gripper"]    = joint_pos[:, 5:6].astype(np.float32)
    obs_dict["annotation.human.task_description"] = [observation_dict["task_description"]]

    # 2) 加 T=1 + 强制 float32 / uint8（wrapper.check_observation 要求）
    wrapped_obs = {}
    for k, v in obs_dict.items():
        if isinstance(v, np.ndarray):
            if k.startswith("video.") and v.ndim == 4:
                wrapped_obs[k] = v[:, None].astype(np.uint8)   # (B,1,H,W,3)
            elif k.startswith("state.") and v.ndim == 2:
                wrapped_obs[k] = v[:, None].astype(np.float32) # (B,1,D)
            else:
                wrapped_obs[k] = v
        else:
            wrapped_obs[k] = v

    # 3) 加 observation envelope
    action_chunk = self.call_endpoint("get_action", {"observation": wrapped_obs})

    # 4) 解 list[0] → dict，再用 mnp.decode 还原 bytes-key dict 成 ndarray
    if isinstance(action_chunk, list):
        action_chunk = action_chunk[0]
    arm  = _to_ndarray(action_chunk["action.single_arm"])
    grip = _to_ndarray(action_chunk["action.gripper"])

    # 5) (1, 40, D) → (40, D)；拼成 (40, 6)；最后 (40, 1, 6) 返回
    if arm.ndim == 3 and arm.shape[0] == 1:
        arm, grip = arm[0], grip[0]
    concat = np.concatenate([arm, grip], axis=-1)
    concat = convert_lerobot_action_to_leisaac(concat)
    return torch.from_numpy(concat[:, None, :])

def _to_ndarray(x):
    if isinstance(x, np.ndarray): return x
    if isinstance(x, dict):
        ks = set(x.keys())
        if "nd" in ks or b"nd" in ks:
            if "nd" in ks:
                # mnp.decode 要 bytes keys —— 反向 re-encode
                x = {(k.encode() if isinstance(k, str) else k): v for k, v in x.items()}
            return msgpack_numpy.decode(x)
        if "__ndarray_class__" in ks or b"__ndarray_class__" in ks:
            key = "as_npy" if "as_npy" in x else b"as_npy"
            return np.load(io.BytesIO(x[key]), allow_pickle=False)
    raise TypeError(f"cannot decode action element: {type(x).__name__}")

同时 server/eval_gr00t.sh 加 POLICY_TYPE 环境变量并默认 gr00tn1.5（与上面 client 配套）。

6. 复盘：为什么调这么久？— Post-mortem

  • 4 个 bug 串行触发，每修一个露下一个，没法一眼看到 root —— 类似 5-layer onion peeling。
  • Bug 3 最隐蔽：MsgSerializer 已有 nd:True 解码逻辑，看代码以为没问题；但 msgpack-python 默认 raw=False 时只把 string-typed dict keys 解成 str，非 ASCII 编码的 bytes keys（来自 msgpack-numpy）保持 bytes，绕过了我们的 if check。
  • Bug 4 是经典反向直觉：看到 bytes keys 反应是"转 str 再处理"，实测 mnp.decode 强制 bytes keys。
  • 5090 32GB OOM 副作用：每次 eval timeout 杀掉 bash，但 conda subprocess 不退出，每次累 6.7 GB GPU。3 次后 PhysX OOM 拒绝创建 scene，看起来像新 bug 实际是僵尸进程。教训：以后 eval 用 pkill -f policy_inference 兜底。
