调研 优先级 SO-101 PickOrange

作者：vitorcen
状态：草案
触发：SmolVLA2 在 60 episode 数据上不稳定（2/5 success），ACT 和 GR00T 第一次就能 60s 内成功——说明 SmolVLA 的 VLM-based 架构在 60 episode 规模上吃力。换一类策略试。

命名澄清：HF 主 lerobot 包（pip v0.5.2）不含 SmolVLA2。但 huggingface/VLAb（LeRobot 的 official reproduction kit for SmolVLA pretraining）里有独立的 policies/smolvla2/ 实现，--policy.type=smolvla2，是 v1 的架构升级版。HF 上目前还没有官方 smolvla2_base ckpt，只能从 HuggingFaceTB/SmolVLM2-500M-Video-Instruct 这个 VLM backbone 起训。注意：edge-inference/smolvla-so101-pick-orange 和我们本机 smolvla2-leisaac-pick-orange 都是 v1 的 fine-tune（config.type="smolvla"），不是 VLAb 的 v2 架构。本地 smolvla2-… 这个名字是历史命名包袱，实际上是 SmolVLA v1。
SmolVLA v1 → v2 关键升级（对比两个 config.py 默认值）：
  • train_expert_only: v1 True → v2 False（v2 同时训 VLM，不只是 action expert）
  • PEFT/LoRA: v1 可选 → v2 默认开启（r=4, alpha=16, dropout=0.1），让训 VLM 显存可控
  • num_expert_layers: v2 可配（默认 -1），expert_width_multiplier=0.75 独立缩放 expert
  • 含义：v1 在小数据上 VLM 没机会适配 PickOrange 视觉分布；v2 通过 LoRA 让 VLM 也学一点，理论上小数据迁移更好
结论一行：60 episode × 36k frame 的数据规模 + 60s episode 预算下，小参数 + 强归纳偏置（ACT 80M ✅）显著优于 大参数 VLA（SmolVLA 450M：第三方 0/3，自家 2/5 @ 120s）。下面候选都满足"小参数 + 强结构先验"或"大规模机器人预训练"。
关键证据：连官方公开的 edge-inference/smolvla-so101-pick-orange 在 60s 预算下也 0/3。这排除"我们的训练流程有 bug"假设，定位到 SmolVLA 架构 + 60 ep 小数据这一组合本身的能力上限。我们的 v2 在 120s 长预算下还能 2/5，说明训练流程是对的，只是 SmolVLA 推理速度 + 动作精度跟不上 60s 严格预算。

2. 候选策略

2.0 SmolVLA2（VLAb 仓库的新架构）— Hugging Face 官方

项值
来源huggingface/VLAb · src/lerobot/policies/smolvla2/（约 75KB Python 实现）· 主仓库 LeRobot 还没合并
架构SmolVLA v1 + LoRA-trained VLM + 可配 action expert（layers/width 独立）；flow matching 同 v1（num_steps=10）
参数量~450M total（同 v1），但 trainable 上升（v1 ~100M → v2 ~100M expert + LoRA on VLM）
预训练 ckpt❌ HF 上目前没有 lerobot/smolvla2_base。只从 HuggingFaceTB/SmolVLM2-500M-Video-Instruct（VLM backbone）起训。VLAb 的定位是"reproduction kit for pretraining"——他们的预期是社区用自家数据预训出一个 smolvla2_base，目前还没公开发布
训练框架VLAb 仓库自带 src/lerobot/scripts/train.py，独立于主 lerobot 包；用 accelerate launch
典型训练步数VLAb README 示例：batch=8, steps=10k–200k（数据规模差很大）
LeRobot 集成⚠️ 主 lerobot v0.5.2 不识别 --policy.type=smolvla2；推理用的 lerobot.async_inference.policy_server 也无法加载 smolvla2 ckpt
LeIsaac 集成成本⚠️ 非平凡：(a) clone VLAb 进 conda env，(b) 给推理写一个 VLAb-aware 的 policy_server，或把 smolvla2 代码 vendor 进我们的 lerobot fork，(c) policy_inference.py 加 lerobot-smolvla2 分支 + 客户端 schema 处理（同 v1）
预计 4090 训练时长~1.5–3h（LoRA 后 trainable 参数和 v1 相近，速度类似）
预计 success中等偏高：v2 让 VLM 也微调（用 LoRA），理论上能解决 v1 "VLM 没适配本任务视觉分布"的问题。但无预训机器人 ckpt 起点，60 ep 直接训 v2 不一定比 v1 + smolvla_base 强
风险(1) 工程量大；(2) 无现成机器人预训 ckpt，相当于 from-scratch 训 VLA，60 ep 数据未必够

2.1 Diffusion Policy（DP）— Columbia / Chi et al 2023

项值
论文arXiv:2303.04137（"Diffusion Policy: Visuomotor Policy Learning via Action Diffusion"）
架构1D-CNN UNet over action chunk，conditioned on ResNet18 vision features + state
参数量~100M（取决于 ResNet 大小 + UNet 深度）
LeRobot 集成✅ 已内置：--policy.type=diffusion（lerobot/policies/diffusion/，Columbia 团队原代码移植）
典型训练步数50k–200k step（chunk_size=16, horizon=16）
LeIsaac 客户端支持⚠️ 需补 lerobot-diffusion 适配：policy_inference.py 已经有 lerobot-<model_type> 分支，但 service_policy_clients.py 的 LeRobotServicePolicyClient 需要确认能接 diffusion 模型（理论上通用，需 smoke test）
预计 4090 训练时长50k step × ~10 step/s（batch 64）≈ ~1.5h
预计 success高：DP 在小数据 visuomotor 上是工业 baseline；同尺度公开数据上社区已多次复现
风险chunk_size 比 ACT 短（16 vs 100），需要更高频再推理；客户端 horizon 调小

2.2 DiT Policy — LeRobot's multi_task_dit

项值
来源LeRobot 内置（Bryson Jones, 2025；lerobot/policies/multi_task_dit/）
架构Transformer-based diffusion，支持 diffusion 和 flow_matching 两种 objective；text + vision 条件；hidden 512 / 6 层 / 8 头（默认）
参数量~50–150M（取决于 layers/heads/hidden）
LeRobot 集成✅ 已内置：--policy.type=multi_task_dit
典型训练步数n_obs_steps=2, horizon=32, n_action_steps=24（~0.8s @ 30fps）
LeIsaac 客户端支持⚠️ 同 DP：需补 lerobot-multi_task_dit 路径 smoke test。policy_type 字符串含 _ 可能要小改 split 逻辑（args_cli.policy_type.split("-")[1]）
预计 4090 训练时长~1–2h（DiT 比 UNet 略重）
预计 success中等偏高：架构比 DP 新，flow_matching 收敛快；多任务能力对单任务可能"过度参数化"
风险"multi_task" 命名暗示设计目标是多任务；单任务上未必比 DP 更好

2.3 RDT-1B — Robotics Diffusion Transformer（清华 TSAIL）

项值
论文 / repoarXiv:2410.07864 · thu-ml/RoboticsDiffusionTransformer · HF ckpt
架构1B-param Diffusion Transformer；SigLIP-SO400M 视觉 + T5-v1.1-XXL 语言；统一动作空间支持单/双臂 + 关节/末端控制；最多 3 路 RGB
预训练数据1M+ 多机器人 episode，46 个数据集（含 Open-X、DROID、BridgeData V2、RoboSet 等）
LeRobot 集成❌ 不在 LeRobot 列表中。要么 fork 一个 LeRobot policy class wrapper，要么直接用 thu-ml repo 的训练脚本，再写 LeIsaac client 适配 RDT 的 ZMQ/HTTP 推理协议
典型微调流程repo 的官方 tutorial：~100 demo 微调 SO-101 类设备；fine-tune 推荐 ≥ 50 episode
预计 4090 训练时长1B 参数 + 双相机微调，单卡 4090 大概 ~12–24h；可能撞显存上限（24GB 不一定够 full fine-tune，需要 LoRA / freeze backbone）
预计 success高但不确定：预训练数据巨大且含 SO-100 / SO-ARM 类似 embodiment，理论上 60 episode 微调即可；但 1B 模型在小数据上可能过拟合，且没有现成 LeRobot adapter
风险显存 / 训练时长 / 客户端集成 全是工作量；显存若不够要 LoRA，效果可能打折

2.4 Octo（Octo-Base 1.5）— RAIL Berkeley

项值
论文 / repoarXiv:2405.12213 · octo-models/octo · HF ckpt
架构93M（ViT-B 级别）Transformer + diffusion action head，window_size=2 obs，4-step action prediction
预训练数据Open-X Embodiment 26+ 数据集（Fractal/Kuka/Bridge/BC-Z 等）
原生框架JAX（原 Octo repo 是 JAX，HF 已 wrap 成 transformers AutoModel；微调 / 推理是否完整支持 PyTorch 需要再确认）
LeRobot 集成❌ 不在 LeRobot 列表中。要么写 LeRobot policy class 包 HF Octo wrapper，要么用 Octo repo 的 fine-tune 脚本
典型微调官方 fine-tune 脚本支持自定义数据，~10k step；JAX 训练 4090 需要 jax-cuda12
预计 4090 训练时长93M 模型，小数据微调 ~1–3h，前提是 JAX 跑通
预计 success不确定：93M 小模型 + Open-X 预训练，理论上很适合 60 episode 微调；但 JAX 环境 + 协议适配是非平凡前置工作
风险JAX env 单独管理；和 LeRobot 通用 policy_server 不兼容（自家 wrapper 才行）；下游 LeIsaac 客户端需要新写一类 client

3. 优先级排序

排序原则（Linus 模式）：

  1. 已有 scaffold 就先用——LeRobot 内置的两个（DP / DiT）几乎零基建成本
  2. 先排除"架构不行"再做"工程量大"——如果 DP/DiT 也搞不定，再说大模型；反之 ACT 已经搞定，说明 60 ep 任务本身可学，问题就在我们 SmolVLA 这一支
  3. 避免一次性吃两个新依赖——RDT 和 Octo 都要新环境 + 新协议；先做能马上跑的
顺序策略预估总工时训练时长预期收益 / 风险
🥇 P0 Diffusion Policy（--policy.type=diffusion） ~3h（含 client smoke + 训 + 评） ~1.5h LeRobot 内置；社区在小数据 visuomotor 上验证最充分；最可能"开箱即用"成功
🥈 P1 DiT Policy（--policy.type=multi_task_dit） ~3–4h ~1–2h LeRobot 内置；架构比 DP 新；如果 DP 通了再跑 DiT 对比 diffusion vs flow_matching
🟦 P2 SmolVLA2（VLAb 仓库 --policy.type=smolvla2） ~6–8h ~1.5–3h v2 用 LoRA 微调 VLM，理论上比 v1 在小数据上更适配；但 (1) 无现成机器人 base ckpt，从 SmolVLM2 VLM 起训风险更大，(2) 推理 server 需要 vendor smolvla2 代码进我们的 lerobot fork，工程量比 DP/DiT 大。价值：直接对比 v1 vs v2 在同任务上的差距，量化"VLM LoRA"的收益
🥉 P3 Octo（fine-tune from octo-base-1.5） ~6–10h ~1–3h Open-X 预训练带来的迁移可能很强；前置工作：JAX env + LeIsaac client；做完是预训练价值的对照实验
🚫 P4 RDT-1B（fine-tune） ~12–20h ~12–24h（4090 可能要 LoRA） 1B 参数 + 显存紧 + 无 LeRobot adapter；如果 Octo 已经验证"大预训练 + 60 ep 微调"是否有用，RDT 边际信息量小；除非 Octo 也失败，才有动机搬 RDT
为什么 SmolVLA2 不是 P0：它的"VLM-with-LoRA"升级是解决 SmolVLA 系列问题的对症药，但代价是无预训机器人 ckpt 起点（HF 上还没有 smolvla2_base）。从 SmolVLM2 VLM 起训等于"少了一层 robotic prior"。在 60 ep 小数据上能不能赢过 v1 微调，需要实测才知道。而 DP / DiT 是架构层面与 ACT 同档（小参数 + 强 visuomotor 归纳偏置），更有把握。先做 P0/P1 拿到工作的轻量 baseline，再做 P2（SmolVLA2）量化 VLM 升级的价值。

7. P3：Octo（条件性）

只有在 DP / DiT 都不达预期、且想测试"Open-X 预训练能否救小数据"时再上 Octo。前置工作：

  1. 新 conda env：conda create -n octo python=3.10 + JAX cuda12 + flax + transformers
  2. LeRobot policy_server 不支持 Octo，需要写一个独立的 octo_server.py（参考 GR00T N1.5 的 ZMQ inference_service.py 模式）
  3. LeIsaac OctoServicePolicyClient：传 RGB + state 给 server，接 action chunk
  4. 训练：用 octo-models repo 的 fine-tune 脚本 + 自家数据

8. P4：RDT-1B（条件性）

仅在 Octo 也不行、且预算允许 12–24h 微调时再考虑。前置工作量比 Octo 大约 2x：

  1. 新 conda env + flash-attn + bfloat16
  2. 显存：1B 模型 + 双相机 480×640，4090 24GB 大概率不够 full fine-tune，必须 LoRA（thu-ml repo 支持）
  3. 独立 server 协议（thu-ml repo 自带 ROS-based 推理 demo，需要改成 ZMQ/gRPC）
  4. LeIsaac RDTServicePolicyClient

9. 关键问题：为什么 SmolVLA v1 不行而 ACT 行？

把这个先想清楚再选下一个，避免重复同一个坑：

因素ACT (✅)SmolVLA2 (🟡)对 P0-P3 启示
归纳偏置强：CNN + chunked Transformer + VAE，专为 visuomotor 设计弱：VLM 是为通用任务训的，action expert 只占小部分参数DP、DiT 都是 visuomotor 专用 → 沿用 ACT 的优势
有效可训参数~80M 全部可训~100M trainable（其余 350M VLM 冻结）DP / DiT 全可训，效率更高
预训练数据仅 ImageNet（ResNet18 backbone）SmolVLM2 通用 vision-language 预训练VLM 预训练 ≠ 机器人预训练；Octo/RDT 的预训练才真正"对口"
chunk 大小100 步（~3.3s @ 30fps），整套 pick&place 的时序结构能整体建模50 步（~1.7s）P0/P1 可以试 chunk_size=64 或 100
训练步数10k step × batch 830k step × batch 8不是步数问题；架构/数据匹配度才是关键
推测：60 episode + ImageNet 预训练的小模型刚好够这个任务的归纳偏置；450M-3B 大模型在 60 ep 上信号过稀。DP/DiT 跟 ACT 同一档"小+visuomotor 专用"，预期能复现成功率。Octo/RDT 是"先靠大规模机器人预训练吃迁移"，是另一条路。

10. 验收标准

所有候选都用同一套 LeIsaac eval：3 episode × 60s，--policy_action_horizon=<chunk-aligned>。
