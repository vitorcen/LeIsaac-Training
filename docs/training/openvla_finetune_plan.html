2.2 模型内部结构 · Internal architecture

OpenVLA 是 Stanford Prismatic VLM (LLaVA-style 拼装) 上的 action token finetune，没有独立的 action head —— action 直接复用 Llama lm_head 输出 32k 词表的顶端 256 个 token 作为离散 bin。

组件来源实现参数估算
Fused Vision BackboneStanford Prismatic（modeling_prismatic.py:63 PrismaticVisionBackbone）DINOv2 ViT-L + SigLIP ViT-L 双 encoder 并行，patch features 沿 channel 维 concat；timm 实现，权重 frozen 不推荐 LoRA~0.6B (300M + 300M)
Linear ProjectorStanford Prismatic2-layer MLP 把 vision patch features (D=2176) → Llama embedding 维 (D=4096)；通常 trainable~30M
OpenLlama-7BOpenLM Research（开源 Llama-2 复刻）32 层标准 causal Transformer；32k vocab；action token 用顶端 256 个 vocab id（token_id = vocab_size - 1 - bin_idx）~7B
合计——~7.6B（其中 LoRA 我们训 ~17M）
每 action = 6 forward → 240 ms @ 4090 → sim 必须 step_hz≤5 反量化 (un-norm) bin_idx = vocab − 1 − token a_norm = bin_centers[bin] a = q01 + (a_n+1)/2 × (q99−q01) → continuous 6-DOF joint image_embeds 关键设计要点 • 没有独立 action head：action 复用 Llama lm_head 输出，顶端 256 个 vocab id 当 256 bin（vanilla 唯一的"创新"是把 action 当 text token） • causal AR decode：每 action = 6 个 sequential forward（与 chunk policy 全反），→ inference 慢，难高频闭环

读图要点：

  • OpenVLA 没有专门的 action head — action 是 Llama 32k vocab 顶端 256 个 token id 的离散化，token_id = vocab_size − 1 − bin_idx
  • Vision = DINOv2 + SigLIP 双 encoder concat channel（不是 sum、不是单 encoder），这是 Prismatic VLM 论文的设计
  • Llama causal AR decode 意味着 6-DOF action = 6 次 forward，没法并行，这是 240ms/action latency 的根因
  • 我们做 QLoRA 微调时只动 q_proj+v_proj 的 16.78M 参数（0.22%），vision + projector + 其他 Llama 层全 frozen
  • OpenVLA-OFT 解决方案就是替换 lm_head 为独立 MLP action head + chunked action + bidirectional attn（见 openvla_oft_finetune_plan.html）

4.1 为什么 step_hz=5 是结构性问题（非可调参） · Why 5Hz is a structural bottleneck

结构性 distribution shift · Structural domain shift：
  • Dataset 采集 @ 30Hz（SO-101 teleop 标准）— 每相邻帧 33ms 位移
  • OpenVLA 闭环只能跑 5Hz（200ms 步）— 每步位移 6× 训练时见过的最大 delta
  • vanilla policy 没见过 200ms 的 state→action 跳变，输出超出 q01/q99 范围概率 ↑↑
  • 这不是 "fine-tune 救" 能解决 — 训练数据本身没有 5Hz 样本，rebuild dataset @ 5Hz 又会损失运动连续性
  • 对照：N1.7 chunk=40 / N1.6 chunk=50 在 240ms latency 下每 chunk 摊薄到 5ms/step 等效频率 ≈ 200Hz，跟 30Hz dataset 兼容
所以 OpenVLA vanilla 在 LeIsaac 上的 0/9 结果，有一半来自这个结构性 mismatch，剩下一半来自 trust_remote_code ckpt 加载 bug。bf16 LoRA 修不了任何一个。

4.2 30Hz 闭环可行性 · 30Hz feasibility

240 ms / action @ 30Hz (33ms/step) = slowdown ~7×。Isaac Sim 不能跟 30Hz 实时跑，但：

#方案实际效果成本
1step_hz 降到 5 Hz每步 200 ms 推理窗口 ≈ 1× 实时；dataset 30Hz → distribution mismatch 待 fine-tune 救低
2切到 OpenVLA-OFT (chunk + parallel decode)论文报告 25× 加速；本质换 model 重训高
3TensorRT-LLM 部署~10ms/token = 60ms/action ≈ 2× slowdown中
常见误区 · Common myth: bnb int4 不会加速 decode（实测中 4-bit 与 bf16 速度持平）；它只省显存。真正的加速需要 chunked / parallel decode 架构（OpenVLA-OFT, PD-VLA）或 TensorRT-LLM 引擎。
bnb int4 saves memory, NOT compute. For real speedup, use OFT-style chunked decode or TensorRT-LLM.

5. 微调资源估算 · Training Resource

5.1 三种训练模式对比 · Three training modes

QLoRA NF4 (memory-only saving) vs bf16 LoRA (the "16-bit" retrain path) vs bf16 full FT (single card infeasible).

⚠ 2026-05-19 实战结论：三种 mode 都不修 root cause
  • 4-bit QLoRA × 5 ckpts × 9 ep = 0/45（已跑）
  • 8-bit + bf16 LoRA = 同样 0（codex 已验证）
  • 根因 = torch 2.3 + trust_remote_code 的 Prismatic VLM ckpt 加载 bug，跟 precision 无关
  • vGPU-48GB bf16 LoRA 跑 5h 烧 ¥10-15 训练能跑通，但闭环 SR 期望 0-2/15（5-round）/ 0-5/60（20-round 严格）
  • 排名压在 leaderboard 末位低 SmolVLA (15/60) 至少 3×，不值得花钱重跑

"16-bit 重训 = bf16 LoRA" 的物理边界：
  • 核心增量：base 14 GB（4-bit）→ 15.2 GB（bf16，2 byte/param × 7.6B），多吃 ~12 GB
  • 不要误以为 LoRA 优化器也会跟着膨胀 —— 我们只训 17M 参数，optimizer 还是 ~0.2 GB，可忽略
  • 真正的瓶颈是 base + activations 的常驻量。4090 24G 在 batch=4 时差 4-8 GB，降 batch=1 + grad_accum=16 勉强可挤进 22-24 GB（高风险），不如直接换 48G 卡
  • bf16 full FT 加 91 GB 的 fp32 AdamW state，单卡只能 80G H100 + offload，性价比远不如 LoRA

踩坑根治 · Memory-corruption fix (2026-05-19)

症状 · Symptoms (bnb 0.43.1)：训练随机崩，3 种不同的 traceback 都指向同一根因 = 4-bit Linear / PEFT 包装下 _named_members tuple 解包失败：
  1. ValueError: too many/not enough values to unpack (expected N, got 2) 触发于 Trainer.log → floating_point_ops → num_parameters → named_parameters → _named_members
  2. 同一类 ValueError 触发于 bnb 4-bit 量化器加载阶段 → find_tied_parameters → named_parameters → _named_members
  3. 纯 SIGSEGV / "段错误（核心已转储）"，无 Python traceback
ValueError 甚至会飘到完全无关的 sre_compile（regex 编译）和 pandas import — 因为 C 扩展层踩 Python 堆，破坏的 tuple 在哪个迭代器解包都炸。
根治 · Root-cause fix： 
pip install bitsandbytes==0.46.1   # 把 0.43.1 升上来，其他包不动
同时把 optimizer 改成 adamw_torch（已是 train.py 默认），避免叠加 8-bit AdamW 这条第二条敏感 bnb 内核路径。

效果对比（本机 4090，save_steps=100 + 20-retry watchdog）：
指标bnb 0.43.1bnb 0.46.1
5500→6000 chunk 1h 进度5500 卡死5700 saved + 5730 still running
每次 attempt 崩点5527-5578（save 之前）5721（已过 save 边界）
新 ckpt 落地数02+（5600, 5700, ...）
belt-and-braces patch（留在 train.py + server/openvla_leisaac/server.py 顶部）： 
import transformers.modeling_utils as _mu
import accelerate.utils.modeling as _am
import accelerate.utils as _au
import transformers.integrations.bitsandbytes as _tb
_mu.PreTrainedModel.floating_point_ops = lambda self, inputs, exclude_embeddings=True: 0
_noop_tied = lambda *a, **kw: []
_am.find_tied_parameters = _noop_tied
_au.find_tied_parameters = _noop_tied
_tb.find_tied_parameters = _noop_tied
A 行屏蔽 Trainer FLOP 估算（日志 TFLOPS 列归零，无功能影响）。B 行让 bnb 4-bit 加载跳过 tie-weights 检测（对 OpenVLA 无影响，它生成 action token 不是 text）。

train.py 还需做的：resume 前覆写 trainer_state.json["save_steps"] = args.save_steps（Trainer 优先用 state-file 值不是 CLI），让 100-step save 真的生效；trainer.train() 返回后强制 trainer._save_checkpoint(model, trial=None) 保证 checkpoint-$MAX_STEPS 落盘。
不要再试 · Anti-list (codex gpt-5.4 review)：
  • ❌ 继续给 Python 层打 monkey patch — 同一根因，换地方再崩
  • ❌ 开 gradient checkpointing — 与 bnb 4-bit 量化态机互踩，2026 还有公开 issue
  • ❌ 训练用 device_map="auto" — HF 文档明确仅 inference 用
  • ❌ torch.cuda.empty_cache() 治本 — PyTorch 文档明确说不会增加可用显存
  • ❌ 期望 CUDA_LAUNCH_BLOCKING=1 修崩 — 只是 debug 工具，会把异步问题变同步
如果 0.46.1 还崩：
  1. 先升 bnb 0.46.1 → 0.47.0（FSDP2 + 大 block size 修补）
  2. 再退到 8-bit base + LoRA + torch AdamW（绕开 Linear4bit）
  3. 最后跑 compute-sanitizer --tool memcheck python train.py ... 抓非法内存访问
来源：bnb 0.46.1 / 0.47.0 release notes、PR #1719 Params4bit fix、PR #1866 state_dict fix、HF transformers bnb docs。
