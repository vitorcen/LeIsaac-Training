StarVLA + Qwen3-VL-4B 微调可行性

StarVLA 框架（MIT）× Qwen3-VL-4B backbone · SO-101 PickOrange 适配评估 · 2026-06-03

Feasibility study for the StarVLA framework with a Qwen3-VL-4B backbone on the LeIsaac SO-101 single-arm PickOrange task. Verdict: the "vision-dividend" second step after WALL-OSS.

TL;DR：StarVLA 是唯一有真实代码路径的 Qwen3-VL 机器人 VLA（MIT，可下载 ckpt，LeRobot v3，四种 action head 可选）。Qwen3-VL 的 DeepStack + 3D-grounding 升级理论上对小物体最有利。估 30–60%，定位为 WALL-OSS 之后吃 vision 红利的第二步。
StarVLA is the only Qwen3-VL robot VLA with a real code path (MIT, downloadable ckpt, LeRobot v3, four action heads). DeepStack/3D-grounding theoretically help small objects most. Est. 30–60%, positioned as the "vision-dividend" second step after WALL-OSS.
主要代价 / Main cost：SO-101 在 StarVLA README 里还是个 未勾选的 TODO，单臂适配链路要自己补；且 Qwen3-VL 对 10–40px 小物体的提升只有定性陈述、无量化证据。

2. 系统架构 · Architecture

3. 核心：Qwen3-VL 的 vision 升级对 PickOrange · The vision dividend

相对 WALL-OSS 的 Qwen2.5-VL，Qwen3-VL 的卖点正好压在本任务的命门上——小物体细粒度感知：

  • DeepStack：把 ViT 多层特征注入 LLM 多层，官方原话 "capture fine-grained details"。对 10–40px 的小橙子，这是结构上最有利的改动。
  • 2D/3D grounding 增强：官方点名 "spatial reasoning and embodied AI"，对"在画面里精确定位橙子"直接相关。
  • native dynamic resolution：与 Qwen2.5-VL 同样可吃满 640×480，不存在固定 224 瓶颈。
⚠️ 诚实边界 / Honest scope：以上 vision 升级是官方定性陈述，没有"对 10–40px 小物体提升 X%"的量化 benchmark。DeepStack/grounding 在通用 VQA 上变强，不能直接外推到"橙子抓取成功率 +N%"。真实增益要靠我们自己跑出来。另外部署像素/token 预算同样需在 config 里确认（同 WALL-OSS 的 256 陷阱风险）。

4. action head 选择 · Head choice

StarVLA 一个框架支持四种 head，可直接做同 backbone × 不同 head 的受控对照（横评价值高）：

Head机制Bridge-RT-1 分PickOrange 建议
PIflow-matching diffusion（同 π0.5/WALL-OSS 制式）69.8✅ 首选
GR00T双系统 flow-matching（同 GR00T-N1.7）65.3✅ 次选
OFTMLP 连续 + L1（同 OpenVLA-OFT）42.7对照用
FAST离散 AR token（同原版 OpenVLA）—不推荐（小数据欠拟合）

已发布 ckpt 自带这三组分数，flow-matching 系（PI/GR00T）显著优于离散——与我们 PickOrange 的经验一致（continuous flow > discrete token）。

5. SO-101 单臂适配（要自己补）· Single-arm adaptation

主要工作量在这里。StarVLA 预训练数据（Bridge/Fractal）本身是单臂，所以不像 LingBot 那样有"双臂→单臂"的架构鸿沟；但 README 里 SO101 是未勾选 TODO，没有现成 config。需要：
  • 写 SO-101 6-DOF 的 robot/dof config（可参考 WALL-OSS 的 S101 示例 + StarVLA 既有单臂 config）。
  • 确认 observation 命名映射（我们的 cam_front/cam_wrist → StarVLA 期望的多视角键）。
  • action 归一化统计接 LeRobot dataset。
相比 WALL-OSS（官方现成 S101 示例），这是 StarVLA 排第二的核心原因——它把 vision 红利和适配成本一起给了你。

7. 实施路径 · Implementation

Phase 0 — 框架落地（~0.5 天）

  • clone starVLA/starVLA，跑通官方 Qwen3-VL-4B-Instruct-Action + LeRobot demo。
  • 确认分辨率 / token 预算配置项（避免 256 陷阱）。

Phase 1 — SO-101 config 适配（~1 天）⭐ 主要工作量

  • 写 SO-101 6-DOF dof_config + observation 映射（参考 wallx_finetune_plan.html 的 S101 示例）。
  • 接 PickOrange LeRobot dataset（v3.0 原生）。

Phase 2 — 微调（~1 天）

backbone: Qwen3-VL-4B-Instruct-Action
action_head: PI            # flow-matching，首选
resolution: 480            # ⚠️ 吃满原生帧，避免 256 陷阱
robot: so101_single_arm    # 6-DOF
lora: true  bf16: true  grad_ckpt: true  batch: 1

Phase 3 — Server + client（~1h）

  • server/starvla_leisaac/server.py 类比 pi05/gr00t ZMQ server。
  • policy_inference.py 加 --policy_type=starvla。

Phase 4 — Eval + 同-head 对照（~0.5 天）

  • 主跑 PI head；时间允许加跑 GR00T/OFT head 做同 backbone × 不同 head 受控对照（独特横评价值）。

8. 成功率预估 · Forecast

估 30–60%（中高把握 / moderate-high）。
落点逻辑：Qwen3-VL 动态分辨率 + DeepStack 理论上 ≥ WALL-OSS 的 vision，进入 work 区；flow-matching head（PI/GR00T，Bridge 65–70 分）sample-efficient。下界 30% 反映 SO-101 适配从零、Qwen3-VL 小物体增益无量化证据、4090 降配三重不确定；上界 60% 若 DeepStack 真对小橙子有效 + 适配顺利，可与 WALL-OSS 并肩。比 WALL-OSS 略保守，纯因工程成本（SO-101=TODO）更高，不是模型更弱。
Est. 30–60%. Vision ≥ WALL-OSS in theory (Qwen3-VL + DeepStack); flow-matching head is sample-efficient. Lower bound reflects from-scratch SO-101 adaptation + unquantified small-object gain + 4090 downscaling. More conservative than WALL-OSS purely on engineering cost, not model capability.
