Flow-Matching 策略选型调研

SO-101 PickOrange · 为 ACT/Diffusion-Policy 寻找非-VLM flow-matching 替代 baseline · 2026-06-13

Goal: survey 5 open-source flow-matching manipulation policies, pick the best fit for the RGB-only SO-101 PickOrange contract — or conclude none drop in and write a minimal one. Three independent reviews (Claude + codex/gpt-5.5 + mimo-v2.5-pro) converge on the same answer.

0. TL;DR

结论：5 个仓库没有一个能直接 drop-in。
  • 2 个是点云/深度派（PointFlowMatch、FlowPolicy）→ 被 PickOrange RGB-only 硬约束直接淘汰。
  • A2A_Flow_Matching 是最佳外部候选（RGB + 单臂 + Isaac + 1-step 0.56ms + Apache-2.0），但绑死 RoboVerse、需写 LeRobot 桥 + serve 适配 → 是 port 不是 swap。
  • streaming / HRI-EU：前者无 LICENSE 且图像变体打不过 DP，后者深度+affordance 为主 → 都只能借思想。
推荐路线 自己写最小 CFM head： 复用现成 lerobot Diffusion-Policy 全套接线 （image encoder + UNet1D + LeRobotDataset loader + serve/eval 适配器），只把 DDPM 噪声预测目标换成 直线 conditional flow-matching（rectified flow）速度场。零新耦合、真·drop-in、RGB 原生。
三方评审一致的最强警告 the head is the wrong bottleneck： DP 现在 8.3% vs ACT 43.3% 的 5× 差距，大概率不是 head 问题，是 DP 把图像缩到 240×320 把橙子（10–40px）糊没了。 换 flow-matching head 不会修好视觉 grounding。 所以第一步不是写 FM，而是：先把 DP 分辨率拉回高分辨率重测 DP baseline，否则你是在跟一个被蒙住眼的对手比，FM 赢了也不知道是 FM 赢还是分辨率赢。

3.1 为什么 A2A 是"最佳外部"但仍不直接用

A2A 在架构层面完全对口：RGB-only、单臂、Isaac Sim 5.0 跑过、绝对/关节 chunk、Apache-2.0、单步极快、在它自己的 sim bench 上打赢 ACT/DDPM-UNet。它的创新点"action-to-action"——把 flow 的起点从高斯噪声换成执行过的动作历史—— 利用运动时序连续性缩短 ODE 路径，因此能 1 步。但：

  • 绑死 RoboVerse：不是 policy.predict(obs) 库，要写 RoboVerse→LeRobot 数据桥 + serve 适配器（同 serve_starvla/serve_wallx 工作量）。
  • proprio 历史是结构性承重：流源依赖干净的执行动作历史 → 对 state 归一化/置换极敏感（正中我们踩过的 sonic-serve-state-permutation-bug 雷区）。
  • 视觉是 ResNet-18、分辨率未明 → 正是 PickOrange 选型铁律警告的小目标瓶颈。

codex 的判词最锋利："best external repo ≠ best experiment"。把 A2A 移植进来，会同时引入 RoboVerse 耦合、history 语义、图像预处理、proprio 约定 4 个不受控变量——它若赢了，你分不清是 FM 赢、A2A 的历史先验赢、 还是移植时顺手修好了 DP 的视觉通路赢。要做干净的 A/B，就只能改一个变量。

3.2 Transformer / DiT 对比 why E2 is conv-UNet, not a DiT

5 个项目里谁用了 Transformer? 只有一个。其余清一色 CNN/MLP/PointNet——这本身就是社区在这个数据规模下的共识信号。

判断：E2 坚决用 conv-UNet，不上 DiT。① 60 demo 是 CNN 的主场、DiT 的死穴；② 调研里唯一做过直接 A/B 的 HRI-EU 结论是 CNN 胜；③ 单变量对照要求 backbone 不变。 DiT 的入场条件 = 数据扩到数百+ demo / 多任务，或回到 VLM token 底座（本轮明确绕开的 VLA），且必须先修视觉分辨率——否则就是拿弱视觉喂数据饥渴架构，双输。记为「E-later 选项」。

4. 三方独立评审一致结论 Claude × codex/gpt-5.5 × mimo-v2.5-pro

问题三方共识
Q1 自研 vs 移植 A2A？自研最小 CFM head。 fork 现成 lerobot DP 接线，dataset/server/eval/动作归一化全保持一致，只换 DDPM→rectified-flow 速度预测。这才是可辩护的 A/B。
Q2 FM 能打赢 ACT 43.3% 吗？视觉不修好就别赌。橙子 10–40px 被 224/240 缩没了，再好的动作生成器也救不回被毁的视觉证据。FM 主要赢在推理速度/chunk 平滑/更少步数/小数据上比 DDPM 更稳——不会提升视觉 grounding。
预期：ACT(高清) > FM(弱视觉) ≈ DP(弱视觉)。真正该测的：FM(高清) vs ACT(高清)。
Q3 最小第一版怎么搭？rectified/线性 CFM · noise→action · 同 DP UNet/chunk/horizon · 同 obs/state/action 契约 · 高分辨率编码器（≥448 路径） · 推理扫 1/2/4/8 Euler 步 · 不上 consistency、不上 streaming、不上 A2A 历史源（留 v1）。理由：隔离"目标函数"这一个变量。
Q4 漏了什么？两个独立都点出：(a) 先 root-cause DP 的 8.3%——多半是 240×320 缩放，修了 DP 可能直接跳到 ~30%，FM 才是在跟诚实的对手比；(b) 抄 π₀ 的 action head 设计（标准 CFM 模块、与 VLM 解耦），而不是抄这 5 个学术 side-project。其他可留作后续：ManiFlow（consistency 1-2 步通用操作）、VFP（变分 FM 解多模态）、RTC（real-time chunking 隐藏延迟）、Latent Action Flow（时序连贯 latent）。

5. 决策与实验阶梯 the honest A/B ladder

核心洞察（三方一致）：ACT↔DP 的 5× 差距是"穿着 action-head 外衣的视觉问题"。 所以先控视觉变量，再问 FM 是否有用。
阶段实验变量判据
E0DP 原版（240×320 低清）—（已有 8.3%）参照点
E1 ⭐先做DP 高分辨率（不缩，≥448 路径）重训重测仅分辨率若跳到 ~30% → 证实瓶颈是视觉非 head
E2CFM head（高清，同 DP backbone）仅"DDPM→rectified-flow 目标"vs E1：FM head 净收益
E3CFM 推理步数扫 1/2/4/8 Euler仅推理步数速度/精度权衡（FM 的真卖点）
E4（可选）ACT 高清对照—FM(高清) vs ACT(高清) = 终极问题
v1（赢了才做）consistency-FM(1步) / A2A 历史源 / streaming逐个加速度或多模态增益

Fair A/B matrix: DP-lowres → DP-highres → CFM-highres(same backbone) → ACT. 只有走完才知道 FM 是否真有位置。

6. 实现草图 minimal CFM head = a small diff on lerobot DP

已落地：LeIsaac/FlowDP submodule（git@github.com:vitorcen/FlowDP.git）。 FlowMatchingModel(DiffusionModel) 只覆写 compute_loss（DDPM→线性插值速度场）+ conditional_sample（DDIM→Euler）， encoder/UNet/processor/normalization 全继承；靠 lerobot 命名约定让 processor_flowdp 自动被发现， train.py 补丁 get_policy_class 注册 --policy.type=flowdp。启动 scripts/train_e2.sh。

DDPM 与 rectified-flow 在 UNet 输出形状完全相同（都是 (B,T,6)），改动集中在三处：

# 1) 训练目标：噪声预测 → 直线速度场（rectified flow / linear CFM）
#    x0 = 高斯噪声, x1 = 真实 action chunk;  x_t = (1-t)*x0 + t*x1
#    target velocity  v = x1 - x0   （时间无关，直线）
t   = torch.rand(B, 1, 1, device=dev)            # U(0,1)
x0  = torch.randn_like(actions)
xt  = (1 - t) * x0 + t * actions
v_pred = unet(xt, t.squeeze(-1), global_cond=obs_feat)   # 复用 DP 的 FiLM-UNet1D
loss = F.mse_loss(v_pred, actions - x0)          # 唯一变化：target 从 noise 改成 (x1-x0)

# 2) 推理：Euler 积分 N 步（N=1/2/4/8 扫），替代 DDIM 32-step
x = torch.randn(B, T, 6, device=dev)
for i in range(N):
    tt = torch.full((B,), i / N, device=dev)
    x  = x + (1.0 / N) * unet(x, tt, global_cond=obs_feat)
action_chunk = x                                 # 绝对关节角，直接喂 serve 适配器

# 3) 视觉：去掉 resize 到 240×320，保留 ≥448 路径（E1/E2 的关键控制变量）

数据加载、归一化、WebsocketPolicyServer/ServicePolicyClient、benchmark harness 全部不动—— 这正是"自研最小 head"相对"移植 A2A"的最大优势：新增耦合为零，且与 DP 形成单变量对照。

7. 不做 / 范围外

  • 不碰点云/深度派（PointFlowMatch、FlowPolicy）——模态不符，除非加深度传感器（不在范围）。
  • 不在 v0 引入 consistency-FM / streaming / A2A 历史源——先隔离"目标函数"单变量。
  • 不移植任何研究 dump 的数据管线（zarr/RoboVerse）——保持 LeRobotDataset 原生。
  • 不先写 FM——先做 E1（DP 高清重测），否则 baseline 不诚实。
