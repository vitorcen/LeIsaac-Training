DreamZero v2 训练方案 — 14B 视频骨干 + SO-101 救活计划 DreamZero v2 finetune recipe — closing the gap to GR00T on 50-demo SO-101 PickOrange

DreamZero Wan2.1-I2V-14B LoRA SO-101 PickOrange v2-finetune v1: 0/3 placed @ ckpt-5k target: ≥ GR00T 13/15

TL;DR — 这是设计 draft，不是承诺
v1 ckpt-3k/4k/5k 全 0/3 placed，GR00T-N1.7 同 50 demos 6500 step 拿 13/15。但 Vizuara 公开 baseline (HF dreamzero-so101-lora) 我们也实测过"也很差"（user 主动终止 eval）→ 这意味着 架构层面在 50-demo SO-101 上本来就难启动，不是单纯 v1 hyperparam 没调对。

v2 提出的 4 个改动（Vizuara head init / horizon sweep / REPA ablation / 分组 LR）都是带验证条件的假设，不是"按下就拿 5/15"的菜谱。每个变量必须先 single-variable 跑 5-round strict ≥ placed=1 才算证伪 null hypothesis。 This is a design draft, not a promise. v1 fails (0/3), Vizuara public baseline also fails when we ran it. Each v2 change is a hypothesis with a falsification gate (5-round strict, placed≥1), not a recipe.

1. 根因分析 — 为什么 14B < 1.7B (Root cause: why bigger lost)

1.1 数据结构层 (Data-structure perspective)

"Linus 三问"自检：
真问题还是假想？ 真问题。v1 三个连续 ckpt 全 0/3，arm barely moves，是 hard fail 不是 noise。
有更简单方案吗？ 有 — 不重做整个 pipeline，只换初始化 + 砍 horizon + 关 REPA 三处。
会破坏什么吗？ Vizuara head shape 需对齐 (max_state_dim=64, max_action_dim=32 一致 ✓)，其余都是 hyperparam 改动，零破坏。

1.2 特殊情况层 (Edge cases v1 fell into)

  • random head + frozen-ish DiT：head 从零，LoRA 只调 19M / 14B = 0.14% → 梯度信号几乎全压在 head 上，但 head 容量 (hidden=64) 远不够吸收
  • horizon=24 + 50 demos：每个 chunk 24 帧目标 → 优化器学的是"24 帧平均动作"，对 imitation 等价于 mode collapse 到训练集均值（接近静止）
  • REPA coeff=1.0：repa loss 量级与 action loss 相当 → 早期 action loss 还没下降，REPA 已在拉视频表征，head 学不到 visuomotor 映射

1.3 复杂度层 (Complexity reduction)

v1 同时在做 4 件事：LoRA 适配 + 动作学习 + 视频动力学约束 (REPA) + 长时域块预测。 对 50 demos，"先做 1 件 (动作)，做对了再叠加其他" 是唯一稳妥策略。

1.4 Vizuara head 复用悖论 (The Vizuara reuse paradox — must address head-on)

事实：Vizuara 公开 ckpt 完整推理 (含 DiT LoRA + head + REPA) 在 LeIsaac 上"也很差"，user 主动终止 eval。
v2-A 提议：从 Vizuara safetensors 抽 仅 action_head 部分 初始化，丢掉 Vizuara 的 DiT LoRA。
悖论：如果整个 Vizuara model 都不 work，凭什么相信"只用它的 head"反而能 work? If full Vizuara fails, why would Vizuara-head-only init succeed?

3 种可能的解释（按可信度排序）：

  1. (50% likely) head 学到了 SO-101 关节 / gripper 量纲先验。action_decoder 输出的 32-dim 包含 5 关节 + gripper + padding，从 random 学这个量纲映射本身需要数百 step，Vizuara 已替我们走完这段 — 即使 head 上游 (DiT hidden) 分布不同，量纲先验仍可能加速 v1 stuck 期出零动作的过渡。验证：v2-A 前 500 step 看 action L1 norm，>0.05 → 至少 head bias 有效
  2. (30% likely) head init 是 random + Vizuara 折中的 noise，效果跟 random 持平。验证：v2-A ckpt-500 与 v1 ckpt-500 横评，placed 差 ≤1 → null hypothesis 不拒
  3. (20% likely) head init 主动有害（Vizuara head 期待的 DiT hidden 分布与 v2 frozen Wan 完全不同 → head 输出反向漂移）。验证：v2-A 前 500 step arm 不动 / 动作发散 → 立即放弃此条变量
Decision gate：v2-A 跑前 500 step quick eval + action norm 监控，三种解释中只要 #1 或 #2 出现就继续；#3 出现立即 abort 这条变量。不允许在没 gate 的情况下 5k step 闷头跑（v1 教训）。

2. v2 方案三大主轴 (v2 three primary changes)

2.1 主轴 #1 — Vizuara action_head 初始化 (Init from pretrained robot head)

纠正前版本错误结论：原 doc 写 "Vizuara r=4 能 work → 架构没问题"。事实是 Vizuara 本身也不 work（见 §1.4）。重新表述：

修正后的假设：DreamZero 架构 + DiT LoRA + REPA 这一整套在 50-demo 上系统性难启动。 v2-A 不是"复刻 Vizuara 配方"，而是试试用 Vizuara head bias 给 random init 一个非零起点，看是否能缩短 v1 前 2k step 的 stuck 期。 v2-A is NOT "reproduce Vizuara recipe" (Vizuara also fails). It's "use Vizuara head as a non-zero warm-start to escape v1's stuck phase faster".

2.2 主轴 #2 — action_horizon sweep（不是 24→8 一刀切）(Horizon sweep, not a fixed reduction)

纠正前版本错误：原 doc 用"horizon=24 害人 / 起点数 ×3 / mode collapse" 论证砍到 8。横评数据反着站：
  • GR00T-N1.7 action_horizon = 40 → 14/15 placed 🥇
  • GR00T-N1.6 action_horizon = 50 → 13/15 placed
  • ACT action_horizon = 100 → 8/15 placed
  • SmolVLA / X-VLA / N1.5 horizon 32-50 均能 work
结论：长 horizon ≠ mode collapse；horizon=24 不是 v1 的显著嫌疑。砍到 8 没有先验证据支持。
修正方案：horizon sweep，不预设答案

v2-B 跑 4 个 horizon ∈ {8, 16, 24, 40}，其余 hyperparam 锁定为 v2-A 的胜出配置。
每个跑到 ckpt-2k，做 5-round strict eval，比较 first-placed step + 5k step placed/15。
如果 horizon=24 不显著差于其他三档，则保持 v1 默认，省一次实验预算 ¥30。

对应 config 改动（仅 sweep 时）：

# groot/vla/configs (Hydra override) — sweep 一组
action_horizon: ∈ {8, 16, 24, 40}
num_action_per_block: 同 action_horizon
num_frames: 33 (保持，独立变量)
# frame_seqlen 由 num_frames × (W/8 × H/8 / patch=4) 计算
# 33 × (40 × 22 / 4) = 33 × 220 = 7260 (原 v1 值，保持)

推理端同步影响（horizon 改了必须连带改的下游）：

  • server/dreamzero_leisaac/server.py:200 --action-horizon default
  • server/eval_dreamzero.sh ACTION_HORIZON
  • LeIsaac client query 频率 = horizon / step_hz；horizon=8 + 60Hz → 133ms 一次 query，要确认 server 推理时间 < 133ms 不然 stuck
  • policy_timeout_ms = max(8000, horizon × per_step_inference_ms × 1.5)

2.3 主轴 #3 — REPA ablation（不直接关）(REPA ablation, not blanket disable)

纠正前版本错误：原 doc 直接说"REPA 在抢梯度，前期是噪声不是信号"，直接砍掉。这是猜测，没有证据：
  • DreamZero 论文 default repa_coeff = 1.0，作者很可能已 ablation 过
  • Vizuara baseline 默认开 REPA 也"很差"，但不开 REPA 是否更差未知
  • "REPA 在学 image→DINO feature" 这条逻辑链对 imitation 不必然是干扰 — DINO feature 是稳定 visual prior，可能反而帮新场景泛化
修正方案：REPA 二选一对照组，不预设结论

v2-C 单变量跑两个 run：repa_coeff = 1.0 (v1 default) 和 repa_coeff = 0.0，其余配置相同，只跑到 ckpt-2k，5-round strict 比较。
如果差距 ≤ 1 placed：保持 v1 default = 1.0，不改。 如果 0.0 显著更好：再考虑 anneal schedule。

5.1 Pivot 选项（v2 全失败时）

  • Option A: 放弃 14B Wan，复用 X-VLA / GR00T-N1.6 验证过的 1.7B-scale VLA 架构（已知能 work）
  • Option B: 增加 demo 量（50 → 200+），看是否是数据 bottleneck
  • Option C: 联系 DreamZero 作者问 Vizuara baseline 是否原本就不为 sim eval 设计 (可能 real-robot 才达标)
