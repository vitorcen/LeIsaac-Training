Diffusion Policy 微调记录

SO-101 PickOrange · LeIsaac benchmark baseline #2 · 2026-05-18

Diffusion Policy (Chi et al. 2023) — image-conditioned action denoising via a temporal UNet1D. ~267M params, no VLM, no language conditioning. Strongest "plain" baseline before VLA-class models.

0. TL;DR

DP 是 mid-size baseline（267M params，DDIM 32-step inference ~147ms/chunk）。 本地 ckpt wsagi/DiffusionPolicy-PickOrange strict 表现 0/3 round, 2/9 oranges, avg 108s, 10.6 GB， 关键工程提速：训练用 DDPM 100-step，部署 swap 为 DDIM 32-step（无需重训）让 latency 393ms → 147ms。

1. 架构核心

组件来源实现参数估算
Vision encodertorchvision ResNet18vision_backbone="resnet18" + GroupNorm 替 BatchNorm + SpatialSoftmax keypoints；可选 per-camera 独立 encoder~11M × N_cam
SpatialSoftmaxLevine et al.把 feature map (B, 512, h, w) → (B, K=32, 2) 个 (x, y) keypoint，再 flatten 成 64D 全局特征~0M
FiLM-conditioned UNet1DDP 自研（modeling_diffusion.py:628 DiffusionConditionalUnet1d）沿 chunk 时间维做 1D 卷积；条件经过 FiLM 注入到每个 conv block；4 stage down + 4 stage up；conv kernel=5~250M
Time embeddingSinusoidal + MLP把扩散时间步 t 编码到 condition vector~0.1M
合计——~267M（全部 trainable，无 LoRA）
train: DDPM T=100 • deploy: DDIM T=32 → 393ms → 147ms 无需重训，热 swap cond → FiLM action_noisy ⚠ must_go trap DP chunk=8 太短 → 抓到 orange 后 model 输出 hold-pose chunk + obs 跟之前几乎一样 → 触发 client dedup → 不发新 obs → server 永远拿同个 obs → 同 chunk → grasp-and-freeze attractor 修复: must_go=True 强发 obs

读图要点：

  • 无 VLM 无 language：DP 是 image-conditioned 纯视觉策略，跟 ACT 一样
  • UNet1D 沿 chunk 时间维做 1D 卷积（不是 transformer），FiLM 在每个 conv block 注入条件（visual+state+t_emb）
  • 训练 DDPM 100-step → 部署 DDIM 32-step，无需重训直接 swap：4090 上 393ms → 147ms（公式 (target_ms - 36) / 3.3 ≈ 29 step）
  • n_obs_steps=2：DP 看 past 2 frame 而不是 1 frame，提供 implicit velocity 信息
  • SpatialSoftmax：经典 manipulation policy 套路，把 feature map → keypoints 让网络专注空间位置而不是 texture
  • must_go trap（我们踩过的）：chunk=8 短 → 抓握后 hold-pose chunk → client dedup → 永远不发新 obs → 卡死。修：long-chunk policy must_go=False，short-chunk 必 True

5. 已踩过的坑

  • DDPM 100-step latency 393ms → 部署后 sim 跟不上，必须降到 DDIM 32-step（详见 dp_inference_speedup_and_dynamic_timeout.html）
  • must_go=False 时 grasp-and-freeze：chunk=8 抓到后 hold-pose，obs 不变，client dedup（atol=1），server 永远拿同个 obs → 死循环。修复：DP 必 must_go=True，长 chunk policy（ACT 100, SmolVLA 50）must_go=False
  • step_hz=60 与 dataset 30Hz 错配：部署时间尺度漂移，arm 速度加倍。修：step_hz=30

6. 重训计划（2026-05-22 起）

Quick eval 中止条件

  • 连续 3 个 slice (e.g. 30%, 40%, 50%) oranges=0 → abort，policy 学到 degenerate "do nothing"
  • 连续 3 个 slice arm stuck < 30s/ep → abort，diagnostic 必须先做：dump action chunk 看 server 实际返回什么
  • 正常应在 30k step 看到 oranges > 0（ACT v0.4.0 6k step 就开始 placement）

如果 full 480×640 仍 0/15

说明 DP 失败不在分辨率，而在更深的 architecture / sampling 问题。逐项排查：

  1. DDIM swap：训练 DDPM 100-step → 部署 DDIM 32-step，inference 393ms → 147ms。延迟过高 → arm 跟不上 chunk，可能学习 OK 但部署崩
  2. must_go semantics：DP n_action_steps=8 短 chunk，server dedup 会冻 obs。debug 时强制 must_go=True 看是否解锁
  3. n_obs_steps=2 客户端兼容性：dump client 实际发送的 obs，确认 server 真的拿到 2 帧而不是 stack 同一帧
  4. 降到 n_action_steps=1：单步执行最稳，没 chunk hold-pose 问题。代价是 inference 频率 30Hz 全程跑
  5. action normalize 检查：DP 用 MIN_MAX or MEAN_STD？跟 ACT 比对

7. v040-fullres 实测结果 + 真 root cause (2026-05-23)

TL;DR — DP "policy dead" 不是 model dead，是 lerobot async server 的一个 stack 空 bug。修了 bug 后 DP 能产合理 action，但 50 demo 仍学不到稳定 task。

DP "policy dead" was a lerobot async-server bug, not a model failure. After patching, actions are valid but the model still does not learn a robust task on 50 demos.

  • 100k step ≈ 3h on RTX 4090, loss 0.5 → 0.014, cosine lr 1e-4 → ~1e-10

真根因：lerobot async server populate_queues bug 🐛

Eval 启动后 server log 立刻报：

ERROR Error in StreamActions: stack expects a non-empty TensorList

Client log：

[CLIENT] no actions after 8 retries (200ms); reusing last action

翻代码：lerobot/policies/diffusion/modeling_diffusion.py

@torch.no_grad()
def predict_action_chunk(self, batch, noise=None):
    # 直接 stack 空 deque → crash
    batch = {k: torch.stack(list(self._queues[k]), dim=1) for k in batch if k in self._queues}
    ...

@torch.no_grad()
def select_action(self, batch, noise=None):
    # 正常路径：populate_queues 在前
    self._queues = populate_queues(self._queues, batch)
    ...

select_action 调 populate_queues（自带 cold-start 复制逻辑：首帧来时复制 n_obs_steps 份填满 deque），但 async server 用的 predict_action_chunk 跳过了这一步 → 首次 inference 必崩。v0.4 和 v0.5 都有这 bug。

Both v0.4 and v0.5 have this. select_action populates queues, but the async-server entry predict_action_chunk does not — so first inference with n_obs_steps>1 always crashes.

One-line patch

@torch.no_grad()
def predict_action_chunk(self, batch, noise=None):
    if self.config.image_features:
        batch = dict(batch)
        batch[OBS_IMAGES] = torch.stack([batch[key] for key in self.config.image_features], dim=-4)
    self._queues = populate_queues(self._queues, batch)
    batch = {k: torch.stack(list(self._queues[k]), dim=1) for k in self._queues if k != ACTION}
    actions = self.diffusion.generate_actions(batch, noise=noise)
    return actions

已 apply 到 /home/david/work/lerobot-v040 editable install。建议向 upstream PR — 任何 n_obs_steps>1 的 lerobot policy (DP / VQBet / 等等) async server eval 都受影响。

Patched 后 server 表现

INFO Running inference for observation #N (must_go: False)
INFO Preprocessing and inference took 0.6840s, action shape: torch.Size([1, 8, 6])
INFO Action chunk #N generated | Total time: 727ms

合理 action 出来了（DDPM 100-step + 全 res → 727ms/chunk，client retry budget 1600ms 够），joint range ±0.6 rad 也合理。但执行后 ckpt-70k 5-round 仍 0/15 → bug fix 是 necessary not sufficient。

错过的两个假设

  1. "crop_shape=(84,84) 是根因" — 看 v0.4 v0.5 default 一致 + 现象一致就假设是它，没继续追 server log。crop 确实是 secondary bug 也得修，但不是 stuck @ 33s 的真因。
  2. "DP 学不动是数据少" — bug fix 后确实学不动，但 actions 是有学到 base structure (joint motion in range)，只是无法稳定完成 task。这个跟 wsagi v0.5 公开 ckpt 0/15 一致，说明上限 ≈ 0/15。

结案

  • DP v040-fullres patched = 0/15，跟 wsagi v0.5 ckpt 一致
  • 主要价值：定位 lerobot async server stack-bug，patch fix 让所有 n_obs_steps>1 lerobot policy 重获新生，可向 upstream 贡献
  • 下一个 DP 变种：如要再试，路线建议 n_obs_steps=1 + crop=null + DDIM 32-step（避开 stack bug + 更快 inference），代价 10h 重训，~50% 概率 > 0/15
