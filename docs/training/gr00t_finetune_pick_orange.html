GR00T N1.5 / N1.6 微调计划

SO-101 PickOrange · LeIsaac benchmark champion (N1.6 🥇 2/3) · 2026-05-18 · 2026-05-20 v2 (X-VLA 经验回灌 + Isaac-GR00T native path 锁定)

NVIDIA Isaac GR00T N1.5/N1.6 — Eagle2 VLM + multi-embodiment flow-matching DiT action head, ~3B params. Already top of the leaderboard via the LightwheelAI public ckpt; the next obvious lever is in-house finetune to push from 6/9 to 8/9 oranges.

0. TL;DR

GR00T-N1.6 是当前 leaderboard 🥇（LightwheelAI ckpt 实测 2/3 strict round, 6/9 oranges, 96s avg, 17.3 GB VRAM）。 架构：Eagle2-VLM (~2.4B, frozen) + Flow-matching Action DiT (~600M, trainable)，3B 总参数。 N1.5 是上一代（0/3, 4/9，但已 commit 落盘 8 颗其中 4 strict）；N1.6 用更新的 Eagle2.5 backbone + 优化 action head。

路径决策（2026-05-20 v2）：用 Isaac-GR00T native launch_finetune.py，不走 lerobot groot wrapper。 理由：lerobot wrapper 硬编码了 GR00TN15（N1.6 不兼容）；native 路径是 LightwheelAI SOTA ckpt 的同款 pipeline；推理直接复用 server/start_server.sh --gr00t-only + nvidia/GR00T-N1.6-3B，0 协议改动。

下一步：N1.6 base 单 4090 全参 bf16 微调（trainable = projector + DiT + top4 LLM layers ≈ 600M），save_steps=500，自动 6-round eval-on-save，预期 2/3 → 3/3。不上 LoRA（Isaac-GR00T native 不暴露 lora_rank；显存够，不必）。
📌 X-VLA session 五条普适经验已回灌到本 doc（详见 §3.2 推理 / §3.3 训练 / §5 Phases）：
  1. Prompt 锁死 "Pick up the orange and put it in the plate"（避开 "Grasp" / "all" / "on the plate"）
  2. 训练不加外置 aug（N1.6 自带 color_jitter + albumentations + crop；X-VLA 50-demo 加 aug 普遍 -10%）
  3. EMA / TAE 不要做（X-VLA RF head 上全部证伪，GR00T 同为 flow-matching DiT，预期同样无效）
  4. n_action_steps sweep 而非锁定（X-VLA 默认 8 → 0/18 灾难；GR00T chunk=50，sweep {8, 16, 25, 50}）
  5. Eval 必须 ≥6-round (18 ep)，决策需 ≥12-round (≥36 ep)；3-round 方差 ±15-30%

1. 架构核心

组件来源实现参数估算

Cross-attention DiT action headGR00T 自研（cross_attention_dit.py:204）N 层 BasicTransformerBlock：每层先 self-attn(action tokens)，再 cross-attn(action, VLM features)，AdaLayerNorm 注入 timestep；输出 noise/velocity prediction；默认 tune_diffusion_model=True 可训~500M

合计——~3B（VLM frozen, trainable ~540M）
cross-attn(action, VLM features) action=Q, VLM=K/V — 经典 DiT ⑤ AdaLayerNorm ⑥ MLP (4× expansion) 类似 DALL-E 2 / SD 3 的 cross-attn multi-embodiment 关键： CategorySpecific 让不同机器人有独立 norm Action decoder CategorySpecificMLP → (B, 50, 32) 取前 6 维 = SO-101 Flow-matching loss L = ||v_pred − (ε−a)||² 仅在 action tokens 推理 (inference) • x_T = N(0, 1) chunk=50 • VLM forward 1 次 　缓存 VLM features (重要!) • for i in range(steps): 　 DiT cross-attn → v_pred 　 x_t ← scheduler.step() → ~80-100ms / chunk @ 4090 • n_action_steps=50 整个 chunk 跑完再 query step_hz=30 / 50 = 0.6Hz query state cat_id VLM features → KV action_noisy + t N1.5 vs N1.6 差异 N1.5 (nvidia/GR00T-N1.5-3B): • Eagle2 vision • 我们 benchmark 0/3, 4/9 N1.6 (nvidia/GR00T-N1.6-3B): • Eagle 2.5 vision • image augmentations • 优化 action head • 🥇 2/3, 6/9 (currently)

读图要点：

  • GR00T 是 flow-matching VLM-conditioned action expert 范式（同 π0.5 / SmolVLA / X-VLA / FastWAM），但 action expert 是 独立的 cross-attn DiT（不与 VLM 共享 KV，是从 VLM features 用 cross-attn 拉信息）
  • Multi-embodiment 关键：CategorySpecificLinear/MLP 让不同机器人（cat_id）有独立的 fc 权重（同 X-VLA 的 DomainAwareLinear 思路），embodiment_tag="new_embodiment" 会自动分配一个 cat_id
  • VLM 一次推理，DiT cross-attn 拉它的 features 做 N 步 denoise，跟 π0.5 一样高效
  • chunk_size=50, n_action_steps=50：1.67s open-loop @ 30Hz — 但 GR00T 50 步比 X-VLA 32 步还长，靠 multi-embodiment 预训练弥补
  • 训练 default：tune_llm=False, tune_visual=False, tune_projector=True, tune_diffusion_model=True — Eagle2 全冻，只训 projector + DiT action head

3. 微调计划 — Isaac-GR00T native finetune from N1.6 base

为什么 Isaac-GR00T native 是合理下一步：
  • N1.6 公开 ckpt 已 2/3 → 离 3/3 只差关键样本细节（SO-101 specialist tuning）
  • launch_finetune.py 是 LightwheelAI 6/9 SOTA ckpt 的同款 pipeline，已验证
  • ~/work/Isaac-GR00T/examples/SO100/ 提供 SO-101 兼容模板（同 5+1=6D arm+gripper）
  • 推理直接复用 server/start_server.sh --gr00t-only（已 zero-modify 接入 LeIsaac client）
  • 预期：从 multi-embodiment generalist → SO-101 specialist，6/9 → 8/9 合理目标
❌ 为什么不用 lerobot groot wrapper（之前 v1 草稿的错误方向）：
  • lerobot/policies/groot/modeling_groot.py:85 硬编码 GR00TN15.from_pretrained，N1.6 base 不能加载（Eagle2 vs Eagle2.5 schema 不匹配）
  • lerobot config max_state_dim=64, max_action_dim=32 与 N1.6 base 128/128 不一致，需要 patch lerobot submodule
  • 原 v1 推荐的 lora_rank=32 也只在 lerobot wrapper 暴露；native path 不支持 LoRA（也不需要）

3.3 GPU 资源估算（单 4090 24GB）

项bf16 全参（推荐）~8-bit base + bf16 train（不建议）
Eagle2.5 base (≈2.0B frozen + top4 LLM layers trainable)~4.0 GB params + ~0.5 GB grads~2.0 GB (8-bit) + 0.5 GB grads
DiT action head + projector (≈500M trainable)~1.0 GB params + ~1.0 GB grads~1.0 + 1.0
Adam optim (trainable ≈600M × 8 bytes)~4.8 GB~4.8 GB
Activations (batch=2, chunk=50, bf16)~3-4 GB~3 GB
合计 (per-step batch=2)~13-15 GB ✅ 4090 安全~11-13 GB ⚠️ bnb flaky
⚠️ 实测显存校准 4 次 OOM + 1 次成功 (2026-05-20 实战)：
尝试per_stepgrad_ckptbb_fp32optim结果
18✗✓adamw_torchOOM 22.35GB ❌
22✗✓adamw_torchOOM 22.35GB ❌
34✓✓adamw_torchOOM 22.35GB ❌
42✓✗adamw_torchOOM 22.36GB ❌
52✓✗paged_adamw_8bit✅ loss 1.07→0.21 over 500 step；峰值 24000 MiB
关键发现：N1.6 静态显存（Eagle 2.4B + DiT 600M trainable + bf16 master + Adam state）≈ 22 GB。激活根本不是瓶颈（per_step=8 和 2 都 OOM 在同一个 22.35 GB）。唯一能救的是 8-bit optim state（paged_adamw_8bit, bitsandbytes 0.49.2），把 32-bit Adam (~4.8 GB on 600M trainable) 压成 8-bit (~1.2 GB)，省 ~3 GB 给激活。

3.3.1 最终敲定配方（2026-05-20 v3）

# LeIsaac/scripts/finetune/gr00t/launch_finetune_ckpt.py 中固定写入：
config.model.backbone_trainable_params_fp32 = False  # save ~3 GB master fp32
config.training.gradient_checkpointing       = True  # save ~half activations
config.training.optim                        = "paged_adamw_8bit"  # save ~3 GB

# train.sh env：
GLOBAL_BATCH=16
GRAD_ACCUM=8                # per-step = 2
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True

显存预算（实测峰值 24000 MiB / 24 GB）：

  • Eagle bf16 base (frozen, ~2.4B): ~4.8 GB
  • top4 LLM layers + DiT trainable bf16 (~600M): ~1.2 GB params + ~1.2 GB grads
  • paged_adamw_8bit state (~600M × 1 byte × 2): ~1.2 GB
  • Activations (grad-ckpt halved, per_step=2, chunk=50, 448×448): ~14-15 GB
  • 合计 ≈ 22.4 GB，4090 上跑得动但留白小（建议训练时不要开其他 GPU 进程）
📌 注意：训练时 4090 几乎打满（24000 MiB / 24576 MiB），不能并行跑 GR00T 推理 server 做 auto-eval。下游 closed-loop eval 必须等训练完成后串行跑（与 X-VLA workflow 一致）。
⚠️ exit 139 偶发 SIGSEGV：尝试 5 在 ckpt-500 save 完后进程 exit 139（无 Python traceback）。怀疑是 bnb 0.49.2 在 24GB 峰值时 paged 分配的 cuda kernel 偶发崩。HF Trainer auto-resume 完美兼容：删 ckpt 之外的孤儿文件后重启 train.sh，会自动从 latest checkpoint-N 续。
对应策略：把 train.sh 包一层 watchdog，detect exit 139 后自动重启（沿用 OpenVLA session 同款方案）。

3.4 训练 launcher 草稿（最终版）

# LeIsaac/scripts/finetune/gr00t/train.sh
# 复用 Isaac-GR00T native launch_finetune.py + uv venv.

GR00T_ROOT="${HOME}/work/Isaac-GR00T"
DATASET_DIR="${HOME}/work/isaaclab-experience/LeIsaac/datasets/raw/leisaac-pick-orange"
OUTPUT_DIR="${HOME}/work/isaaclab-experience/LeIsaac/outputs/gr00t-n16-leisaac-pick-orange"

SAVE_STEPS="${SAVE_STEPS:-500}"
GLOBAL_BATCH="${GLOBAL_BATCH:-32}"
GRAD_ACCUM="${GRAD_ACCUM:-4}"          # per-step batch = 32/4 = 8

cd "$GR00T_ROOT"
CUDA_VISIBLE_DEVICES=0 uv run --extra=gpu python \
    gr00t/experiment/launch_finetune.py \
        --base_model_path nvidia/GR00T-N1.6-3B \
        --dataset_path "$DATASET_DIR" \
        --modality_config_path "$MODALITY_CFG" \
        --embodiment_tag NEW_EMBODIMENT \
        --num_gpus 1 \
        --output_dir "$OUTPUT_DIR" \
        --save_steps "$SAVE_STEPS" \
        --save_total_limit 25 \
        --max_steps "$MAX_STEPS" \
        --warmup_ratio 0.05 \
        --weight_decay 1e-5 \
        --learning_rate 1e-4 \
        --global_batch_size "$GLOBAL_BATCH" \
        --gradient_accumulation_steps "$GRAD_ACCUM" \
        --dataloader_num_workers 4 \
        --shard_size 1024 \
        --num_shards_per_epoch 100000 \
        --episode_sampling_rate 0.1

3.5 Auto-eval-on-save 循环

训练 background 跑，同时 scripts/auto_sweep_gr00t_ckpts.sh 监控 $OUTPUT_DIR/checkpoint-*，每发现新 ckpt 就：

  1. 启动 server/start_server.sh --gr00t-only，GR00T_MODEL_PATH=<ckpt>
  2. 跑 6-round eval（EVAL_ROUNDS=6 EPISODE_LENGTH=60 MAX_ROUND_WALL_S=90）
  3. 解析 metrics JSON，append 一行到 logs/gr00t_ckpts.csv
  4. kill server，继续等下一个 ckpt

训练不中断；eval 串行在 server pause 间隙跑（每 ckpt ≈ 90s × 6 = 9 min 评测；训练每 500 step ≈ 25-30 min @ 4090，eval 不会撞车）。
