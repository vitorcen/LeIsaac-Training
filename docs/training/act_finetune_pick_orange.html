ACT 微调记录

SO-101 PickOrange · LeIsaac benchmark baseline #1 · 2026-05-18 → 2026-05-22 锁版本 v0.4.0 重训后翻倍

Action Chunking Transformer (Tonyzhao et al. 2023, ALOHA paper) — the simplest and cheapest baseline in the leaderboard: ~80M params, no VLM, no diffusion, just a ResNet18 + Transformer encoder-decoder predicting an action chunk in one forward pass.

0. TL;DR

ACT 是 leaderboard 里最便宜的 baseline（52M params，~10ms/chunk forward）。
2026-05-22 锁 lerobot v0.4.0 重训后 8/15 oranges (ckpt-20k h=70, 5-round) — 超过 shadowHokage 真实水平 (3-4/15，旧 9/15 anchor 2026-05-22 复测未能复现)，2.7× 旧 v0.5.2 训的 3/15。
关键发现两个：
  1. Framework drift：lerobot v0.4→v0.5 间 PR #3406 (dataloader uint8/persistent_workers) + PR #3442 (ACT padding loss fix) 是 root cause，shadowHokage 上传时间在两 PR 之前 — 详见 README 脚注 与 memory act-framework-drift-root-cause
  2. Chunk-execution horizon trap：默认 h=16 跑不动，v0.4.0 ckpt 最佳 h=70（旧 v0.5.2 ckpt 是 h=32）— 每个模型自己的最优 horizon 不一样

1. 架构核心

组件来源实现参数估算
Vision backbonetorchvision ResNet18vision_backbone="resnet18" + ImageNet pretrained；可选 final stride→dilation；feature map flatten + projection 到 dim_model~11M
Transformer encoderACT 自研（modeling_act.py:514）4 层标准 multi-head self-attention，非 causal（全 bidirectional）；输入：[CLS, image patches × N_cam, state, (训练时) latent z]~10M
Transformer decoderACT 自研（modeling_act.py:572）1 层 decoder block（原 paper 写 7 但代码 bug 只用首层，lerobot 复刻该行为）；query = learnable positional embedding（chunk_size 个）；KV = encoder output；输出每个 query → 1 个 action~4M
CVAE encoder（训练 only）ACT 自研训练时把 GT action chunk 编码成 latent z；推理时 z = 0；提供"action prior"正则~10M
合计——~52M（lerobot 实测 num_learnable_params=51,597,190；全部 trainable，无 LoRA）

读图要点：

  • 没有 VLM：ACT 是纯 vision-conditioned policy，不接 language。任务身份由训练 dataset 决定，inference 时无 prompt
  • CVAE 是 trick 不是核心：训练时把 GT action 编成 latent z 提供正则，推理时 z=0；理论上简化版（去掉 CVAE）也能跑，paper 报 z=0 + KL 退火帮助 small-data 收敛
  • 单 forward 出整个 chunk（chunk_size=100）：跟 ACT/DP/SmolVLA 一脉相承，但 ACT 是 直接 regression，无 diffusion，最便宜。10ms 跑完一个 100-step chunk = 等效 0.1ms/action
  • Temporal ensembling（可选）：每 sim step 都 forward，把多个过去 prediction 加权求平均 chunk[0] → 抗噪；我们 benchmark 没开，开后 latency 会上去

为什么 shadowHokage 用 h=16，自训 v0.4.0 用 h=70？
baselines.tsv 规则："horizon 必须 per-ckpt sweep 实测，不能套统一公式 — train chunk_size 是上限不是 best"。每个 ckpt 的最优 horizon 由它自己的 gradient 轨迹决定 — 同架构同 dataset，但 framework drift 导致 chunk 内 action 一致性不同，最优 replan 频率自然不同。
2026-05-22 实测验证：shadowHokage h=70 5-round = 4/15（vs h=16 = 9/15），反向证明 h=70 是 v0.4.0 自训 ckpt 专属最优，不能套到 shadowHokage。这恰恰是 framework drift 真实存在的旁证 —— 同架构同 dataset 但训练曲线分叉，导致两个 ckpt 的最优 replan 频率截然相反。

3 个数字关系（2026-05-22 复测后更新）：

  • v0.4.0 8/15 ≫ shadowHokage 真实 3-4/15 — 不是"追平 band"，是 2× 超过 shadowHokage real score。旧 shadowHokage 9/15 anchor 2026-05-22 同 pipeline 复测未能复现（3/15），证明是 anchor outlier / artifact
  • v0.4.0 8/15 ≫ v0.5.2 3/15 — 同 dataset / 同 seed / 同 config，唯一变量是 lerobot 版本 → 2.7× 差距
  • v0.5.2 3/15 ≈ shadowHokage 真实 3-4/15 — 后期 framework 训出来的 ACT 上限在 3-4/15 一带，v0.4.0 重训 8/15 才是真正跳出 drift 上限
  • ckpt-18k h=70 single-run 13/15 — 是 lucky run（3-run mean 7.7/15 ±35%），不作 canonical；canonical 用 ckpt-20k h=70 = 8/15
  • 5-round variance 在 ACT 上 ±40%（h=16 跨 2 次复测 = 9 vs 3），eval-5round-mandatory memory ±20% 估计偏低

Leaderboard 位置：第 3 梯队（VLM 之后 / Diffusion 之上）。完整对比见 LeIsaac/README.md：

  • 🥇 hi-space GR00T-N1.7 = 14/15 (3B + Cosmos VLM)
  • 🥈 自训 GR00T N1.6 = 13/15 (3B + Eagle VLM)
  • 🥉 shadowHokage / 自训 ACT v0.4.0 = 8-9/15（本文档；52M 纯 vision）
  • hi-space GR00T N1.6 / 自训 SmolVLA 15k = 8-9/15
  • LightwheelAI N1.5 = 8/15

原因：52M params + 30Hz fine motor 任务对 ACT 来说在抓握精度上欠拟合（同样数据 GR00T-N1.6 3B 能 13/15）。但锁版本后 ACT 已经跟 SmolVLA / N1.5 / N1.6-hispace 三个 ≥2B VLA 持平 — 性价比之王。

5. v0.4.0 训练问题根因 / Root cause of lerobot framework drift

After three weeks of debugging (torchcodec, ckpt sweep, sim_warmup, step_hz alignment all useless), the gap between our 3/15 and shadowHokage 9/15 traced to lerobot v0.4 → v0.5 framework drift. Three-model joint analysis (Codex gpt-5.4 + Opus) converged on commit-level root cause.

5.2 两个致命 PR 详解

PR #3406（dataloader）PR #3442（padding loss）
commita8b72d96 2026-04-191add4606 2026-04-23
v0.4.0 行为
（shadowHokage 训练时）		 DataLoader(prefetch_factor=2, persistent_workers=False)；视频 decode 直出 float32 / 255 在 dataloader 内完成 l1 = (F.l1_loss(pred, gt, reduction="none") * ~pad_mask.unsqueeze(-1)).mean()
除以 total count（含 padded 0）
v0.5.2 行为
（我们旧 3/15 训练时）		 DataLoader(prefetch_factor=4, persistent_workers=True) + dataloader 返 uint8，x/255.0 挪到 train loop GPU 上 l1 = (abs_err × valid_mask).sum() / (valid_count × D).clamp_min(1)
只除以 valid count
对训练的影响 mini-batch ordering 完全不同：persistent_workers RNG state 不每 epoch 重置；uint8 → float32 精度损失 ~0.4% per pixel；prefetch_factor 翻倍改变 worker 队列时序 PickOrange chunk=100, ep~605 frames, padding~8% → effective gradient 提升 ~9%；PR 描述 "60-70% padding 时 effective gradient 提升 2-3×"
叠加效果（2026-05-22 ablation 实测） 实测结论（见 5.4 表）：PR #3442 loss fix 单独贡献 +2 oranges，PR #3406 dataloader 无显著增量。
v0.5.2 raw h=70 = 3/15；+ PR #3442 loss fix = 5/15（+2）；+ PR #3406 dataloader = 4/15（noise band，无增量）。
但 v0.4.0 重训 = 8/15，patches 路只补回 2/5 = 40% 的 drift，剩 60% 在依赖侧（torch 2.7.1 / torchvision / cuda12.6 / 其他 commit）— 单靠 PR-level patch 不够。

6.4 Phase 2 — 固定 batch 单步法医对比（仅当 Phase 1 不收敛）

codex 推荐头号方案，工程量大但定位最精。流程：

  1. dump 一份 batch.pt + 初始 state_dict（绕开 dataloader / video decode / worker seed）
  2. 两个 env 各加载同一 batch + 同一初始权重 + 同一 optimizer state，跑 1 step forward+backward+optimizer.step()
  3. 四级对比：
    1. input tensor hash → 数据管线是否同源
    2. 每层 activation mean/std/max → 数值路径分叉点
    3. 每个 param group grad norm → loss/AMP/attention kernel 路径
    4. optimizer.step() 后 weight delta → optimizer/foreach/fused/accelerate 路径

codex 关键提示：同 seed ≠ 同初始化。先 hash 初始 state_dict，可能直接抓到 lazy init / config 解析顺序 drift。

6.5 Phase 3 — 短跑代理指标 commit bisect（仅当 Phase 2 仍未定位）

v0.4.4 → v0.5.0 commit 二分。不用 5-round eval 做 bisect（变量太多噪音太大），改用：

  • 固定 seed + 固定 batch 跑前 200~1000 step
  • 记录 train loss curve + 一个 fixed val batch loss + 第一/最后层 weight delta norm
  • 找第一处曲线明显分叉的 commit

6.6 codex 补充的 5 个角度（按 ROI 排序）

  1. 先 hash 初始 state_dict（便宜，可能直接抓到脏东西）
  2. 强制禁用 flash/mem-eff attention（torch SDPA backend 选路是 ACT transformer 重灾区）
  3. 对齐真实 optimizer step 数：global_step / optimizer_step / scheduler.last_epoch / 实际 seen samples 数
  4. normalization stats cache：action/image normalize 的 stats 加载路径是否变了
  5. 别用 15-ep 成功率做 bisect 信号：加 fixed val loss / train-val gap / rollout 前几步 action L2 这种便宜代理

6.7 优先级与避免做的事

✅ 推荐顺序：Phase 1 (2×2 矩阵, 1-2 天) → 必要时 Phase 2 (法医对比, 1 天) → 必要时 Phase 3 (commit bisect, 2-3 天)。
❌ 避免做：
  • 在没做 2×2 之前再 patch 第 4/5/6 个 PR — patches 路已证伪只补 40%
  • 直接 commit-level bisect — 区间太大，先用 Phase 1 砍掉一半（dep 侧 vs code 侧）
  • 用 5-round oranges 做 bisect 信号 — 噪音 ±20%，会被 3/15 → 4/15 → 5/15 这种垃圾信号拖死
  • 跑 full 20k step ablation — 用 200~1000 step + loss curve 代理更快

7. 统计噪音底线 / Statistical noise floor — 5-round 协议在 ACT 上失效

Pooled-distribution analysis (10 ACT runs × 5 ep = 48 eps; 2026-05-22) shows that current "X / 15 oranges" reporting cannot distinguish 4/15 from 8/15 — the per-episode oranges histogram is bimodal (over-dispersed), not binomial, so 5-round X/15 has ±5-6 noise floor.

7.1 实测分布 vs Binomial(3, p=0.333)

合并 10 个 ACT 5-round runs（shadowHokage h=16/32/70 各 1 + 自训 v0.4.0 ckpt-20k×2 + ckpt-18k×3 + v0.5.2 patched）= 48 episode，per-orange success rate p̂ = 0.333（48 placed / 144 max）。若每颗 orange 独立 Bernoulli(p)，每 ep 应服从 Binomial(3, 0.333)。实测对比：

oranges/ep实际频次Binomial 预期偏离
02114.21.5× 多
11421.30.7× 少
2510.70.5× 少
381.84.4× 多
🔴 分布是 bimodal（U 型），不是 binomial
要么 0 颗全废，要么 3 颗全成，中间状态偏少。物理原因：ACT chunk-policy 输出 100-step trajectory commitment。一旦 first orange pick-place 成功 → model 进入正确 trajectory mode → 后续 2 颗大概率连续成功（cluster）。一旦 trajectory 偏 → 整个 ep 报废。
→ 橙子不独立，按 ep 成 cluster。over-dispersion 直接破坏 binomial 假设，5-round X/15 variance 远大于教科书 binomial。

7.4 Linus 式结论 — 5-round 协议在 ACT 上已失效

🔴 8/15 vs 4/15 在当前 5-round 协议下不显著
  • 所有 10 个 ACT runs（shadowHokage + 自训 v0.4.0 / v0.5.2）大概率就是同一水平（p ≈ 0.33 per orange）
  • per-ep oranges 是 over-dispersed bimodal，X/15 在 5-round 下 variance ±5-6 颗就是噪音底线
  • 所谓 ckpt-18k 13/15 "lucky run" 本质是 ep clustering — 同 ckpt 不同 sim seed 出 [2,3,3,3,2] 类高 cluster 不奇怪
  • framework drift 故事（v0.4.0 vs v0.5.2 训练 loss/grad 数值不同）仍然成立（loss 是 deterministic 物理量），但 5-round eval 协议太弱无法直接检测出端到端 task success 差距
✅ 后续 ACT eval 协议升级方案（ROI 排序）
  1. 主指标改用 per-orange success rate (oranges_placed / 3·n_ep)，不用 X/15 — 直接消除 episode-cluster over-dispersion
  2. round 数 ≥ 15（不是 5），目标 ±10% variance；ACT 当前 5-round ±40%
  3. sim seed 固定，不同 ckpt 跑 same seed → 把 trajectory variance 消掉，只比 ckpt 差异
  4. 报告 95% CI，不报单一 X/15 数字。e.g. shadowHokage h=16 = 3/15 [95% CI: 1/15 – 7/15]，与自训 8/15 [95% CI: 4/15 – 12/15] CI 重叠 = 不显著
  5. per-orange Bayesian 估 p + posterior compare（更严谨，但复杂度高）

Linus 式判读：
- 差距真实存在：per-orange 2.22× ratio，per-ep mean 1.33 vs 0.60，Z test p=0.013（iid 假设）
- 但样本不足以严格证明：per-ep Welch t-test p=0.064，95% CI 重叠，over-dispersion 拉高真实 variance
- 实操结论：再补 1-2 runs each side（累积 25-30 ep per side）就能把 p 压到 < 0.05；或直接切到 per-orange Bayesian 估 p
- 不再让 single-run X/15 数字下结论：5-round 单次的 8/15 vs 3/15 跨 binomial 概率 13% 距离，不构成证据
对前 5 节的影响 / Impact on §1–§6：
- §5.5 "v0.4.0 重训 8/15 vs v0.5.2 3/15 = 2.7× drift" 仍然定性正确，但 "2.7×" 数字本身在 5-round 下不显著；要量化定 drift 大小必须升级 eval 协议
- §5.4 表格的所有 X/15 数字 应改为带 ±5-6 CI 的区间，否则误导
- §6 待验证清单的 "Phase 1 判停规则" 应改：从 "≥7/15 → dep 锁定" 改成 "fixed val loss + first-N-step action L2 收敛到 v0.4.0 水平"（codex 第 5 点已经预警过这点）
