X-VLA QLoRA 微调计划

SO-101 PickOrange · 第 6 个 baseline · pivot from OpenVLA / OFT / LingBot · 2026-05-18

Cross-embodiment soft-prompted Transformer with 0.9B params, natively integrated into LeRobot. Replaces OpenVLA-OFT and LingBot-VLA on the roadmap because of dramatically lower engineering cost and a more direct architectural fit for SO-101 single-arm pick-and-place.

0. TL;DR

为什么 X-VLA：（1）LeRobot 原生集成 — lerobot.policies.xvla.XVLAPolicy 是 PreTrainedPolicy 子类，复用 lerobot trainer；（2）0.9B 参数，bf16 全精度吃得下 24GB 4090；（3）SO-101 已被官方 action_hub 支持（BimanualSO101ActionSpace 单臂可仿写）；（4）chunked + iterative denoising 规避 vanilla OpenVLA 的 step_hz + discrete bin 痛点；（5）cross-embodiment 是设计目标。

Base ckpt：lerobot/xvla-base（3.86k DL / 20 likes）。
训练栈：bf16 LoRA（不必 QLoRA）+ LeRobot 原生 lerobot.scripts.lerobot_train。
⚠️ Codex 评审 (2026-05-18) 抓到 6 个真 🔴，原 v1 plan 不能直接执行。此版已修订 → v2。带行号的源码引用都是 lerobot/lerobot/src/lerobot/policies/xvla/ 下：
  • 🔴 #1 不是 regression，是 iterative denoising (rectified-flow)。modeling_xvla.py:203 训练采样时间 t，action_noisy = noise*t + action*(1-t)，喂模型后算 noise 预测 loss；modeling_xvla.py:249 推理从高斯噪声起步循环 num_denoising_steps=10 次。每次 chunk inference = 0.9B forward × 10 steps。架构图与 latency 已重画。
  • 🔴 #2 domain_id 没有"fresh slot"。soft_transformer.py:340 soft_prompt_hub = nn.Embedding(30, ...) 全 30 slot 都是预训练参数。配 domain_id=29 是 复用 slot 29 的预训练权重，不是 from scratch。要 fresh 必须显式 reinit 那个 embedding slice。v2 默认走 slot 0（最稳）。
  • 🔴 #3 CLI 写错。entry 是 lerobot.scripts.lerobot_train（不是 scripts.train），pretrained 字段是 --policy.path（不是 --policy.pretrained_path）。最坏：看似能跑但没加载预训练权重。
  • 🔴 #4 AutoActionSpace 是偷偷改协议。预训练 action head 有 channel semantics（EE6D/JointActionSpace/BimanualSO101 都把 gripper 放在特定 index 用特定后处理）。AutoActionSpace 把 6D 硬塞前 6 dim = 发明新 wire format，预训练参数对不上。v2 推荐自写 SingleArmSO101 子类继承正确的 channel layout。
  • 🔴 #5 chunk=32 + n_action_steps=32 = 1.07s open-loop。无 TAE / overlap averaging（modeling_xvla.py:421 select_action 纯 popleft）。v2 锁定 n_action_steps=8 重规划周期，chunk_size 训练时可保留 32 拿长上下文。
  • 🔴 #6 IDENTITY normalization + continuous gripper 是隐性陷阱。configuration_xvla.py:67 默认 normalization_mapping 对 state/action 是 IDENTITY，gripper 2.4-58.3° 直接进模型，跟预训练分布不对齐。v2 必须显式配 MIN_MAX 或 MEAN_STD 归一化。

1. X-VLA 架构核心（v2 修订后）

关键校正：X-VLA 不是 chunked regression，而是 rectified-flow / iterative denoising action diffusion。训练通过 noise 注入 + flow loss，推理通过 10-step 去噪循环。这跟 π0.5 / DP 同家族。

300-500 ms / chunk inference → 必须 chunk_size=32 训练长上下文 + n_action_steps=8 重规划周期 (266ms)，吞吐才正

1.1 模型内部结构（实际代码逐行确认）

X-VLA 不是单一 LLM/VLM 微调，是 Microsoft Florence2 (VLM) + 自研 SoftPromptedTransformer (policy 主干) 的拼装。

组件来源实现参数估算
Florence2 VLMMicrosoft 2024（modeling_florence2.py:1 Copyright）DaViT vision tower + BART encoder-decoder language model；X-VLA 仅用 vision tower + language encoder，不用 decoder~0.23B (base) ~ 0.77B (large)
SoftPromptedTransformerX-VLA 自研（soft_transformer.py:292）24 层标准 pre-LN Transformer block（vanilla MHSA + MLP），没有特殊架构。Innovations 在 4 个外围模块（见下）~0.66B（depth=24, hidden=1024, mlp_ratio=4）
合计——~0.9B ✓

X-VLA 的 4 个 custom design

组件位置作用
DomainAwareLinearsoft_transformer.py:215每个 domain 一套 fc/bias 参数（存在 nn.Embedding 里），输入 (x, domain_id) 用对应 domain 的 weight 做线性变换。30 个机器人 embodiment = 30 套独立 linear 权重
soft_prompt_hubsoft_transformer.py:340nn.Embedding(30, 32*1024) — 每个 domain 学 32 个 1024 维 soft prompt token，作为额外输入序列拼到主干
rectified-flow lossmodeling_xvla.py:225action_noisy = noise*t + action*(1-t)，model 预测 noise；推理从 N(0,1) 起步迭代 10 步去噪
timestep_embeddingsoft_transformer.py:183sinusoidal time encoding（同 DDPM/Diffusion），让 model 知道当前噪声 level

内部数据流（前向）

读图要点：

  • Florence2 是 外购的 Microsoft VLM，X-VLA 把它当 frozen 视觉/语言编码器使用，不动其权重
  • 真正的 X-VLA 创新在 policy transformer 的外围（DomainAwareLinear 让 30 个 embodiment 共享主干但有独立 linear 投影；soft_prompt_hub 让每个 embodiment 学独立的 prompt prefix；rectified-flow 风格的训练/推理）
  • 主干本身（24 层 TransformerBlock）是 vanilla pre-LN Transformer，没有 causal mask（全 bidirectional attn），跟 OFT 论文那种 "action position bidirectional + lm causal" 不同
  • 训练单 forward，推理 10 次 forward（rectified-flow 去噪循环）

2. 我们的 6D SO-101 单臂 → X-VLA 对接 contract（v2）

🔴 codex 校正：AutoActionSpace 是偷偷改协议，不是"通用入口"。
预训练 action head 有严格的 channel semantics（EE6D 把 gripper 放在 idx 9/19；BimanualSO101 在 5/11）。AutoActionSpace 把 6D 硬塞前 6 dim 等于发明新 wire format，**预训练 head 参数对不上你的语义**，迁移收益只剩 backbone。
方案怎么做优缺v2 评价
A. AutoActionSpaceaction_mode="auto", real_dim=6, max_dim=20零代码偷改 channel layout → 预训练 head 失效🟡 仅作"快速看起来跑得动"的 baseline
B. SingleArmSO101JointActionSpace（v2 默认）仿 BimanualSO101ActionSpace，REAL_DIM=6, gripper_idx=(5,), dim_action=20，注册 @register_action("so101_single")显式 channel semantics 跟预训练 head 对齐（gripper 在 idx 5，joints 在 0-4）；继承 BimanualSO101 的 MSE+BCE 损失分流需写 ~80 行子类 + 决定 gripper 是 binary（BCE）还是 continuous（MSE）🟢 v2 推荐
C. BimanualSO101 + 屏蔽右臂用 12-dim 双臂，前 6 dim 单臂，后 6 dim 0 + mask lossaction_hub 完全不动半屏蔽脏耦合，gripper_idx=(5,11) 跟单臂 (5,) 半对齐❌ 拒

2.1 Gripper 决策（关键）

SO-101 dataset gripper 是 continuous degrees（2.4° 夹紧 → 58.3° 张开）。两种处理：

策略LossX-VLA pretrained 期望风险
continuous (norm 到 [-1,1])MSE未知（可能期望 binary）预训练 gripper bias 可能误导
binarize (threshold @ ~30°)BCEEE6D/BimanualSO101 头都用 BCE，对齐损失 gripper 速度信息，部署时需 thresh→degree 反推

v2 推荐：先 continuous + MSE 跑（B 方案），保留 dataset 原始度数信号。如 ckpt-2000 gripper 闭合精度差，再 binarize 试 BCE 版。

lerobot/xvla-folding105 / 12叠衣服特化任务✗
lerobot/xvla-google-robot / xvla-agibot-world9 / 7/4不同机器人特化✗

为什么 xvla-base 而不是 xvla-widowx：base 训过 30 个 domain 的 soft prompts，转移到新 embodiment 时**只需要学一个新 domain 的 32 token soft prompt**，主干完全保留。任务特化 ckpt 的 soft prompts 已经偏向特定 domain，对我们 SO-101 的迁移收益小。

4. 资源估算 — 0.9B 不是事

项bf16 LoRA（默认推荐）QLoRA NF4（备选）
Base 0.9B 权重~1.8 GB~0.5 GB
Soft prompts（32 × 30 × 1024）~0.01 GB~0.01 GB
LoRA r=32 (attention)~0.06 GB~0.06 GB
Optimizer (AdamW fp32)~0.1 GB（只对 trainable）~0.1 GB
Activations (batch=4, chunk=32, no grad_ckpt)~5-7 GB~5-7 GB
合计~7-9 GB ✅ 余量 15GB+~6-8 GB ✅

结论：bf16 LoRA 即可，不必上 QLoRA。0.9B 全精度都不挤显存，去掉 bnb 4-bit 的 flaky bug（vanilla OpenVLA 那 4 类 crash）= 大幅简化训练稳定性。

5. 训练 launcher — 100% 复用 lerobot 原生（v2 CLI 校正）

X-VLA 已注册到 lerobot policy registry（@PreTrainedConfig.register_subclass("xvla")），entry 是 lerobot.scripts.lerobot_train（不是 scripts.train），pretrained 字段是 --policy.path（不是 --policy.pretrained_path，codex 🔴 #3）。

# LeIsaac/scripts/finetune/xvla/train.sh  (~40 行, v2 修订)

#!/usr/bin/env bash
set -euo pipefail

LEISAAC_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/../../.." && pwd)"
LEROBOT_ROOT="${LEROBOT_ROOT:-$HOME/work/lerobot-experience/lerobot}"

CONDA_ENV="${CONDA_ENV:-lerobot}"
OUTPUT_DIR="${OUTPUT_DIR:-$LEISAAC_ROOT/outputs/xvla-leisaac-pick-orange}"
DATASET_DIR="${DATASET_DIR:-$LEISAAC_ROOT/datasets/raw/leisaac-pick-orange}"
BASE_CKPT="${BASE_CKPT:-lerobot/xvla-base}"

cd "$LEROBOT_ROOT"
exec conda run -n "$CONDA_ENV" --no-capture-output \
    python -m lerobot.scripts.lerobot_train \
        --policy.type=xvla \
        --policy.path="$BASE_CKPT" \
        --policy.chunk_size=32 \
        --policy.n_action_steps=8 \
        --policy.action_mode=so101_single \
        --policy.max_state_dim=32 \
        --policy.max_action_dim=20 \
        --policy.freeze_vision_encoder=true \
        --policy.freeze_language_encoder=true \
        --policy.train_policy_transformer=true \
        --policy.train_soft_prompts=true \
        --policy.normalization_mapping.STATE=MEAN_STD \
        --policy.normalization_mapping.ACTION=MIN_MAX \
        --policy.domain_feature_key=null \
        --policy.num_denoising_steps=10 \
        --policy.resize_imgs_with_padding="[224,224]" \
        --dataset.repo_id=local \
        --dataset.root="$DATASET_DIR" \
        --output_dir="$OUTPUT_DIR" \
        --batch_size=4 --num_workers=4 \
        --steps=10000 --save_freq=500 \
        --optimizer.type=xvla_adamw \
        --optimizer.lr=1e-4 \
        --scheduler.type=cosine_decay_with_warmup --scheduler.num_warmup_steps=200

v2 主要变化（vs v1 / codex 🔴 修复）：

  • scripts.train → scripts.lerobot_train（codex 🔴 #3）
  • --policy.pretrained_path → --policy.path（codex 🔴 #3）
  • --policy.n_action_steps=32 → 8（codex 🔴 #5，266ms 重规划周期 vs 1.07s open-loop）
  • --policy.action_mode=auto → so101_single（codex 🔴 #4，自写子类后用我们注册的名字；如果暂时没写子类，回退 auto 但接受预训练 head 失效的代价）
  • 显式 normalization_mapping 防 IDENTITY 默认值陷阱（codex 🔴 #6）
  • 显式 freeze_vision/language_encoder=true（codex 抓到的注释 vs 默认值不一致 bug）
  • 显式 num_denoising_steps=10（确认 v1 latency 估算缺失）

7. 风险清单（v2 codex 整合后）
