3. 致命问题：单臂不支持 · Critical Issue: Single-Arm NOT Supported

风险等级：高 · Risk: HIGH。
LingBot-VLA 论文原文："future research will focus on scaling the model versatility by integrating single-arm and mobile robotic data" — 即 当前 ckpt 仅在双臂数据训练，单臂支持是未来工作。
我们的 SO-101 是单臂 6-DOF，与 LingBot 14-DOF（双臂 7+7）action head 维度不匹配。

3.1 适配 SO-101 的 3 条路径 · 3 paths to adapt

路径做法风险评估
A. Head 重训砍掉双臂 action head，新增 6-DOF flow-matching head，从 scratch 训新 head + LoRA backbone丢失 20k 小时双臂预训练优势；新 head 收敛速度 ≈ 从 zero 开始🟡 技术可行但成本高
B. 假装双臂把 SO-101 数据 pad 成双臂格式（第二臂置零或镜像）数据分布严重 OOD，预训练学到的双臂协同先验反而干扰🔴 不推荐
C. 等官方放单臂 ckpt论文承诺将整合单臂数据 → 等时间不可控（2026-Q2/Q3?）🟡 长期最优

对比 OpenVLA 优势 · Advantages over OpenVLA：
  • ✅ 参数更少（~3.5B vs 7B），LoRA 显存更宽裕
  • ✅ chunk=50 摊薄推理，30Hz sim 不会 timeout（vs OpenVLA 必 timeout）
  • ✅ Qwen2.5-VL backbone 比 OpenLlama-7B + DINOv2 + SigLIP 更新更强
  • ✅ LeRobot v3.0 原生支持
  • ✅ Apache-2.0 商用友好

5. 实施路径（选路径 A：Head 重训）· Implementation Path A

Phase 0 — 现实检查 (~0.5h)

  • 确认 LingBot-VLA repo 提供的 fine-tune 脚本是否暴露 action_dim 参数
  • 查 configs/ 是否允许 single-arm 配置（即便预训没训过）
  • 查 action_expert 是否模块化（能否 swap 6-DOF head）

Phase 1 — Action head 替换 (~3h)

  • 找到原 action head：action_expert/heads/dual_arm_head.py（推测）
  • 新建 single_arm_6dof_head.py：flow-matching head, output dim=6
  • config 加 action_dim=6 分支

Phase 2 — 数据接入 (~1h)

LeIsaac dataset 已是 LeRobot v3.0 — 直接喂。需要确认 LingBot 是否对 observation 命名有硬约束（dual-cam 命名 vs 我们的 cam_top / cam_wrist）。

Phase 3 — LoRA 微调 (~10h)

LoRA 配置（待 repo 内细节确认）：

--lora_rank=16 --lora_alpha=16
--lora_targets q_proj,v_proj  # Qwen2.5-VL attention
--freeze_action_expert=false  # head 必须训
--batch=4 --lr=5e-5 --bf16
--steps=10000 --grad_ckpt=true

Phase 4 — Server + client (~2h)

  • server/lingbot_leisaac/server.py — 类比 pi05 server
  • client: LingBotServicePolicyClient，chunk=50 wire protocol
  • policy_inference.py 加 --policy_type=lingbot

Phase 5 — Eval (~2h)

  • 类比 π0.5 horizon sweep：从 chunk*0.5 = 25 起步
  • 预期 horizon trap 模式与 π0.5 类似（同 chunk=50, 同 flow-matching）
