设计文档：SmolVLA2 在 LeIsaac SO-101 PickOrange 上微调与推理验证

实验性 VLA 微调 SO-101 仿真闭环 LeRobot

作者：vitorcen
状态：草案 (draft)
目标分支：vitorcen/LeIsaac fork
命名说明：HuggingFace 官方暂无 SmolVLA2 模型（截至 2026-05），本设计将 基于 lerobot/smolvla_base 在 LightwheelAI/leisaac-pick-orange 上微调出的 ckpt 命名为 SmolVLA2，作为后续 fork 系列的版本号起点。

1. 背景与动机

本机已有的 LeIsaac SO-101 PickOrange 推理链路覆盖了 GR00T N1.5 / N1.6 / N1.7 三档大模型（3B 参数，单卡显存 ~15–25 GB），但缺少一个 轻量基线：模型小、训练快、推理延迟低，方便在 4090 / 笔记本 GPU 上做快速消融。

HuggingFace 上还没有官方 SmolVLA2 模型，所以本设计 自己基于 SmolVLA 微调一版，命名为 SmolVLA2，作为 fork 的命名锚点：

  • Base：lerobot/smolvla_base，~450M 参数（SmolVLM2-500M-Video-Instruct backbone + Action Expert）。
  • 已有第三方微调可作对照：edge-inference/smolvla-so101-pick-orange（本机已 zero-shot 实测通推理链路）。
  • 训练框架：HuggingFace LeRobot 的 lerobot-train CLI，与本机现有 policy_server（端口 8080）天然兼容。
  • 输出 ckpt 计划上传到 vitorcen/smolvla2-so101-pick-orange-v0。

本设计的核心问题：

能否用 SmolVLA base（~450M）+ 同一份 LightwheelAI/leisaac-pick-orange 数据集，微调出一版 SmolVLA2 ckpt，复现 edge-inference 第三方版的成功率，并作为评估 GR00T 系大模型 ckpt 的轻量对照基线？

2. 范围

范围内范围外
  • 用 LightwheelAI/leisaac-pick-orange 数据集 + lerobot/smolvla_base 微调出 SmolVLA2 ckpt
  • 把 SmolVLA2 ckpt 接入 LeIsaac policy_inference.py --policy_type=smolvla
  • 在 LeIsaac-SO101-PickOrange-v0 仿真环境跑 ≥ 10 轮 episode 评估成功率
  • 与 edge-inference/smolvla-so101-pick-orange 第三方 ckpt 横向对比
  • 与 GR00T N1.5 ckpt（已实测 1/1）作为大模型上限参照
  • 真机部署（仅仿真验证）
  • 多任务（仅 PickOrange）
  • 数据增广 / 合成数据（未来扩展）
  • 对 VLM backbone 做继续预训练（仅微调 action expert + state proj）

3. 数据集

来源：LightwheelAI/leisaac-pick-orange（LeRobot Dataset v2.1）

字段值
codebase_versionv2.1
robot_typeso101_follower
total_episodes60
total_frames36,293 （avg ~605 frame / episode ≈ 20s @ 30fps）
fps30
observation.images.front / wrist480×640 RGB，av1 编码 mp4
observation.state / actionfloat32 [6]：5 关节 + 1 gripper
任务描述"Pick up the orange and place it on the plate"

下载：

hf download LightwheelAI/leisaac-pick-orange --repo-type dataset
相机键命名陷阱（实测复盘）：数据集用自然键 front/wrist @ 480×640，但 lerobot/smolvla_base 的 config.json 声明了 3 个占位槽 camera1/camera2/camera3 @ 256×256。

draccus 对 dict 类字段的 CLI override 是 合并而非替换——首次实验里我尝试 --policy.input_features='{...自然键...}' 想覆盖 base 的 schema，结果 两套键被合并（5 个槽），训练出来的 ckpt 既有占位 camera1/2/3（其中 camera3 完全没数据 → 死权重）又有 front/wrist，且模型按 256×256 下采样训练，分辨率损失大。同时也试过 --policy.input_features={}，draccus 视作 no-op。

正解（见 §5）：把 smolvla_base 克隆到本地、改 config.json 清空 input_features 和 empty_cameras，让 lerobot-train 在 make_policy() 走到 if not cfg.input_features: cfg.input_features = features_from_dataset 这一行，自然键 + 真实 480×640 自动生效。

关键差异 vs GR00T 流程：
  • 训练命令是 lerobot-train --policy.path=lerobot/smolvla_base --dataset.repo_id=...，不再用 GR00T 的 gr00t_finetune.py。
  • 推理 server 是通用的 LeRobot policy_server（端口 8080，gRPC），不再用 GR00T 的 ZMQ :5555。Server 启动时不绑定 ckpt，由客户端在请求里指明。
  • 没有 embodiment_tag 概念——SmolVLA 直接消费 LeRobot Dataset feature schema。

5. 训练计划

5.1 命令模板（推荐用 LeIsaac 仓的 scaffold）

# 一次性：准备 schema-free 的本地 base（清掉 input_features + empty_cameras）
bash scripts/finetune/prepare_smolvla_base.sh
# → outputs/.bases/smolvla_base_no_features/

# 微调
BASE_MODEL=$(pwd)/outputs/.bases/smolvla_base_no_features \
DATASET_REPO_ID=LightwheelAI/leisaac-pick-orange \
OUTPUT_NAME=smolvla2-leisaac-pick-orange \
STEPS=30000 \
BATCH_SIZE=8 \
NUM_WORKERS=2 \
SAVE_FREQ=5000 \
EXTRA_ARGS='--dataset.video_backend=pyav' \
bash scripts/finetune/lerobot_finetune.sh

等价的直接 lerobot-train 调用（无 scaffold）：

conda activate lerobot
lerobot-train \
  --policy.path=$(pwd)/outputs/.bases/smolvla_base_no_features \
  --policy.push_to_hub=false \
  --dataset.repo_id=LightwheelAI/leisaac-pick-orange \
  --dataset.root=datasets/raw/leisaac-pick-orange \
  --output_dir=outputs/smolvla2-leisaac-pick-orange \
  --batch_size=8 --steps=30000 --save_freq=5000 --num_workers=2 \
  --policy.device=cuda --wandb.enable=false \
  --dataset.video_backend=pyav

5.2 超参选择（与 edge-inference/smolvla-so101-pick-orange 对齐）

项值依据 / 实测踩坑
steps30,000edge-inference 使用的步数；4090 单卡约 9 step/s ⇒ ~56 min。先前 20k 步 + batch 64 训出的 ckpt 推理时会动但完不成任务，故对齐第三方
batch_size8edge-inference 默认；4090 实测 ~9 step/s。反直觉：先前 batch=64 才 1.2 step/s（PCIe / pyav 解码 IO 撞瓶颈），等同步骤×7.5× 倍训练时间，且不见 batch 越大收敛越好
num_workers2torchcodec + 4 worker 在长跑时会段错误（先前训练在 1601 步崩）；2 worker 稳定
video_backendpyav慢一点但稳。torchcodec 偶发 segfault
input_features自然键 480×640用 prepare_smolvla_base.sh 把 base 的占位 schema 剥光，让 lerobot-train 从数据集 features 自动生成。不要 用 --policy.input_features CLI override（draccus 合并而非替换）
optimizer_lr1e-4（默认）SmolVLAConfig 默认；warmup 1000 步 + cosine decay 至 2.5e-6
freeze_vision_encoderTrue（默认）只调 action expert + state proj，约 100M trainable / 450M total
chunk_size / n_action_steps50SmolVLA 默认，~1.7s @ 30fps
checkpoint 保存每 5000 步 + last方便消融

6. 推理验证

6.1 启动 LeRobot policy_server

Server 是通用的，启动时不绑定 ckpt；客户端在请求里通过 policy_path 指定具体模型。

bash ~/work/isaaclab-experience/scripts/policy_server.sh start lerobot
# 或直接：
conda activate lerobot && \
  python -m lerobot.async_inference.policy_server --host 0.0.0.0 --port 8080
端口 8080（gRPC），与 GR00T N1.5/N1.6/N1.7 的 :5555 完全独立，可以并发起 server。

6.2 启动 Isaac Sim 客户端（加载 SmolVLA2 ckpt）

cd ~/work/isaaclab-experience/LeIsaac && \
  PYTHONUNBUFFERED=1 python -u scripts/evaluation/policy_inference.py \
    --task=LeIsaac-SO101-PickOrange-v0 \
    --eval_rounds=10 --episode_length_s=120 \
    --policy_type=lerobot-smolvla \
    --policy_host=127.0.0.1 --policy_port=8080 \
    --policy_timeout_ms=15000 \
    --policy_language_instruction='Pick the orange to the plate' \
    --policy_checkpoint_path=/home/david/work/LeIsaac/outputs/smolvla2-leisaac-pick-orange/checkpoints/last/pretrained_model \
    --policy_action_horizon=16 --device=cuda --enable_cameras
camera 自动适配：LeIsaac 客户端 _build_camera_feature_map(ckpt_path, sim_cameras) 会读 ckpt 的 config.json/input_features：
  • 自然键 front/wrist（我们用 prepare_smolvla_base.sh 训出来的 SmolVLA2，以及 edge-inference）→ 返回 (None, [])，不 rename、不 pad。
  • 占位键 camera1/2/3（直接 --policy.path=lerobot/smolvla_base 训出来的劣化 ckpt）→ 返回 (rename=front→camera1, wrist→camera2, empty=[camera3])；client 自动 zero-pad 多余 camera3 槽。
上述两条 fallback 在 LeIsaac fork 已实现（patch 见 patches/leisaac/）。
推理时另两个必坑（实测）：
  1. policy_server 状态在 client 失败时不会自清：上一次失败的 client 会留下 last_processed_obs，导致下一次 client 的首帧被去重过滤器丢掉（server log "Observation #1 has been filtered out"）。规避：每次重试前 pkill -9 -f "policy_server" && bash scripts/policy_server.sh start lerobot。
  2. Isaac Sim 进程在 Python 异常后不一定释放 GPU：失败的 episode 留下 ~6GB VRAM 占用，下一次再起 Isaac Sim 会 OOM 或共享 GPU 慢。规避：pkill -9 -f "policy_inference.py" 后 nvidia-smi 确认 GPU 干净再重试。

  • 平均推理耗时 / chunk < 80ms（SmolVLA2 ~450M 参数预计远快于 GR00T 3B）
  • 训练完成 train_loss 收敛趋势记录（对比 edge-inference ckpt 公开数据）
  • 训练显存峰值（用 nvidia-smi 记录）

8. 风险与缓解

风险影响缓解
SmolVLA 在 60 episode 小数据上对 PickOrange 这种几何敏感任务能力不足 SmolVLA2 成功率 << GR00T baseline 这正是实验目的：定量量化"轻量 VLA"和"3B VLA"在小数据微调上的能力差；不达 MVP 也是有价值的负结果
LeRobot v0.5.1+ SmolVLAConfig 字段与 LeIsaac 旧版 vendored v0.4.2 不一致 policy_server 加载 ckpt 报 unknown fields 已知问题，已在 server/start_server.sh 显式用 conda env 的 v0.5.1+ 而非 vendored v0.4.2；本设计沿用此修复
LeIsaac sim 相机 fps（30）和 SmolVLA expected 不匹配 动作 chunk 时序错位 数据集本身就是 30fps 采集，训练-仿真完全一致，无错位
Policy server gRPC 返回 Empty（动作队列耗尽） 客户端 timeout / 复用旧动作 LeIsaac client 已实现 _receive_action 有界重试（max_retries=8, sleep=25ms），无需额外改动
"SmolVLA2" 命名与 HF 官方未来发布的同名模型冲突 命名空间撞车 上传时用 vitorcen/smolvla2-so101-pick-orange-v0，加 owner 前缀；如官方真发 SmolVLA2，README 里加 disambiguation

9. 里程碑 + 已完成步骤

M动作验收 / 实际结果耗时
M1下载数据集（670 MB）+ smoke test 训练 50 步✅ loss 下降，pipeline 通~15 min
M2.v1 失败批量 64、20000 步，--policy.path=lerobot/smolvla_base 直接微调 + rename_map 占位键❌ 训出了 256×256 + 3 占位相机 schema 的 ckpt；客户端能跑、机器人会动，10 轮 episode 全部 timeout4h49min
M2.v2 ✅用 prepare_smolvla_base.sh 剥光 base schema，批量 8、30000 步，对齐 edge-inference 配方🔄 训练中（~56 min ETA）~1h
M3仿真推理 10 轮评估 SmolVLA2 v2 ckpt⏸ 待 M2.v2 完成~30 min
M4同环境跑 edge-inference 第三方 ckpt 10 轮做对照⏸30 min
M5结果写入 fork README + 上传 SmolVLA2 ckpt 到 vitorcen/smolvla2-so101-pick-orange-v0⏸2 小时
M2.v1 失败原因复盘：
  1. --policy.path=lerobot/smolvla_base 加载的 base schema 含 3 个占位相机 camera1/2/3 @ 256×256；试图用 --policy.input_features=... 覆盖时 draccus 合并而非替换，结果训练出来 schema 是 5 个相机槽 + 256×256 下采样。
  2. 同等 60 episode 数据下，shadowHokage/act_policy（80M ACT @ batch 8, 10k steps, 480×640）成功完成任务；说明数据本身够，问题在我们 SmolVLA 训练 schema 不对。
  3. 对照 edge-inference/smolvla-so101-pick-orange（同 base 同数据 ✅）的 train_config.json，他们的 ckpt schema 是纯自然键 + 480×640。推断只能通过本地剥光 base 的 input_features 实现。
  4. 另：batch=64 反而比 batch=8 慢 7.5×（PCIe / pyav 解码 IO 撞瓶颈，不是 GPU 计算 bound）。
