DreamZero (Wan2.1-I2V-14B World Action Model) — 4090 24G 微调 & 推理可行性

TL;DR (2026-05-21 实测验证)

DreamZero = NVIDIA GEAR Lab 提出的 World Action Model (WAM)，把 Wan2.1-I2V-14B 视频扩散模型作为 backbone，加 chunk-wise causal attention + 24 action tokens + flow matching，实现「先想象一段未来视频，再从中提取动作」的范式（vs VLA「直接预测动作」）。

核心实测结论（4090 单卡 NF4 推理）：
  • 权重 8.54 GB · forward peak 10.43 GB · activations 仅 1.89 GB · 余量 13.57 GB 显存充裕
  • 单 step 推理 6318 ms（无 flash-attn），4 step denoise ≈ 25s/inference → 0.04 Hz 实时性 7Hz 不可能
  • 结论：本地 4090 适合 batch eval / 演示 / QLoRA 训练；闭环 7Hz 实控必须 H100×2 TP
DreamZero is NVIDIA GEAR Lab's World Action Model — Wan2.1-I2V-14B video diffusion + chunk-wise causal action tokens + flow matching. Verified NF4 weights 8.54 GB on a single 4090; activations only 1.89 GB; total peak 10.43 GB with 13.57 GB headroom. But 4090 can't hit the paper's 7 Hz (single-step 6.3 s without flash-attn). Local 4090 is good for batch eval, demos, and QLoRA finetune; real-time closed-loop control still needs H100×2.

3. NF4 推理 5 个关键技术坑 (5 Critical Hacks)

3.1 Meta device init 避免 CPU OOM

直接 WanModel(**cfg) 在 CPU fp32 = 56 GB（14B×4byte）。本机 62GB RAM + 4.5GB swap，第一次跑 SIGKILL exit 137 = OOM kill。
修复：用 torch.device("meta") 上下文构建（0 byte placeholder）： 
with torch.device("meta"):
    model = WanModel(**I2V_14B_CONFIG)

3.2 Shard-by-shard streaming load

7 个 safetensors shard 总 60 GB（bf16 + fp32 master weights），不能一次 load 进 RAM。修复 = 逐 shard 流式读取，对每个 tensor 立即处理：

  • 2D Linear weight → 立即用 bnb.nn.Linear4bit + Params4bit 量化 → .to("cuda") → cuda hook 触发 NF4 量化 → 释放 CPU 临时
  • 非 Linear (norm, bias, embed, RoPE) → bf16 → cuda → assign

3.3 Identity remap（Wan2.1 ≠ diffusers naming）

踩坑：参考 DreamZero 的 WanModelStateDictConverter.from_diffusers，假设 raw safetensors 是 diffusers 命名（blocks.0.attn1.to_q.weight）。实际跑发现 raw key 0 个匹配，1303 个 model param 一个都没 load。
修复：Wan2.1-I2V-14B-480P 的官方 safetensors 本来就是 DreamZero 原生命名（blocks.0.cross_attn.k.weight），identity passthrough 即可，195/195 全 match。 
def remap(raw_k):
    return raw_k  # identity, no diffusers rename needed

3.4 RoPE freqs 必须在 cuda device 上重算

meta init 时 RotaryPositionEmbedding.__init__ 调 precompute_freqs_cis_3d，用了 torch.arange / outer / cos / sin — 这些在 with torch.device("meta") 下都是 meta tensor。后续 post_initialize 试 .to("cuda") 报 Cannot copy out of meta tensor。 
head_dim = I2V_14B_CONFIG["dim"] // I2V_14B_CONFIG["num_heads"]  # 128
with torch.device("cuda"):
    model.rope.freqs = model.rope.precompute_freqs_cis_3d(head_dim)

3.5 distributed primitives 必须绕开 (单卡 hack)

官方 inference server socket_test_optimized_AR.py 用 torch.distributed.run --nproc_per_node=2，里面 init_device_mesh + dist.get_rank()。2-GPU TP 是为了 H100 切 14B 模型，4090 不需要。
单卡需做 3 件事：
1. dist.init_process_group(backend="nccl", world_size=1, rank=0) 起 1-rank world
2. sim_policy.py:346-350 已有 try: model.parallelize(device_mesh=device_mesh) except: pass fail-soft，传 device_mesh=None 自动跳过
3. server 层 DeviceMesh / dist.broadcast 改成本地直传

⚠️ Fast-WAM 教训 复习（fastwam_qlora_inference_debug.html）：
50 demo + 5B Wan2.2 + 39M LoRA = step_008000 模型在 eval 时 joint 输出 -170°，训练 range 是 [-100, +64]，严重 extrapolate。video diffusion subtask 在小数据集上会强迫模型 memorize 训练帧而非学到 generalization。

对 DreamZero 的预测：14B 比 5B 更容易 overfit。Vizuara 用了 ~800 episode 跨 8 个 task 的 SO-101 aggregate dataset 才训出可用的 LoRA — 我们 LeIsaac 只有 50 demo + 1 task，大概率重蹈覆辙。

建议：先用 Vizuara LoRA zero-shot，验证架构在 SO-101 PickOrange 上的 transfer 能力，再决定是否自己训。
