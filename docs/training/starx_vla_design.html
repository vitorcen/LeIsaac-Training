StarX-VLA 设计文档

StarVLA「搭积木」可插拔 × X-VLA「跨具身」soft-prompt 条件化 · 设计草案 v1 · 2026-06-08

Design for StarX-VLA: graft X-VLA's cross-embodiment conditioning (soft-prompt hub + domain-aware projections) onto StarVLA's lego-like swappable-backbone / swappable-head codebase. One checkpoint, many robots, riding the Qwen3-VL vision dividend.

核心命题 / Thesis：把 X-VLA 的跨具身条件化机制（soft_prompt_hub + DomainAwareLinear）从它自带的弱 backbone（Florence2 @224px）里抽出来，做成 StarVLA 的第三根可插拔轴，嫁接到冻结的 Qwen3-VL backbone + PI_v3 action head 上。
一句话本质 / In one line：StarVLA 给你「换骨干 × 换头」，X-VLA 给你「一个 ckpt 跨多机器人」——StarX-VLA = 三轴同时可插拔（VLM backbone × embodiment adapter × action head），且让跨具身能力吃上 Qwen3-VL 的高分辨率 vision 红利（PickOrange 铁律：小橙子 10–40px 需 ≥448px，Florence2 的 224px 吃不下）。
StarVLA = swap backbone × head. X-VLA = one checkpoint across robots. StarX-VLA = both, on a third pluggable axis, with X-VLA's embodiment conditioning finally riding a high-resolution Qwen3-VL backbone instead of Florence2@224.
⚠️ 评审后定调（codex gpt-5.5 + mimo v2.5-pro 双独立审，均比对真源码，见 §13）：这不是「移植 X-VLA 条件化」，而是「受 X-VLA 启发、新建一个 domain-conditioned PI_v3 head」——X-VLA 的 soft prompt 之所以有效，靠的是它那个全可训练的 24 层 transformer 协同适配；StarX 里 VLM 冻结、只有 PI_v3 head 可训，soft prompt 的语义已改变（变成 action DiT 的「逐机器人条件 token」，不再是通用 embodiment prompt）。
因此落地顺序倒过来（两审一致的头号建议）：先做不带 soft prompt 的最小版（V0），再按消融证据决定要不要加 soft prompt（V1）。见 §7.5 推荐落地顺序。

1. 动机：两个框架各缺一半 · Why — each framework is half a solution

这个项目对 PickOrange 横评已经积累了两条硬结论，正好分别戳中两个框架的软肋：

  • 铁律 A（vision 分辨率）：橙子在画面里只有 10–40px，backbone 必须 ≥448px 动态分辨率才看得清（见 .memory/vla-pickorange-vision-resolution-selection）。Qwen3-VL DeepStack + dynamic resolution 是已验证的赢家。
  • 铁律 B（head 选型）：同 Qwen3-VL-8B 骨干下，PI_v3 head（LayerwiseFM cross-DiT）= 63.3% > GR00T head 53.3%（已发布实锤），flow-matching 系显著优于离散 token。
框架强 / Strength弱 / Weakness（对本任务）
StarVLA「搭积木」：VLM backbone（Qwen3-VL 2B/4B/8B）× action head（FAST/OFT/PI_v3/GR00T）自由组合；冻 VLM 只训 head；原生 LeRobot v3；吃满 Qwen3-VL 高分辨率（满足铁律 A+B）。锁单具身。每个 ckpt 只服务一个机器人；PickOrange 60-demo 小数据 regime 峰值卡在 3–4 ep（18–63%）。换机器人 = 从头再训一个 head。
X-VLA跨具身是设计目标：soft_prompt_hub（30 个 domain slot × 32 token）+ DomainAwareLinear（每具身独立投影）让一个主干服务 30 种 embodiment；新机器人只学一个 soft-prompt slot。锁 Florence2 @224px。backbone 写死、偏弱，违反铁律 A（224px 看不清 10–40px 小橙子）；想换强 backbone 要动主干。
StarX-VLA取两者交集：StarVLA 的可插拔骨干/头（满足铁律 A+B）+ X-VLA 的跨具身条件化（一个 ckpt 跨机器人）。把「具身」升格为第三根独立可插拔轴。

2. 三问自检（Linus 模式）· Sanity check

问答
1. 真问题还是假想？真问题。当前 leaderboard 全是单具身 SO-101 ckpt，每个机器人从头训 head，且被 60-demo 小数据天花板压住。跨具身预训练（Bridge+RT-1+双臂+SO-101 共训）直接给 head 喂更多数据，攻击的正是这个天花板。X-VLA 已证明跨具身可行，但它的 224px backbone 看不清橙子——缺的恰好是 StarVLA 的强 backbone。
2. 有更简单方案吗？「直接用 X-VLA」最简单，但过不了铁律 A（Florence2 224px）。「直接用 StarVLA」过得了 vision 但不跨具身。两者都不可省——StarX-VLA 是能同时满足铁律 A+B+跨具身的最小组合，不是叠床架屋。
3. 会破坏什么吗？零破坏。单具身退化 = 现有 QwenPI_v3 路径（见 §7 范式 3）。已发布的 StarVLA-Qwen3-VL-8B-PI_v3 就是 StarX-VLA 在「embodiment 数=1」下的特例。Never break userspace：旧 config 不改照跑。

3. 三轴可插拔架构 · Three pluggable axes

StarVLA 现在是两轴（backbone × head）。StarX-VLA 在两者之间插入第三轴 = Embodiment Adapter，三轴互相正交、各自可换。

读图要点 / Reading the diagram：

  • 轴 1（backbone）和 轴 3（head）是 StarVLA 已有的两轴（冻 VLM、运行时按 VLM hidden_size 对齐）。
  • 轴 2（Embodiment Adapter）是新增，从 X-VLA 抽取：soft_prompt_hub 提供每具身的 K 个 prompt token，DomainProj 提供每具身独立的 proprio/action 投影，EmbodimentSpec 驱动通用 pad/unpad。
  • 三轴正交：换 backbone 不动 adapter，换 head 不动 backbone，加机器人不动前两者。

4. 核心数据结构：EmbodimentSpec（消除 X-VLA 的特殊情况）· The good-taste move

Linus 视角 · "好程序员关心数据结构"：X-VLA 把每种机器人的通道语义硬编码成一个子类——ee6d / joint / auto / so101_bimanual / agibot_ee6d / franka_joint7，6+ 个类，每个 ~80 行、各自规定 gripper 放哪个 index、用 MSE 还是 BCE。这就是「10 个 if/else」的味道。加一台机器人就要写一个新子类（见 X-VLA 文档 P0d：写 SingleArmSO101JointActionSpace 子类 ~80 行 + 注册）。

StarX-VLA 的做法：把 N 个特殊情况类压成 1 个数据结构 + 1 条通用路径。机器人之间的差异本质只是几个数字（多少 DOF、gripper 在第几维、gripper 是连续还是二值、用什么归一化）——这些是数据，不该是代码。

@dataclass
class EmbodimentSpec:
    name: str            # "so101_single"
    emb_id: int          # soft-prompt / domain slot 编号
    state_dim: int       # 6   —— 真实 proprio 维度
    action_dim: int      # 6   —— 真实 action 维度
    gripper_idx: tuple   # (5,) —— 哪几维是夹爪
    gripper_loss: str    # "mse" | "bce"
    state_norm: str      # "mean_std"
    action_norm: str     # "min_max"

# 一个通用 adapter 读 spec —— 不再有 per-robot 子类
class EmbodimentAdapter(nn.Module):
    def __init__(self, specs: list[EmbodimentSpec], hidden, K, max_state, max_action):
        self.soft_prompt = nn.Embedding(len(specs), K * hidden)   # 每具身 K 个 prompt token
        self.proprio_proj = DomainProj(len(specs), max_state, hidden)
        self.action_in    = DomainProj(len(specs), max_action, hidden)
        self.action_out   = DomainProj(len(specs), hidden, max_action)
        self.specs = {s.emb_id: s for s in specs}

    def encode(self, proprio, emb_id):          # 入：spec 驱动 pad → 投影
        spec = self.specs[emb_id]
        x = pad_to(proprio, self.max_state)     # 通用 pad，无特殊情况
        return self.proprio_proj(x, emb_id) + self.soft_prompt(emb_id)

    def decode(self, h, emb_id):                # 出：投影 → spec 驱动 unpad + gripper 路由
        spec = self.specs[emb_id]
        a = self.action_out(h, emb_id)[..., :spec.action_dim]
        return route_gripper(a, spec.gripper_idx, spec.gripper_loss)
X-VLA（现状）StarX-VLA（本设计）
加一台新机器人写 ~80 行子类 + @register_action + 决定 channel layout加 1 条 EmbodimentSpec 数据 + reinit 1 个 slot
通道语义散落在 6+ 个类里，易错配（gripper idx 5 vs 11 半对齐）集中在 spec 一处声明，一眼可审
gripper MSE/BCE子类里硬编码spec 字段，通用路由
代码量N 机器人 × 80 行1 个 adapter + N 条数据

N special-case subclasses → 1 data structure + 1 generic path. This is the linked-list-deletion refactor applied to embodiment handling: the differences between robots are data (a few numbers), not code.

⚠️ 评审修正（两审一致，比对 action_hub.py:55/530）：纯数据不够，要「数据优先 + 命名 hook」。X-VLA 的子类除了 channel layout，还编码了真正需要代码的行为：rotation 转换（ee6d 6D↔旋转）、delta vs absolute 控制、action clip/rate-limit、gripper sigmoid/hysteresis、双臂耦合、controller-specific 反归一化、preprocess 时把 dummy 维清零。这些不能压成几个数字。
修正后的 EmbodimentSpec = 数据字段 + 一组可选「命名 transform hook」（按名字从注册表取函数，不是写子类）： 
@dataclass
class GripperSpec:                 # gripper 升格为子结构（mimo 提）
    idx: tuple                     # 可多维 / 双臂多组
    loss: str                      # "mse" | "bce"
    norm: tuple | None             # gripper 自己的反归一化（异构 gripper）

@dataclass
class EmbodimentSpec:
    name: str; emb_id: int
    state_dim: int; action_dim: int
    grippers: list[GripperSpec]    # 支持双臂 / 多 DOF gripper
    state_norm: str; action_norm: str
    control_mode: str              # "absolute" | "delta"
    pre_hooks:  list[str] = ()     # 命名 hook：rotation/zero-dummy/...
    post_hooks: list[str] = ()     # 命名 hook：clip/rate-limit/unnorm/...
数据先行、行为走注册表 hook——既消除了 N 个子类的重复，又不把真需要代码的逻辑硬塞进数据。「纯 data-only 是天真的」（codex 原话）。
⚠️ DomainProj 参数爆炸（两审一致，比对 soft_transformer.py:222）：X-VLA 的 DomainAwareLinear 把每个 domain 的整张权重矩阵存进 nn.Embedding。在 Qwen/PI 的 hidden（2560/4096）下，30 个 embodiment × 整张矩阵 = 参数暴涨 + 小 slot 过拟合。StarX 必须改用「共享 linear + per-domain 低秩/LoRA + 仿射」，不要照搬 X-VLA 的整矩阵存法。且 DomainProj 只插在 head 的输入/输出端（2 处），不要每层 DiT 都插。

5. 跨具身条件化注入点：head 侧、落在 PI_v3 · Where conditioning is injected

X-VLA 的 soft prompt / DomainAwareLinear 原本是给它可训练的 24 层 SoftPromptedTransformer（那个 transformer 本身就是 policy）设计的。但 StarX-VLA 里 VLM 冻结、head 才是被训的小模块——条件化该注到哪？

方案注入位置评价
(a) prompt 侧soft-prompt token 拼到冻结 VLM 的输入序列（P-tuning 风格 prefix）🟡 干净，但 VLM 冻结时 prefix 离 action 输出太远，条件化信号衰减；且每加一具身要重跑 VLM 前向。
(b) head 侧 ✅soft-prompt token 注入 PI_v3 head 的逐层 cross-attention；DomainProj 替换 head 的 proprio/action in/out 投影最佳契合。PI_v3 LayerwiseFM 本就逐 VLM 层做 cross-attention 注入——这正是 soft prompt 最想待的地方。条件化紧贴 action，且 emb_id 只是选 slot，不增加 VLM 前向成本。
设计决策：走 (b)，宿主选 PI_v3。理由有二：(1) 铁律 B —— PI_v3 已实测 63.3% > GR00T 53.3%，是当前最强 head；(2) PI_v3 的 LayerwiseFM cross-DiT 结构上就是逐层注入，把 X-VLA 的 soft prompt 接进它的 cross-attention KV，是顺势而为而非硬塞。GR00T head 作对照（末层注入，预期条件化偏弱）。
⚠️ 评审修正 1 —— 不是「每层」，是「每偶数层」（mimo，比对 PI_v3 interleave_self_attention=True）：PI_v3 默认奇数层是纯 self-attention、忽略 encoder_hidden_states；Qwen3-VL-8B 36 层里只有 18 层（偶数层）做 cross-attention。所以 soft prompt 实际注入点是 18 个，不是 36。上方 SVG 的「逐层」应读作「逐 cross-attn 层」。
⚠️ 评审修正 2 —— KV 直接 concat 会让 DiT 忽略 VLM 特征 / 过拟合 emb_id（codex，比对 LayerwiseFM_ActionHeader.py:335）：PI_v3 每个 block cross-attend 到 vl_embs_list[layer_idx]；在 KV 里硬塞 K 个学习 token，可能盖过真正的 VLM 视觉/语言特征。必带门控： 
# 逐 cross-attn 层 gated 注入，alpha_l 初始化为 0
KV_l = vl_embs_list[l]  ⊕  alpha_l * prompt_proj(soft_prompt[emb_id])
# alpha_l = nn.Parameter(0.0)  → 训练初期 KV ≡ 原 PI_v3
alpha_l 初始化 0 的关键收益：N_emb=1 时模型逐位等于原 PI_v3 → §8 的 E0 复现 63.3% 才真有意义（否则 adapter 一接上就改了数值，E0 不再是干净基线）。这也直接兑现 §2 的「零破坏」承诺。

6. 维度对齐：两套机制正交、不冲突 · Dimension alignment — two orthogonal axes

易混点（必须钉死）：StarVLA 和 X-VLA 各有一套「维度对齐」，层级完全不同，不要混为一谈：
机制对齐什么谁负责StarX-VLA 处理
hidden_size 对齐（StarVLA）特征维：head 的 cross-attention dim 运行时对齐到所加载 VLM 的 hidden_size（2B=2048 / 4B=2560 / 8B=4096）head（QwenGR00T.py:155 / PI_v3）原样保留。换 backbone 自动对齐，adapter 不掺和。
action/state padding（X-VLA）动作空间维：不同机器人 DOF 不同，统一 pad 到 max_state/max_action，且 gripper 必须落在约定 indexEmbodiment Adapter（EmbodimentSpec）由 spec 驱动，集中声明、通用 pad/unpad。

一个管特征宽度（跟 backbone 走），一个管动作语义布局（跟机器人走）——正交，互不干扰。这正是三轴可插拔成立的前提。

7. 三种训练范式 · Three training regimes

范式冻 / 训数据用途
1. 跨具身共训
Co-train		冻 VLM；训 PI_v3 head + 全部 soft-prompt slot + DomainProj{Bridge, RT-1, 双臂, SO-101} 混合，温度采样平衡（见风险 R3）预训练一个跨具身 ckpt，攻击 60-demo 小数据天花板。
2. 新具身适配
Adapt
⚠️未验证		冻 VLM + 冻共享 head；只训新 emb_id 的 1 个 slot + 它的 DomainProj新机器人少量 demo（如 SO-101 60 demo）存在性存疑（两审头号风险）：一个冻住的 head 凭什么被「它从没针对训练过的新 slot」引导？X-VLA 没这问题因为它的 transformer 是活的、会适应新 slot。前提条件：范式 1 共训时必须用 prompt-dropout / 多 slot 泛化训练让 head 学会「读任意 slot」，且新机器人若 控制模式/运动学与预训练集差异大，单 slot+投影救不了共享去噪向量场（codex）。见 §10 R7。
3. 单具身
Single（= 现状）		冻 VLM；训 PI_v3 head（embodiment 数=1，slot 退化）单机器人 SO-101向后兼容：完全等价现有 QwenPI_v3。已发布的 8B-PI_v3 ckpt 就是这个特例。
不破坏保证：范式 3 让 StarX-VLA 在 N_emb=1 时逐位退化为现有 StarVLA QwenPI_v3（配合 §5 的 alpha_l=0 门控初始化，是数值逐位等价，不只是「概念等价」）。新框架是超集，不是替代——旧 config、旧 ckpt 照常工作。

7.5 推荐落地顺序：先 V0（无 soft prompt），再按证据上 V1 · Build order

两审独立给出几乎相同的头号建议：soft prompt 不是起点，是「证明 I/O 投影不够」之后才加的可选项。「最小有用轴 = EmbodimentSpec + DomainProj；soft prompt 在被证明必要前都是可选的」（codex）。「先试无 soft prompt 的跨具身共训——最小改动、零风险、实现量是 StarX 的 1/3、风险是它的 0」（mimo）。
阶段做什么验证 / 决策门
V0（先做）
无 soft prompt 的跨具身共训		多数据集混训 PI_v3 head：统一 EmbodimentSpec 做 pad/unpad + DomainProj 只在 I/O 端 + domain-balanced 采样。不要 soft prompt、不要每层 DomainAwareLinear。把跨具身当「更多 head 数据 + 正则」而非条件化机制——flow-matching 去噪器本就学「VLM feature → action」的通用映射。跑 E1：V0 在 SO-101 strict 20-round 是否 > 单具身 63.3%？若 V0 就赢 → soft prompt 整个故事不必要，到此为止（codex/mimo 都明确这条退出）。
V1（仅当 V0 不够）
加 gated soft prompt		仅当消融证明「I/O 投影 + domain-balanced 共训」不足以区分 embodiment 时，才按 §5 加带门控（alpha_l=0 初始化）的逐 cross-attn 层 soft prompt + 范式 1 的 prompt-dropout 训练（为范式 2 适配铺路）。V1 相对 V0 的净增益必须 > 它带来的复杂度/参数成本，否则回退 V0。

这把原设计从「一步到位三轴」改为「证据驱动渐进」：V0 本身已是完整可发布的跨具身 PI_v3；soft prompt（轴 2 的「条件 token」部分）降级为待证伪的增强，而 EmbodimentSpec + DomainProj（轴 2 的「动作空间统一」部分）是无论如何都要的地基。

8. 在 SO-101 PickOrange 上的验证计划 · Validation

不搞空想，用项目已有的 strict 20-round leaderboard 当标尺，做一个能证伪的实验阶梯：

  1. E0 基线对齐：StarX-VLA 范式 3（N_emb=1, SO-101 only）应复现现有 PI_v3-8B 的 63.3%。复现不出来 = adapter 接错，先修。
  2. E1 跨具身预训练增益（用 V0 跑）：V0 = 无 soft prompt 的多数据集共训 {Bridge + RT-1 + SO-101}（统一 spec + I/O DomainProj + domain-balanced 采样），再在 SO-101 上 strict 20-round。假设：V0 > 纯 SO-101 63.3%（更多 head 数据 + 正则）。这是整个设计的决策门：V0 不赢，soft prompt 也大概率救不回来；V0 赢，则 soft prompt 是否还需要由 E1b 定。
  3. E1b 关键消融（决定 soft prompt 存废）：V0（无 soft prompt，仅 domain-balanced + I/O 投影 + robot-ID one-hot）vs V1（加 gated soft prompt）。若 V0 ≈ V1 → soft prompt 故事被证伪，砍掉（两审都要求这个消融）。
  4. E2 适配效率（仅当 V1 立住）：范式 2，冻共享 head 只学 SO-101 slot。假设：用 <5% 参数、<10% 训练步达到 E1 的大部分性能——但前提是范式 1 用 prompt-dropout 训过（见 §10 R7）；这是跨具身的省钱卖点，也是最可能失败的一环。
  5. E3 head 消融：同上但换 GR00T head，验证「PI_v3 的逐 cross-attn 层注入比 GR00T 末层注入更吃 soft prompt」。
诚实边界 / Honest scope：
  • 所有「>63.3%」是假设，不是承诺。跨具身预训练能否突破小数据天花板，要 E1 真跑出来。两审都指出：X-VLA 自己在 SO-101 从 1/9 慢慢爬，这是「冷启动没那么惨」的证据，不是「跨具身给了大 boost」的证据——「突破 63.3% 天花板」可能过于乐观，真增益更可能来自「更多 head 数据」而非 soft prompt 本身。
  • 多数据集共训的采样平衡 + 数据清洗是成败关键：Bridge 巨大且含大量失败轨迹 / 人类抖动 / 静止帧，SO-101 仅 60 条干净 demo——光温度采样不够，需去失败轨迹、去静止帧，否则干净小数据被噪声淹没（mimo）。
  • 数据语义错配 > 架构风险（两审一致，列为头号风险 R0）：Bridge/RT-1/SO-101 在 相机位姿、action frame、控制频率、gripper 编码、horizon、归一化上各不相同，盲混会教给共享 head 互相矛盾的动力学。这是比「soft prompt 怎么接」更先要解决的问题。
  • 本设计是架构方案；真正 ROI 排序仍按 .memory/vla-distill-data-diversity-roi：prompt 多样性 > DR > multi-clip > 训更长。跨具身属于「multi-embodiment」一类，ROI 待标定。

10. 风险清单 · Risks

风险概率影响缓解
R0 数据语义错配（两审列为头号）：相机位姿/action frame/控制频率/gripper 编码/归一化跨数据集不一致 → 教给共享 head 矛盾动力学高致命先做数据审计 + 统一帧/频率/归一化 + 去失败轨迹去静止帧；这比 soft prompt 怎么接更优先。属于 V0 的前置工作。

R2 冻结 VLM 下条件化信号太弱（X-VLA 原本主干可训，这里主干冻）中中走 head 侧注入（§5 方案 b）让条件化紧贴 action；必要时给 soft prompt 更大 K 或解冻 head 顶层。
R3 多数据集采样不平衡饿死 SO-101（Bridge 巨 vs 60 demo）高高温度采样 / 每 embodiment 配额；E1 监控 SO-101 子集 loss 单独下降。
R4 通道语义对齐错（gripper idx 跨具身放错维）中高EmbodimentSpec 集中声明 + round-trip 单测（P1）；这正是 spec 取代散落子类要解决的问题。
R5 跨具身预训练无增益（小数据天花板是 vision/data 本质限制，不是 head 容量）中中E1 可证伪；若无增益则结论本身有价值（负面归档，见项目惯例），退回范式 3 等价现状、零损失。
R7 范式 2（冻 head + 新 slot）根本不工作：冻住的 head 无法被「从未训练过的新 slot」引导（soft prompt 的有效性本依赖可训练 backbone）高高范式 1 共训必须 prompt-dropout / 多 slot 泛化训练让 head 学会「读任意 slot」；控制模式/运动学差异大的新机器人单 slot 救不了 → 退回少量解冻 head 顶层或小规模微调。E2 先证伪再依赖。
R8 DomainProj/DomainAwareLinear 参数爆炸 + 小 slot 过拟合（X-VLA 存整张 per-domain 矩阵）中中改用共享 linear + per-domain 低秩/LoRA + 仿射；DomainProj 只插 I/O 端、不每层插（§4）。
R6 显存：跨具身共训 batch 内多 embodiment低中8B PI_v3 冻 VLM 在 4090-48G 占 43.7G；共训不增 VLM 成本，主要是 dataloader。OOM 第一杠杆仍是 repeated_diffusion_steps（见 .memory/starvla-vlm-variant-2b-4b-8b）。

12. 开放问题 · Open questions

  • Q1：soft prompt 注入 PI_v3 应该每层都注（同 LayerwiseFM 逐层）还是只注前几层？逐层更强但参数/算力更多。
  • Q2：DomainProj 用 X-VLA 的「每 domain 独立全套权重」（参数随 N_emb 线性涨）还是「低秩 per-domain LoRA」（更省，但容量上限低）？小数据下后者可能更稳。
  • Q3：跨具身预训练用现成 X-VLA base 的 soft prompt 权重初始化（但它配 Florence2，特征空间不同，未必可迁），还是全新随机初始化？倾向后者 + 借 StarVLA 已发布的 Bridge-RT-1 PI_v3 ckpt 暖启 head。
  • Q4：gripper 跨具身语义不一致（SO-101 连续度数 vs Bridge 二值）——EmbodimentSpec 的 gripper_loss 字段够不够？是否需要每具身独立 gripper 反归一化表？

13. 评审记录 · Review log

2026-06-08 两个独立模型评审，均自行读取并比对了 X-VLA / PI_v3 真源码（非泛泛而谈）。结论高度一致，已据此把设计从「一步到位三轴」改为「证据驱动渐进（V0→V1）」。见 .memory/feedback-mimo-independent-review 习惯。

两审一致的核心结论（已全部纳入修订）：
  1. 重新定性：这不是「移植 X-VLA 条件化」，而是「受启发新建 domain-conditioned PI_v3 head」——X-VLA soft prompt 的有效性依赖全可训练的 transformer，StarX 里 VLM 冻结后语义已变。→ 改 TL;DR + §7.5。
  2. 先做 V0（无 soft prompt）：最小有用轴 = EmbodimentSpec + DomainProj；soft prompt 在消融证明必要前都是可选项。「实现量 1/3、风险 0」。→ 新增 §7.5 + 改 §8 E1/E1b 决策门。
  3. 范式 2（冻 head + 新 slot）存在性存疑：冻住的 head 无法被没训练过的新 slot 引导。→ §7 标「未验证」+ §10 R7。
  4. EmbodimentSpec 纯数据不够：rotation/delta/clip/gripper hysteresis/双臂耦合需代码 → 「数据优先 + 命名 hook」+ GripperSpec 子结构。→ 改 §4。
  5. DomainAwareLinear 参数爆炸：X-VLA 存整张 per-domain 矩阵，StarX 须改低秩/LoRA + 只插 I/O 端。→ §4 + §10 R8。
  6. 数据语义错配 > 架构风险：列为头号 R0。→ §8 honest scope + §10 R0。
codex gpt-5.5（要点）：
  • 「不是 grafting，是 build a new domain-conditioned PI_v3 head inspired by X-VLA」；soft prompt 变成 action DiT 的逐机器人条件 token。
  • head-side 注入方向对，但 KV 直接 concat 会让 DiT 忽略 vl_embs_list[layer_idx]（LayerwiseFM_ActionHeader.py:335）→ 必带 gated adapter，alpha_l=0 初始化，使 N_emb=1 数值近似原 PI_v3、E0 才有意义。
  • EmbodimentSpec 要 data-first + named transform hooks；「pure data-only is naive」（引 action_hub.py:55/530）。
  • E1 plausible not strong；真增益多半来自「更多 head 数据」，需做 no-soft-prompt + robot-ID one-hot 的消融，赢了则 soft prompt 故事不必要。
  • 最小有用轴 = EmbodimentSpec + DomainProj，soft prompt optional until proven。
opencode mimo-v2.5-pro（要点）：
  • 致命问题 = 范式 2 的存在性：随机初始化的新 slot 想影响冻住的 head，信号传不出去；X-VLA 无此问题因 transformer 是活的。
  • PI_v3 interleave_self_attention=True → 只有偶数层（18/36）做 cross-attention，「每层注入」实为「每偶数层」，设计图需声明。
  • gripper 需子结构（双臂多 idx / 多 DOF / 各自 norm）。
  • Bridge/RT-1 含失败轨迹+抖动，PI_v3 head 容量小于 X-VLA 24 层 transformer，需数据清洗，否则 60 条干净 demo 被淹没；X-VLA 自身增益更像「冷启动没那么惨」而非大 boost。
  • 更简单替代：无 soft prompt 的跨具身共训（统一 spec pad/unpad + I/O DomainProj + domain-balanced），把跨具身当数据增强；「先试这个，失败再加 soft prompt」。
  • 评分：🟡 凑合 —— 方向对但过度设计。
