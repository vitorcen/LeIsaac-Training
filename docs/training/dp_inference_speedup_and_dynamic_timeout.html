DP DDIM 加速 + eval wall-clock timeout 动态算法 Speeding up DP inference with DDIM & user-patience-based dynamic eval timeout

DP DDIM eval-recipe wall-clock SO-101 PickOrange

作者vitorcen 日期2026-05-17 背景本地 DP 267M 在 100k step ckpt 上 eval，inference 393 ms / chunk 抢满 GPU SM，把 Isaac sim 渲染挤掉帧；shell timeout=600s 远不够 sim 跑完 120s × 3 round wall-clock，episode 在第一个 round 就被 SIGTERM。 结论(1) DP 慢的根因不是模型大，是 DDPM 100-step 串行采样；DDIM scheduler 不需要重训直接换；(2) eval timeout 不能按 inference latency 放宽，应该按 user-patience 封顶 — 跑不完的模型该判别为"不适合实时部署"。

0. TL;DR (One-page summary)

两行 config 改动：把 ckpt config.json 里 noise_scheduler_type 从 DDPM 改 DDIM、把 num_inference_steps 从 null（fallback 100）改成 GPU sweet spot（4090 上是 30–32）。重启 server load 一次，DP inference 从 393 ms → 147 ms。 Two-line config change: in the ckpt's config.json, switch noise_scheduler_type from DDPM to DDIM and set num_inference_steps from null (which falls back to 100) to a GPU-tuned sweet spot (30–32 on a 4090). Restart the policy server once, and DP inference drops from 393 ms to 147 ms per chunk.
动态 timeout 公式：total = startup + n_rounds × per_round_budget。per_round_budget 按用户耐心定（GR00T baseline 30 s / round，给 3× 容差 = 90 s）。不要按 inference slowdown 放宽 — 那是给慢模型放水。 Dynamic timeout formula: total = startup + n_rounds × per_round_budget. Set per_round_budget by user patience (GR00T baseline ~30 s/round, ×3 tolerance = 90 s). Do not stretch the timeout to accommodate slow inference — that just lets bad models limp through.

1. 起因：DP 把 GPU 3D 算力打满 (Trigger: DP saturates the GPU and starves the renderer)

沿用上一篇 postmortem 修好的 eval 配方（--policy_action_horizon=16 --sim_warmup_steps=30），首跑 DP 100k ckpt 时观察到： Carrying forward the fixed eval recipe from the previous postmortem (--policy_action_horizon=16 --sim_warmup_steps=30), the first DP 100k-step ckpt run produced:

  • 机械臂能放完前 2 颗橙子，第三颗后界面完全定格不动 The arm completes the first two oranges, then on the third the GUI freezes solid.
  • nvidia-smi 显示 GPU util 100%，但 Isaac sim 渲染明显掉帧、操作卡顿 nvidia-smi shows 100% GPU util, while Isaac sim's rendering visibly drops frames and the UI is laggy.
  • shell timeout=600s 设的，但 episode 1 都没跑完就被 kill The shell timeout=600s killed the process before episode 1 even finished.

1.1 用 nvidia-smi pmon 拆 engine

进程Compute SM%Graphics SM%说明
lerobot policy_server (DP)~97%0UNet 100-step diffusion 长时间占 CUDA SM
Isaac sim (rendering)0~10%RTX render head/wrist cameras，3D queue 排队等 SM 空闲
gnome-shell / Xorg0~3%桌面合成
洞察：nvidia-smi 顶部一个综合 util %，分不开 Compute (C) 和 Graphics (G)。pmon 才能拆。DP 这种长 CUDA kernel 持续占 SM → 3D 渲染 queue 等 SM 空闲 → 用户看到的"GPU 3D 算力打满"不是 leak，不是 spin-lock，是 Compute 抢占 Graphics。 Insight: nvidia-smi's headline util% is a single combined number — it can't separate Compute (C) from Graphics (G). Only pmon breaks them out. A long-running CUDA kernel from DP keeps the SMs busy, so the 3D rendering queue has to wait. The "3D engine pegged at 100%" the user sees is not a leak or spin-lock — it's Compute starving Graphics.

2. 根因 #1：DP 慢的物理本质是 DDPM 100-step (Root cause #1: the real bottleneck is 100-step DDPM, not model size)

实测分解：

指标值
DP 模型大小1018 MB safetensors (267M params × fp32)
单次 chunk 总 inference393 ms
diffusion 步数（DDPM training）100
每个 denoising step393 / 100 = 3.93 ms
UNet 单步算力需求很轻（4090 上不到 4 ms）
结论：DP 在 4090 上慢，不是因为模型太大，是因为 DDPM scheduler 要求串行跑完 100 个 denoising step 才能得到一次 action。UNet 本身飞快，量化 / 8bit / fp16 加速 1.5–2x 是错的优化方向。 Conclusion: DP is slow on a 4090 not because the model is large, but because the DDPM scheduler requires running all 100 denoising steps serially before producing one action chunk. The UNet itself is fast — quantization / 8-bit / fp16 (1.5–2× speedup at best) is the wrong optimization direction.

2.1 量化方向为啥不对 (Why quantization is the wrong direction here)

方案加速重训精度工作量是否真值得
DDIM 10–30 step + fp164–10×否几乎无损改 2 行 config✅ 立刻试

torch dynamic int8—否小CPU only，GPU 无效❌
TensorRT static INT8 PTQ3–4×否（需 calibration）中ONNX export + engine 编译，1 天仅在部署阶段考虑

记住：diffusion 类模型的第一优化永远是 减少 sampling step，不是量化权重。DDIM / DPM-Solver / consistency model 全都是这条路。SD 社区里 fp16 + 20-step DDIM 是部署标配，权重量化是边际优化。 Remember: for diffusion models the first optimization is always fewer sampling steps, not weight quantization. DDIM / DPM-Solver / consistency models all live on this path. In the SD community, fp16 + 20-step DDIM is the deployment default; weight quantization is a marginal add-on.

3. 修复 #1：DDIM scheduler 不重训直接 swap (Fix #1: hot-swap to DDIM scheduler without retraining)

LeRobot DP 实现里 _make_noise_scheduler 支持 DDPM 和 DDIM，且 DDIM 是 DDPM 的确定性子集，同一份 DDPM-trained 权重直接用 DDIM scheduler 推理是 OK 的（这是 Stable Diffusion 社区标准做法）。 LeRobot DP's _make_noise_scheduler supports both DDPM and DDIM, and DDIM is a deterministic subset of DDPM. The same DDPM-trained weights run correctly under a DDIM scheduler at inference time — this is the standard Stable Diffusion community workflow.

只需要改 pretrained_model/config.json 里 2 个字段：

{
  "noise_scheduler_type": "DDIM",   // was: "DDPM"
  "num_inference_steps": 32         // was: null → fallback 100
}

然后重启 policy_server（让它重新 load model with new config）。客户端无需任何改动。 Then restart the policy_server (it reloads the model under the new config). No client-side change is needed.

4. 根因 #2：DDIM 步数应该按 GPU 算力反推 (Root cause #2: DDIM step count must be derived from GPU compute headroom)

从 0 步开始加 step，实测 DP inference 时间近似线性： Measuring inference time as we increase DDIM steps gives a near-linear curve:

拟合公式（4090 + Isaac sim 并行渲染场景）： Empirical fit (on a 4090 while Isaac sim is also rendering):

inference_ms ≈ overhead + step × per_step_ms
              ≈ 36       + step × 3.3

# overhead   ≈ ResNet18 image encode + preprocess + ZMQ RTT
# per_step_ms ≈ UNet single denoising step on this GPU

4.1 Sweet spot 反推公式 (Closed-form for the sweet spot)

target_inference_ms = effective_chunk × (1000 / step_hz) × safety
                    = 8 × 16.67 × 0.85
                    ≈ 113 ms          # 0.85 留 15% 给 Isaac sim 渲染

max_steps           = (target_inference_ms − overhead) / per_step_ms
                    = (113 − 36) / 3.3
                    ≈ 23 step          # 安全档

# 临界档 (safety=1.0, sim 渲染开始抢但还能跑)：
target = 8 × 16.67 × 1.0 = 133 ms
max_steps = (133 − 36) / 3.3 ≈ 29 step  # 等价实验里测出的 30

4.2 不同卡的迁移 (Transferring to other GPUs)

卡预估 per_step_mssafety 0.85 推荐 stepsafety 1.0 临界 step
RTX 4090 (24G, AD102)~3.32329
RTX 3090 (24G, GA102)~5~15~20

The production-grade approach is to probe before launching (send one dummy observation to the server to measure real inference latency), then derive the step count from the formula. That removes the hard-coded table and adapts across GPUs automatically.

6. 根因 #3：eval timeout 应按 user-patience，不按 inference latency 放宽 (Root cause #3: eval timeout should be capped by user patience, not stretched for slow inference)

6.1 错误的算法（之前我写的）(Wrong algorithm — what I tried first)

# BAD — 给慢模型放水
slowdown      = inference_ms / (effective_chunk × realtime_per_step_ms)
total_timeout = startup + n_rounds × episode_s × slowdown × safety

# DP DDPM 100-step: 60 + 3 × 120 × 2.96 × 1.5 = 1653 s ≈ 28 min
问题：用户耐心 3–5 分钟。等 28 分钟跑完 3 round 没有意义。GR00T 30 s 完成一个 episode 就 success，那"能用"的标准就该是 30 s × 3 倍容差 = 90 s / round。把 timeout 拉到 28 分钟，本质是在帮 deploy-不该 deploy 的模型争取时间。 Problem: user patience is 3–5 minutes. Waiting 28 minutes for 3 rounds is pointless. GR00T finishes one episode in ~30 s — so the standard for "this model is deployable" should be 30 s × 3× tolerance = 90 s / round. Stretching the timeout to 28 minutes essentially buys time for models that shouldn't be deployed in the first place.

6.2 正确的算法 (The right algorithm)

# GOOD — 按 user-patience 封顶
STARTUP            = 60         # Isaac sim 冷启动 (s)
TIMEOUT_PER_ROUND  = 90         # GR00T 30s baseline × 3 tolerance (s)
total_timeout      = STARTUP + n_rounds × TIMEOUT_PER_ROUND
                   = 60 + 3 × 90 = 330 s   # ≈ 5.5 min

# 失败 mode: 模型跑不完 90s wall-clock 一个 round = 该模型不适合实时部署
# Failure mode: a model that can't finish a round within 90s wall-clock = not deployable in real-time.
设计哲学：timeout 是 判别器，不是 包容器。慢模型跑不到第 N 颗就被 kill，是正确的判决信号，应该报"DP DDPM 不适合实时部署"，而不是"再等 10 分钟试试"。 Design philosophy: timeout is a discriminator, not a buffer. A slow model getting killed before reaching the Nth orange is the correct verdict — it should be reported as "DP DDPM is not deployable in real-time," not "let's wait 10 more minutes."

6.3 wrapper 脚本 run_eval.sh (Wrapper script)

位置：~/work/isaaclab-experience/LeIsaac/scripts/evaluation/run_eval.sh

功能：

  1. 从 policy_server log 提取最近 50 次 inference 延迟平均值（informational only） Read the avg of the last 50 inference latencies from the policy_server log (informational only).
  2. 从 ckpt config.json 自动读 n_action_steps 算 effective_chunk Auto-read n_action_steps from the ckpt config.json to derive effective_chunk.
  3. 算 slowdown，仅作为警告输出（>2× 提示 "model too slow for 60Hz control"） Compute slowdown but only emit it as a warning (>2× prints "model too slow for 60Hz control").
  4. 实际 timeout 用 STARTUP + n_rounds × TIMEOUT_PER_ROUND 固定公式 Actual timeout uses the fixed formula STARTUP + n_rounds × TIMEOUT_PER_ROUND.
  5. 支持 --probe-only dry-run 查看 timeout 不启动 sim Supports --probe-only for a dry-run that prints the computed timeout without launching the sim.

用法：

bash run_eval.sh -- \
    --task=LeIsaac-SO101-PickOrange-v0 --eval_rounds=3 \
    --episode_length_s=120 --step_hz=60 --sim_warmup_steps=30 \
    --policy_type=lerobot-diffusion --policy_host=127.0.0.1 --policy_port=8080 \
    --policy_checkpoint_path=<ckpt> \
    --policy_action_horizon=16 --device=cuda --enable_cameras

# 环境变量调节
TIMEOUT_PER_ROUND=60 bash run_eval.sh -- ...   # 更严格：每 round 60 s
TIMEOUT_STARTUP=30 bash run_eval.sh -- ...     # 更短启动 buffer

8. 给以后做 LeIsaac eval 的清单 (Checklist for future LeIsaac evaluations)

  1. 启动 GPU 任务前先 nvidia-smi --query-gpu=memory.used,utilization.gpu + pgrep -af "policy_inference|isaac-sim" 检查残留进程 Before any GPU job, run nvidia-smi + pgrep to check for leftover processes that hold VRAM or stale server state.
  2. 用 run_eval.sh 而不是直接 timeout × N，让 timeout 跟着用户耐心走 Use run_eval.sh instead of a hand-rolled timeout N command, so the timeout follows user patience rather than inference latency.
  3. diffusion 类策略先看 ckpt config.json 的 noise_scheduler_type 和 num_inference_steps。DDPM + null → fallback 100 步 → 一定卡。改 DDIM + GPU sweet spot For diffusion-based policies, check the ckpt's noise_scheduler_type and num_inference_steps first. DDPM + null falls back to 100 steps and will choke. Switch to DDIM and pick a GPU-tuned step count.
  4. 用 nvidia-smi pmon 分 Compute (C) / Graphics (G) 看占用，单一 util% 数字会混淆 inference 和 rendering 的争抢 Use nvidia-smi pmon to separate Compute (C) and Graphics (G) utilization — the headline util% conflates inference vs. rendering contention.
  5. 多次实验取平均。task 有 randomization noise，单 episode 数据不能严肃归因到 DDIM 步数 / lr / 任何超参 Average over multiple runs. The task has initialization randomness, and single-episode outcomes cannot be rigorously attributed to DDIM steps, learning rate, or any hyperparameter.
