1. 背景与失败现象 (Background & failure mode)

本地 60 episodes / 36293 frames 的 LightwheelAI/leisaac-pick-orange 数据集，逐字复刻 shadowHokage/act_policy 的 config.json + train_config.json，10k step / batch 8 / lr 1e-5 / chunk 100，4090 上训了 11:35，loss 收敛正常。

eval 结果：3 round 全部 fail，机械臂能定位橙子、走过去、夹爪开合，但反复夹不中（与橙子有几厘米错位）。 Eval result: 0/3 across all rounds. The arm can visually locate an orange, drive toward it, and open/close the gripper, but it keeps missing by a few centimeters.

更糟的是：HF 上公开能成功的 shadowHokage/act_policy ckpt 在本机同 pipeline 下也 0/3。这就排除了"我们训练有问题"假设——问题在 eval pipeline 本身。 Even worse: the publicly-known-to-work shadowHokage/act_policy ckpt also failed 0/3 under our pipeline. This rules out "our training is broken" and points to the eval pipeline itself.

2. 三个真实根因 (Three real root causes)

2.1 根因 #1：Isaac sim 第一帧渲染崩坏 (Root cause #1: Isaac sim's first frame is visually broken)

用 LEISAAC_DUMP_FIRST_OBS=/tmp 同时 dump sim step 1 vs dataset frame 0 的 head/wrist 摄像头图像：

dataset frame 0sim step 1（无 warmup）
桌面温暖木质冷灰塑料
盘子白色蓝纹圆盘黑色圆环（plate 未渲染）
橙子3 颗橙子1 橙子 + 1 黑球（assets 加载未完成）
下半画面正常漆黑（viewport 未 settle）
光照暖光 DomeLight无 / DomeLight 未 warm up
失效机制：ACT chunk=100，policy 第一次 inference 基于这个 garbage 视觉做 100 step trajectory commitment。如果 inference 端把 action_horizon=100（错的），那 100 步全跑完前没有第二次观察机会，robot 走偏到不可救药位置。这就是 robot 看似在乱抓的视觉表现。 Failure mode: ACT with chunk=100 makes a 100-step trajectory commitment based on this garbage visual at the very first inference. If the client also runs action_horizon=100 (wrong), there is no second observation until all 100 steps have played out, by which time the arm has drifted into an unrecoverable state. That is why the robot looks like it's "blindly fumbling".

修复

# scripts/evaluation/policy_inference.py
parser.add_argument("--sim_warmup_steps", type=int, default=30, help=...)

def _sim_warmup(obs_dict):
    if args_cli.sim_warmup_steps <= 0:
        return obs_dict
    hold = obs_dict["policy"]["joint_pos"].clone()
    for _ in range(args_cli.sim_warmup_steps):
        obs_dict, _, _, _, _ = env.step(hold)
    return obs_dict

obs_dict, _ = env.reset()
obs_dict = _sim_warmup(obs_dict)            # env.reset 后
# ... 以及 controller.reset_state 分支、episode success/timeout 后

warmup 30 step (≈ 0.5s @ 60Hz) 后 sim step 1 dump 与 dataset frame 0 几乎一致（DomeLight 暖了 / plate 渲染好了 / 所有橙子加载完）。

2.2 根因 #2：step_hz 与 dataset 录制频率"对齐"是误判 (Root cause #2: forcing step_hz to match dataset recording rate was a misjudgment)

事实：dataset 60 episodes / 36293 frames / 平均 timestamp_max ≈ 20.1s → 605 / 20.1 ≈ 30 Hz。

误判：以为 eval 默认 --step_hz=60 让 chunk=100 actions 在 2x 速度下播放，所以改成 --step_hz=30。

反平反：LeIsaac upstream policy_inference.py 默认就是 step_hz=60，且这是有意的——sim control freq 与 policy inference rate 是解耦的。dataset record 30Hz 但 sim 跑 60Hz 没有问题，因为： Reversal: LeIsaac upstream's default is step_hz=60, and that is intentional — sim control frequency is decoupled from policy inference rate. A 30 Hz recording with a 60 Hz sim is fine because:
  • policy 看 obs → 出 16 步 action chunk → 客户端按 sim 60Hz 一步一步喂下去
  • 16 步在 60Hz sim 下 = 0.267s 真实时间，模型预测的 trajectory 跨越的物理时间还是合理的
  • 真正的对齐发生在 convert_lerobot_action_to_leisaac 内（motor deg → isaac rad），与频率无关
之前 step_hz=30 跑出来"看起来好像有点动"是巧合，不是修复。 Earlier step_hz=30 producing "kind of moving" output was coincidence, not a real fix.

2.3 根因 #3（最致命）：policy_action_horizon ≠ chunk_size (Root cause #3, the fatal one: policy_action_horizon is NOT chunk_size)

这是这次 postmortem 的核心发现。 This is the core finding of this postmortem.

参数含义训练 vs eval
chunk_size = 100ACT 模型一次 inference 输出 100 步 action 的 tensor训练超参，写入模型 config，inference 时固定不变
policy_action_horizon = 16客户端从 100 步里只取前 16 步执行，然后丢掉剩 84 步，重新观察重新推理eval 时的客户端策略，与模型无关

这就是 ACT 论文里的 temporal ensembling / receding-horizon control： This is the temporal ensembling / receding-horizon control trick from the ACT paper:

  • 训练时学习长 chunk → VAE latent 编码完整 trajectory，鼓励模型学整体规划 Train on long chunks → the VAE latent encodes the full trajectory, encouraging the model to learn global planning.
  • inference 时频繁 replan → 用最新观察纠偏，对 randomization / 动力学漂移鲁棒 Replan frequently at inference → correct drift with the latest observation, robust to randomization and dynamics drift.
我错在哪：在 act-training-plan.md 里写过 --policy_action_horizon=100 # 对齐 chunk_size。这是把训练超参和 eval 客户端策略搞混了。"对齐 chunk_size" 是反 LeIsaac 设计意图的——chunk 100 + horizon 16 才是设计意图。 Where I went wrong: in act-training-plan.md I wrote --policy_action_horizon=100 # match chunk_size. That conflates a training-time hyperparameter with a client-side eval strategy. "Matching chunk_size" goes against the LeIsaac design intent — chunk=100 + horizon=16 is the intended pair.

把 horizon 调到 100 的后果：

  1. policy 看 sim step 1（garbage 视觉）→ 出 100 步 action
  2. 客户端跑完全部 100 步才再次观察
  3. 这 100 步 ≈ 1.67s @ 60Hz，机械臂已经走偏
  4. 第二次观察时位置完全不可救药，第二次 inference 出的 chunk 又跑 100 步……

结果就是 robot 看起来"在乱抓"——其实是 commit 到错误轨迹后没机会改。把 horizon 调回 16 后，robot 每 0.267s 重新看一次场景，能纠错，前两颗橙子放置非常顺畅。

类比：horizon=100 = 开车前看一眼路线就闭眼开 1.67 秒；horizon=16 = 每 0.27 秒睁眼看一次。视觉伺服任务后者明显赢。 Analogy: horizon=100 is like glancing at the road once then closing your eyes for 1.67 s; horizon=16 is opening your eyes every 0.27 s. For visual-servoing tasks the latter wins by a wide margin.

2.4 三因合一的失效路径 (How the three causes compound)

env.reset()
  ↓ Isaac sim 第 1 帧 garbage（根因 #1 未修）
  ↓
policy inference 1 → garbage 100 步 trajectory
  ↓ horizon=100 死跑（根因 #3 未修）
  ↓
机械臂走偏到 OOD 位置
  ↓
inference 2 → 看到 OOD 视觉 → 又一个 garbage chunk
  ↓
...
60s 后 episode timeout，0 颗橙子在 plate

修复任一根因都不够： Fixing any single root cause is not enough:

  • 只修 #1（sim_warmup）：第一帧好了但 horizon=100 还是死跑，第 2 个 chunk 走偏后还是不可救药 Fix #1 only (sim_warmup): the first frame is good but horizon=100 still runs blind; once the second chunk drifts off, it's unrecoverable.
  • 只修 #3（horizon=16）：第一帧 garbage 喂进 policy 出 garbage chunk，前 16 步就够把 arm 推到 OOD Fix #3 only (horizon=16): the garbage first frame still feeds the policy a garbage chunk, and the first 16 steps alone push the arm into out-of-distribution territory.
  • 必须同时修 #1 + #3 Both #1 and #3 must be fixed together.

修复 #1 + #2 + #3 后，用本地 10k ckpt 跑 --eval_rounds=3 --episode_length_s=120 --policy_action_horizon=16： After applying fixes #1 + #2 + #3, the local 10k ckpt was run with the command above.

这是从 "完全夹不中" 到 "前 2 颗顺、第 3 颗磨叽" 的质变。模型本身工作正常。 This is a qualitative jump from "misses every grasp" to "first two clean, third one stuck". The model itself is working.

4.1 第 3 颗为什么磨叽 (Why the third orange gets stuck)

这是 ACT 在该 dataset 下的真实能力上限。可能根因排序： This is the real capability ceiling of ACT on this dataset. Ranked candidate causes:

假设估计概率验证成本
dataset 时序偏置：第 3 颗 pick 在 episode 后 1/3，且 + 回 rest_pose 是最长子任务（~10s），训练数据信号弱~60%改 terminations.py 放宽到 "2+ oranges in plate"，1 行改动跑 1 round
遮挡/物理碰撞：前 2 颗放入后 plate 已堆，第 3 颗被推到 plate 边缘 → 视觉 OOD~25%关 randomization（pose_range=0）跑 deterministic 1 round
horizon=16 还是太长，应该 1 或 4~10%horizon=4 / horizon=1 重跑（注意 inference rate 上限）
success 条件还要 "arm 回 rest_pose"，3 颗都进 plate 但 arm 没回中位~5%dump termination 内部值
关于 horizon 调更小：理论上 horizon=1 = 完全 closed-loop，每个 sim step 都用最新 obs 推理，对动态变化最敏感。但： On shrinking the horizon further: in theory horizon=1 is fully closed-loop — re-inferring on every sim step, maximally responsive to dynamic changes. However:
  • 60Hz sim 下 16 ms/step，ACT 80M 推理 ~10 ms，algorithmic latency + ZMQ RTT 可能让 horizon=1 触发流控停顿
  • ACT 的 VAE 学的是 100-step 连续 trajectory，chunk 内动作有运动平滑性；horizon=1 让相邻推理之间的预测 jitter 暴露
  • LeIsaac upstream 选 16 是 60Hz / 16 ≈ 3.75Hz replan，对追橙子位移/机械臂漂移已经足够
优先验证假设 #1（dataset 时序偏置），不优先调 horizon。 Verify hypothesis #1 (dataset temporal bias) first; do not start by tweaking horizon.

5. 给以后做 ACT-like 复刻的清单 (Checklist for future ACT-like reproductions)

  1. 训练参数逐字复刻参考 ckpt 的 config.json（chunk / dim_model / vae / normalization / lr / batch / steps），不要"凭直觉调整"。 Copy the reference ckpt's config.json verbatim for training (chunk / dim_model / vae / normalization / lr / batch / steps). Do not "tweak by intuition".
  2. eval 参数用上游仓库 policy_inference.py 的 default，不要把 policy_action_horizon 改成 chunk_size。 For eval, use the upstream policy_inference.py defaults. Do not change policy_action_horizon to match chunk_size.
  3. Isaac sim 启动后必须有 sim_warmup（≥ 30 step），否则 chunk-policy 第一次 inference 会喂 garbage 视觉。 After Isaac sim starts, always run a sim_warmup of ≥ 30 steps; otherwise a chunk-style policy's first inference will receive garbage visuals.
  4. 排查顺序：先 dump sim step 1 vs dataset frame 0 看视觉是否对齐，再看 action_horizon vs chunk_size 是否搞混，最后再怀疑 model/dataset。 Debug order: first dump sim step 1 vs dataset frame 0 to verify visual alignment; next check whether action_horizon was confused with chunk_size; only then suspect the model or dataset.
  5. byte-exact 等价不等于行为等价——同一份 ckpt 在不同 eval 配方下结果可以是 1/1 success vs 0/3 fail。eval pipeline 是模型的一部分。 Byte-exact equivalence is not behavioral equivalence — the same ckpt can produce 1/1 success or 0/3 failure depending on eval-time settings. The eval pipeline is part of the model.
