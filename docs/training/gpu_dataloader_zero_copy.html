GR00T 训练 GPU 不空转 — 视频零拷贝 dataloader 优化

Killing CPU dataloader bottleneck for GR00T-N1.7 fine-tuning on 5090/4090: pre-decoded RAM cache > NVDEC GPU decode > pin_memory/non_blocking > GPU augmentation.

日期：2026-05-23 · 任务：LeIsaac SO-101 PickOrange · 模型：GR00T-N1.7-3B · 数据集：50 demo / 30Hz / 480×848 H264 mp4

TL;DR 云端 5090 训练 N1.7 时 GPU util < 50%，原因不是模型太大，是 dataloader 拿 ffmpeg 软解码当主路径。 Opus + Codex (gpt-5.4) + Opencode (deepseek-v4-pro) 三模型一致结论：
  1. 把"每 step 现解 mp4"这个特殊情况干掉 —— 预解码到 npy memmap，训练 zero-decode → GPU util 50→85%
  2. pin_memory + non_blocking H2D 重叠 →再 +5-10%
  3. NVDEC GPU 解码作为备选（环境脆）
  4. 不要切 WebDataset / DataLoader2，那是伪优化
Pre-decode to RAM/NVMe memmap kills 95% of CPU dataloader cost on small (50 demo) datasets. Total cache for PickOrange 30k frame × 480×848×3 uint8 ≈ 34 GB — easily fits in 62 GB host RAM.

每 step 总账：micro_batch=8 × 单 sample CPU 解码+aug ≈ 200ms（4 worker 并行 ≈ 50ms），GPU forward+backward ≈ 150ms。理论上 4 worker 够喂 GPU，但实测 GPU util < 50% 说明：

  • worker 内部 VideoDecoder(device="cpu") 硬编码 + num_ffmpeg_threads=0（让 ffmpeg 自决，常退化成单线程）
  • 每次访问 sample 都 re-open mp4 + seek，cache miss 严重
  • 多 cam 时 albumentations apply_with_replay 串行执行
  • worker IPC 把整个 batch dict pickle 过来，包含 large uint8 frames

必做 #1 离线预解码 mp4 → npy memmap，训练零解码 — Pre-decode video to memmap, zero runtime decode

Why: 50-demo PickOrange 数据集只有 30k frame × 480×848×3 uint8 = 34 GB，完全可以 RAM-cache 或 NVMe memmap。"每 step 现解 mp4 + ffmpeg software decode" 是当前最大瓶颈，没必要每 epoch 重做同样的解码工作。

How:

# 一次性预处理脚本（伪代码）
import numpy as np, torchcodec
from pathlib import Path

CACHE = Path("/dev/shm/leisaac_cache")  # or NVMe
CACHE.mkdir(parents=True, exist_ok=True)

for ep in range(50):
    for cam in ["front"]:
        vid = f"data/PickOrange/videos/chunk-000/observation.images.{cam}/episode_{ep:06d}.mp4"
        dec = torchcodec.decoders.VideoDecoder(vid, device="cpu", dimension_order="NHWC")
        frames = dec[:].data.numpy()  # (T, H, W, 3) uint8
        # 落 memmap，按 (frame, H, W, C) 连续布局，加 metadata header
        out = CACHE / f"ep{ep:03d}_{cam}.npy"
        np.save(out, frames)

# 训练 dataset 改一行：原来的 get_frames_by_indices() 替换为
# np.load(cache_path, mmap_mode="r")[indices]

改动点：

  • gr00t/data/dataset/lerobot_episode_loader.py:398 _load_video_data 接一个 cache_dir 参数：当 cache_dir/ep{i}_{cam}.npy 存在时直接 memmap slice，否则回落 torchcodec
  • 预处理脚本放 LeIsaac/scripts/finetune/gr00t/precache_videos.py
  • 统一 uint8 不做 normalize，留给 GPU 端

Expected gain: GPU util 50% → 80-85%，吞吐 ~1.7×

Risk: 磁盘/RAM 占用上升（小数据集无所谓）；多 cam 时按 cam 分文件，避免单 npy 太大

必做 #2 pin_memory + non_blocking + cuda.Stream H2D 重叠 — Pinned H2D + stream prefetch

Why: dataloader 即使数据准备好了，.to(device) 默认是同步阻塞的。开 pin_memory=True + non_blocking=True + 专用 cuda stream 可以让"下 batch H2D"和"当前 step forward"重叠。

How:

# trainer.py:248 改 dataloader_params
"pin_memory": True,                # HF Trainer 默认就是 True，确认没被关
"prefetch_factor": 4,              # 默认 2，加到 4 让 worker 多 buffer 一些

# 自定义 training step（或 hook 进 HF Trainer.training_step）：
class GpuPrefetcher:
    def __init__(self, loader, device):
        self.loader = iter(loader); self.device = device
        self.stream = torch.cuda.Stream()
        self._prefetch()
    def _prefetch(self):
        try: self.next = next(self.loader)
        except StopIteration: self.next = None; return
        with torch.cuda.stream(self.stream):
            self.next = {k: v.to(self.device, non_blocking=True) for k,v in self.next.items()}
    def __next__(self):
        torch.cuda.current_stream().wait_stream(self.stream)
        batch, self.next = self.next, None
        self._prefetch()
        return batch

改动点：

  • trainer.py:248 加 "prefetch_factor": 4
  • 可选：在 HF Trainer 外层用 GpuPrefetcher 包一层，最干净的做法是 monkey-patch _inner_training_loop

Expected gain: +5-10% GPU util（已经预解码后）

Risk: pinned memory 吃更多 host RAM（~prefetch_factor × batch × frame_size）；不会出错只会更慢

必做 #3 num_workers 重调 + persistent_workers — Tune workers

Why: 当前 DATALOADER_NUM_WORKERS=4 固化，i9-13900KF / 5090 服务器都有 16+ 物理核，浪费严重。但 worker 数 ≠ 越多越好：video decode 是 IO + CPU 重活，过多 worker 会让 ffmpeg context 互相挤。

锦上添花 #4 CPU image aug 迁移到 GPU — Move aug from albumentations to kornia/torchvision.v2 on CUDA

Why: image_augmentations.py 用 albumentations 在 worker 内做 color jitter / crop / mask transform，串行 + PIL 转换有开销。如果迁到 GPU（kornia 或 torchvision.transforms.v2），可以在已经 H2D 的 batch 上一次性 vectorize。

How:

# worker 只产出原始 uint8 frame，不做任何 aug
# 训练 step 开头：
batch_imgs = batch_imgs.to(device, non_blocking=True)  # uint8 (B, T, C, H, W)
batch_imgs = batch_imgs.float() / 255.0
batch_imgs = gpu_aug(batch_imgs)   # kornia.augmentation.ColorJitter / RandomCrop on cuda

Expected gain: 配合预解码后 +3-8%；不预解码时这个收益被 video decode 掩盖看不出来

Risk: 数值实现差异（albumentations vs kornia 的实现细节不完全等价），seed 不可复现，需要重新调实验

不要做 #5 切 WebDataset / DataPipes / DataLoader2 — Don't migrate dataloader framework

Why not: WebDataset/DataPipes 主要解决"海量远程数据流"问题。LeIsaac 50 demo 是个本地小数据集，瓶颈在解码不在 IO。换框架只会增加复杂度 + 引入新 bug + 破坏现有 Trainer 兼容，零收益。

同理：不要试图魔改 HF Trainer 换成 Lightning / 自己写 loop。Linus 第一铁律是不破坏，第三铁律是实用主义。

4. 本机实测结果 — Bench results (RTX 4090 / i9-13900KF, 2026-05-23)

实测结论： memmap + pipeline non_blocking 联合最佳，GPU util 50% → 61.3%、wall-clock 92.6s → 76.5s (-17.4%)。但 steady-state it/s 卡在 1.99 不动 — CPU 100% 即使加了所有 patch 也满载，说明真瓶颈是每 step 的 albumentations image aug + Qwen3VLProcessor tokenize，不是 video decode 也不是 H2D。下一步必须攻 aug→GPU 才能再涨。

Memmap + non_blocking H2D pipeline overlap got us to GPU util 61% with 17% wall-clock improvement. But CPU stayed pegged at 100% — the actual ceiling is per-step CPU work (image aug + VLM tokenize), not the data movement Opus/Codex/Opencode initially flagged.

4.4 时间预算重算 — Recomputed step budget

每步 502 ms (1.99 it/s) 分配：

  • GPU forward+backward × 4 micro-batch（grad_ckpt + grad_accum=4）：~320 ms (~64%)
  • H2D + sync overhead（已减但仍存在）：~30 ms (~6%)
  • CPU per step（4 worker × 250 ms wall = 1000 ms wall-clock distributed）：~150-180 ms 主线程 + worker bottleneck
  • 剩余 ~150 ms 是 image aug (albumentations) + VLM tokenize (Qwen3VLProcessor) 的实际开销

把 albumentations 移 GPU 可省 ~80-100 ms / step → 期望 step/s 2.0 → 2.4-2.6 (~25% 真实 steady-state 涨幅)。这才是下一阶段的方向。

4.6 二轮外部评审后的最大单点优化 — 调 grad_accum 而非 dataloader — The single biggest win

Codex + Opencode 二轮评审打脸结果：我守着 dataloader 优化绕圈子，真正瓶颈是 GR00T 默认配置 GLOBAL_BATCH=8 GRAD_ACCUM=4 → micro_batch=2。把 GRAD_ACCUM=1（保持 effective batch=8）单点提速 +178%。

External 2-model review revealed: the dataloader was a sideshow. The single biggest win is reducing grad_accum from 4 → 1 (keeping effective batch=8), which gives +178% steady step/s.
配置wall (80 step)steady it/svs baseline steadyGPU mem peak
① baseline (mb=2, ga=4)92.6s1.99—15.0 GB
④ memmap+pipe (前面最佳)76.5s1.990%15.0 GB
⑦ mb=4 ga=265.2s3.50+76%19.8 GB
⑧ mb=8 ga=152.3s5.54+178%19.7 GB
⑨ ⑧ + torch.compile70.3s5.73+188% (warmup 吃 18s)19.9 GB

原理（为什么 dataloader 优化 +0% steady 而 grad_accum 调整 +178%）：

  • 原 mb=2 ga=4：每 outer step 跑 4 micro-batch × 2 sample = 8 sample，但 4 个 GPU 同步点 + 4 次 Python 调度 overhead + 4 次小 matmul 不能充分喂饱 4090 Tensor Core。
  • 新 mb=8 ga=1：每 outer step 跑 1 micro-batch × 8 sample = 8 sample，1 个同步点 + 1 次调度 + 1 个大 matmul 充分用 SM。
  • VRAM 几乎不涨（15→19.7 GB）：optimizer state（adafactor）+ param storage 占大头，activation 增量小。

反直觉细节：GPU util mid_window 看着从 61% 跌到 21%。不是变差，是 step time 缩到 180 ms 后 1Hz sampler 大概率抓到 step 间 idle。真信号：train_steps_per_second（收敛值）和 GPU util p90（仍 86-90%）。

torch.compile(action_head.forward)：steady 仅 +3.4% (5.54 → 5.73)，JIT warmup ~18s 让 80-step 短跑反而慢 34%。6000+ step 真训练才回本。建议默认开但短验证关。

flash-attn：CN→GitHub release wheel 下载反复 EOF/corrupted（ghfast 镜像也炸），ROI 预期仅 +5-12%，跳过。云端跑可装。

5. 证伪：collator 拆分把 image_proc 移到 worker 是 反优化 — Falsified hypothesis

假设：把 Qwen3VLProcessor.image_processor() 从主线程 collator 拆到 worker 并行做，预期 step/s 1.99 → 2.85 (+43%)。

实测结果（2026-05-23，本机 RTX 4090 + i9-13900KF）：

配置runtimestep/sGPU mid util
memmap + 4w + non_blocking76.5s1.0561.3%
+ collator split81.8s0.9843.4%

根因：

  1. CPU 已 100% 饱和（8 P-core + 16 E-core 都跑满），workers 之间已经在抢 P-core。
  2. 主线程 collator 通过 non_blocking=True 已和 GPU forward 天然重叠（GPU 在算时 collator 在准备下一 batch）。
  3. 把 image_proc 移到 worker = 让 worker 多做事，worker 喂数据更慢 → GPU 等更多 → util 反而下降。
  4. 实测估算错：我以单次 16 imgs 的 43.8 ms 估收益，但实际 collator 每次只 4 imgs（micro_batch=2 × n_cam=2）→ ~10 ms / call，节省被 worker 阻塞吞没。

结论：默认 不启用。代码留在 launcher 内，COLLATOR_SPLIT_ENABLE=1 显式开。仅当满足以下条件才考虑：

  • micro_batch × n_cam × T ≥ 16 imgs / collator call
  • 主线程 collator 是单点瓶颈（profile 显示主线程占 step ≥ 40%）
  • worker 仍有 CPU slack（CPU 总 util < 70%）

Sweet spot：H100 96GB / RTX PRO 6000 大 GPU 上 micro_batch=8 × 2 cam = 16 imgs / call 的场景。4090 24GB 上 grad_ckpt + micro_batch=2 配置不在此 sweet spot。

Lesson: At CPU saturation, moving work from main thread to workers is zero-sum and breaks the natural overlap with GPU forward (already established by non_blocking H2D).

6. Opencode 独特发现：当前代码每次解码整个 episode — A wasteful path Opus + Codex both missed

定位： gr00t/data/dataset/lerobot_episode_loader.py:603

# LeRobotEpisodeLoader.__getitem__
video_data = self._load_video_data(episode_id, np.arange(actual_length))
#                                              ^^^^^^^^^^^^^^^^^^^^^^^^^
#                                              整个 episode 所有 600 frame

问题： shard sampler 在拿到整 episode 后只用其中 50-100 step（按 batch / time-window 切），但 episode_loader 在这一行强制解码所有 600 帧。等于 6-12× 浪费 CPU decode。

修法（两条路）：

  • 路 A（彻底）： 重构 shard 接口让 sampler 把 "需要哪些 step index" 传下来，__getitem__ 只解码这些；改动面大，touch shard/sharded_single_step/sharded_mixture 三处
  • 路 B（推荐）： 配合 §3 预解码 memmap，整 episode 加载从 ~60ms decode 退化为 ~60μs memmap read — 这个浪费被 §3 完全吸收，不需要单独修

这是 opencode 读源码时发现的 quick win；Opus 和 Codex 都没读到这一层。但因为 §3 预解码本身就让 decode cost 归零，这个发现的实际价值是定义了"如果暂时不上预解码"的兜底优化。
