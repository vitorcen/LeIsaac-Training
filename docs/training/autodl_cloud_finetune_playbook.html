AutoDL 云端微调实战手册

AutoDL cloud finetuning playbook — observed limits, working configs, and recovery patterns for VLA training (GR00T N1.7 / X-VLA / SmolVLA on RTX 4090 / RTX PRO 6000)

2026-05-22 v1 · 基于 LeIsaac GR00T-N1.7 finetune 实战环境搭建总结

0. TL;DR

  • 选机：单 4090 24GB 是 N1.7 的 "刚好够" 配置；RTX PRO 6000 96GB 可放心开大 batch，训练快 3-4×
  • 磁盘：默认 50GB 不够，必须扩到 100-140GB；旧机不可扩需"克隆系统盘 + 数据盘到新机"
  • HF 加速三件套：公开 repo → hf-mirror.com；gated repo → /etc/network_turbo + canonical HF；大文件 → 单流 curl（多线程被代理掐死）
  • PyTorch wheel（本机 uv sync）：默认 download.pytorch.org 不通 TLS handshake EOF；必须改 aliyun mirror https://mirrors.aliyun.com/pytorch-wheels/cu128（14 MB/s 稳）
  • 模式切换：clone repo / 下载基础模型用无卡模式（便宜）；uv sync（torch + tensorrt build）必须 GPU 模式（无卡 nvidia-smi 权限拒绝）
  • ckpt 策略：save_only_model=True 单 ckpt 12GB；LossDrivenPrune 留 top-K 自动删差的；100GB 可留 6 个 ckpt

1.2 磁盘容量

大小结论能放

2.3 大文件 LFS：curl 单流

实测多线程在 AutoDL 代理下全部失败：
  • hf download --max-workers 8：httpx.RemoteProtocolError: Server disconnected
  • aria2c -s 8：连接 CN:1 DL:0B 卡死
  • git clone（含 LFS）：LFS smudge filter 多线程拉文件，同样掉
  • 唯一稳的：curl 单 connection

反直觉：bandwidth 不是被限制，是 connection 数 被代理限制。单流 ~600 KB/s 稳定，多流被掐到 0。

# 大 LFS 文件下载脚本模板
nohup bash -c '
URL="https://huggingface.co/nvidia/Cosmos-Reason2-2B/resolve/main/model.safetensors"
TOKEN="hf_xxx"
OUT="model.safetensors"

source /etc/network_turbo >/dev/null 2>&1

# 关键参数：
# --speed-time 60 --speed-limit 10240  →  60s 内速度低于 10KB/s 自动 abort，触发 retry
# --max-time 14400  →  单次连接最长 4 小时
# --retry 50 --retry-delay 15  →  最多 50 次重试，每次等 15s
# ⚠️ 不要用 -C - (resume)，跨 session resume 会写出 sparse 空洞文件
curl -L --retry 50 --retry-delay 15 --retry-max-time 14400 \
     --max-time 14400 \
     --speed-time 60 --speed-limit 10240 \
     -o "$OUT" \
     -H "Authorization: Bearer $TOKEN" "$URL"
' > download.log 2>&1 &
⚠️ sparse 空洞陷阱（实战踩过）
多次中断的 curl -C - resume 跨 session 会在错误偏移写入字节，剩余部分变成 NULL 空洞。stat 显示文件 4.6 GB，du 实际只 128 MB（其余是空洞）。
检测：stat -c "%s %b" file — Size 和 Blocks×512 不一致就是空洞。
修复：删除重下；不用 -C -，单次 curl 跑到底。

4. uv + tensorrt 在无卡模式的坑 (uv sync OOMs without GPU)

无卡模式下 uv sync 卡死在 tensorrt-cu12 build： 
Failed to generate package metadata for tensorrt-cu12-libs ...
  PermissionError: [Errno 13] Permission denied: 'nvidia-smi'
原因：tensorrt-cu12-libs 的 setup.py 探测 GPU 能力时调 nvidia-smi，AutoDL 无卡模式权限阻止访问。

5.3 LossDrivenPruneCallback 实现

# 加到 launch_finetune_ckpt_n17.py 的 _patch_save_pruning() 旁边
class LossDrivenPruneCallback(TrainerCallback):
    """Keep top-K ckpts by training loss + always keep last 1.
    AutoDL 上没有 sim 跑 closed-loop eval，用 train_loss 当代理信号。"""
    def __init__(self, top_k=5):
        self.top_k = top_k
        self.ckpt_losses = {}  # step → loss

    def on_save(self, args, state, control, **kwargs):
        if state.log_history:
            last = state.log_history[-1]
            if 'loss' in last and last.get('step') == state.global_step:
                self.ckpt_losses[state.global_step] = last['loss']
        # 排名 + 选 winner
        all_steps = sorted(self.ckpt_losses.keys())
        last_step = all_steps[-1] if all_steps else None
        sorted_by_loss = sorted(self.ckpt_losses.items(), key=lambda x: x[1])
        keepers = set(s for s, _ in sorted_by_loss[:self.top_k])
        if last_step:
            keepers.add(last_step)
        # 删 loser
        out = args.output_dir
        for d in os.listdir(out):
            m = re.match(r'^checkpoint-(\d+)$', d)
            if m and int(m.group(1)) not in keepers:
                shutil.rmtree(os.path.join(out, d), ignore_errors=True)
                print(f'[loss-prune] del ckpt-{m.group(1)} '
                      f'(loss={self.ckpt_losses.get(int(m.group(1)), "?")})', flush=True)

10. 未实施的优化提议 (future work, not done yet)

10.1 NVDEC 硬件视频解码 (boost GPU util from 30% → 60-80%)

背景：VLA 训练 GPU util 通常只 30-40%，根因是 CPU bound — 每 batch 都要 torchcodec → FFmpeg 串行解 32-256 个 H.264 视频帧。Cosmos+DiT forward/backward 在 GPU 上只 ~0.5-1s，但 dataloader 准备 batch 要 3-5s。
项当前（CPU 解码）NVDEC 改造后（预期）
video decode 路径torchcodec → libavcodec → CPU H.264 → CPU memcpy → GPUtorchcodec(device='cuda') → NVDEC 硬件单元 → GPU memory 直读
每帧延迟10-50ms / frame1-3ms / frame
CPU 负载208 cores 满载（4-16 workers）近闲（仅文件 IO）
GPU util30-40% avg（实测）60-80% avg（理论）
10k step 时长~18h（per-step=8, A 配）~6-9h 估

10.1.1 改造步骤

# gr00t/utils/video_utils.py 大约 line 96 附近：
# torchcodec VideoDecoder 默认 CPU device，加 device="cuda" 参数

from torchcodec.decoders import VideoDecoder
decoder = VideoDecoder(video_path, device="cuda")    # ← 当前默认空，改成 "cuda"

10.1.2 风险 / 验证清单

  1. NVDEC 硬件兼容性：PRO 6000 / 4090 / 4090D / A100 / H100 都有 NVDEC（NVENC 不重要），但某些 H.264 profile（High 10 / High 4:4:4）NVDEC 不支持。需 ffprobe -v error -select_streams v:0 -show_entries stream=profile 验证 LeIsaac 数据集 profile。
  2. torchcodec device 行为：CUDA decoder 输出 tensor 在 GPU 上，转 numpy / albumentations 增强需要 .cpu()。要么把 augmentation 也搬 GPU（kornia 替代），要么接受这段 transfer overhead。
  3. 多 dataloader worker 与 CUDA context：每个 worker 进程 fork 后需要独立 init CUDA context，可能造成 GPU 内存碎片。建议从 workers=16 降到 4-8。
  4. 验证方式：smoke 50 step 测 GPU util（nvidia-smi --query-gpu=utilization.gpu --format=csv --loop=1）+ step time，目标 GPU util > 60%。

10.1.3 何时做

不在当前 session 做：当前 PRO 6000 + A 配（per-step=8, workers=4）已验证 6000 step 11h ¥77 可接受。NVDEC 改造需要 1-2h 验证 + 可能需要回滚，session 中段折腾性价比低。
下次新机一开始就做：先 smoke 验证 NVDEC 兼容，再启正式训练。预期 10k step 时间 ~6-9h vs 当前 18h，省 50% GPU 租金。
