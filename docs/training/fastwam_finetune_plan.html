Fast-WAM QLoRA 微调计划

SO-101 PickOrange · World Action Model 横评接入 · 2026-05-17

Plan for QLoRA fine-tuning Fast-WAM (Wan2.2-5B DiT + 1B action expert) on LeIsaac SO-101, as the 4090-tractable replacement for DreamZero (which is unviable on single 4090).

1. 为什么是 Fast-WAM 而不是 DreamZero · Why Fast-WAM, not DreamZero

核心架构差异imagine-then-execute（生成视频后再出动作）训练时 co-train video, 推理时 SKIP future imagination

关键洞察 · Key insight：Fast-WAM 解决了 DreamZero 推理慢的根源 — 它在推理时跳过 multi-step video denoise，只保留 action prediction。论文证明"future imagination is NOT needed for strong action performance"。这让 video-world-model 类模型首次进入单卡消费 GPU 可推理范围。

4.1 模型内部结构（实际代码逐行确认）

FastWAM 是 world model：拼装 Wan2.2 视频生成 DiT（5B 主干）+ ActionDiT（action 专家）+ WanVAE（视频编解码）+ UMT5（文本编码） 五件套，用 MoT (Mixture of Transformers) 编排 video / action 双 expert 共享 attention。训练 joint flow-matching loss，推理时 video 可关掉只跑 action 快路径。

组件来源实现参数估算
WanVideoDiT (video expert)Tongyi Wan-AI 2024（wan_video_dit.py:310）视频扩散 DiT，input/output 都是 latent video（VAE 编码后），continuous flow matching loss。~5B 参数，标准 RoPE-attention + Adaptive LN~5B
WanVAETongyi Wan-AI3D 视频 VAE，把 (H, W, T) 视频压成 latent；推理只跑 1 步~80M
UMT5 Text EncoderGoogle UMT5-XXL预编码 prompt embedding，全程 frozen；可 CPU 预算（避免 fp32 转向 GPU 时 22GB 瞬时高峰）~5.5B (frozen)
ActionDiT (action expert)FastWAM 自研（action_dit.py:32）小型 DiT 风格 transformer，复用 WanVideoDiT 的 DiTBlock 实现（同 RoPE 注意力）；额外有 action_encoder（linear）+ text_embedding（MLP）+ ActionHead（modulated linear）；continuous flow matching loss~1B
MoT (Mixture of Transformers)FastWAM 自研（mot.py）把 video + action 两个 expert 的 attention 串起来共享 KV，但 Q/MLP 独立（类似 SmolVLA 的 cross_attn 思路但更深度耦合）orchestrator only
Proprio encoderFastWAM 自研（fastwam.py:59）simple Linear(proprio_dim → text_dim)，把 robot state 投到跟 text embedding 同维~1M
合计——~12B（含 UMT5；QLoRA 训只动 39M LoRA params）
SKIP（不生成视频） • action 从 N(0, 1) 起步 • for i in range(num_steps): 　 t_i + RoPE + flow step 　 update action_t • num_steps 默认 ~10-20 → ~220ms / chunk @ 4090 → chunk=24 → sim 一次塞 24 步 • 训练 5B world model 但 推理只跑 1B action expert

读图要点：

  • FastWAM 是 world model 范式：训练时同时学 "未来视频" 和 "未来动作"，loss 加权求和。推理时 video 可关掉（只走快路径），所以部署 latency 不至于太可怕
  • 核心是 MoT (Mixture of Transformers)：video 和 action 两个 expert 在每层 attention 共享 KV cache 但有独立 Q/MLP（跟 SmolVLA 的 cross_attn 类似但更深度耦合）
  • 真正决定 robot policy 性能的是 ActionDiT，~1B 参数；WanVideoDiT 的 5B 在推理时是 dead weight
  • 训练 joint loss + 推理时只跑 action 是 FastWAM 的核心 trick — 用大世界模型监督但不付推理代价
  • 我们 2026-05-18 失败的根因：50 demo 数据量太小（OFT/SmolVLA 是 100-500 demo），过参数化 → step 8000 之后 extrapolate 出训练 range，gripper 失控

# 单帧 inference (待自己 reverse-engineer 出 API)

Phase 4 — Server + Isaac Sim demo（~2h）

6. 风险点 · Risks

  1. flash-attn 编译坑：已知 nvcc 12.6 + glibc 2.34 冲突（LingBot 已踩）；解法用 prebuilt wheel
  2. 单 GPU benchmark hack：LIBERO eval 默认 8 GPU；我们绕过 benchmark 直接调 model.predict_action，所以无所谓。但训练默认 deepspeed zero1 也是 8 GPU，需要降到 batch=1+grad_accum 单 GPU
  3. Wan2.2 量化兼容：DiT 不是标准 HF AutoModel，bnb 4-bit 加载需要侵入 fastwam.models.wan22.helpers.loader.load_wan22_ti2v_5b_components — 1-2 天工作
  4. 没有 SO-101 base ckpt：LIBERO ckpt 是 Franka EEF + LIBERO 视觉分布。LeIsaac 与 LIBERO 视觉差异大（Isaac Sim vs MuJoCo）+ action 不同 — fine-tune 后可能仍然差
  5. 训练 step 时长不确定：估 2-4 s/step 是基于 DiT 经验，没实测。可能 5-10 s/step（DiT heavy）→ 10k step 跑 1-3 天

9. 三个 HF ckpt 怎么协同 · Three-checkpoint cooperation

三个仓库一起塞进同一个 FastWAM 实例，角色完全不同：

HF repo大小 (disk)角色在 FastWAM 内的用途
Wan-AI/Wan2.2-TI2V-5B25 GBVideo DiT 主干① preprocess 阶段：5B DiT 权重 1-D 线性插值 → ActionDiT init（hidden 3072→1024）；② 推理：Video DiT 跑 1 次 prefill 把 frame-1 latent 编进 KV cache（不跑 video denoise）
Wan-AI/Wan2.1-T2V-1.3B17 GB公共组件源提供 UMT5 tokenizer / text encoder + WanVAE（redirect_common_files=true，Wan2.2 借用 Wan2.1 的这几个共用文件）
yuanty/fastwam23 GB训练 ckptlibero_uncond_2cam224.pt (12 GB) + robotwin_uncond_3cam_384.pt (12 GB)：load_checkpoint 时覆盖 ActionDiT + 微调 Video DiT 权重（LIBERO 7-DoF Franka EEF 训练产物）
关键洞察 · Key insight：Fast-WAM 的“Skip imagination”指的是推理时跳过 video 去噪，但 Video DiT 本身仍然在场 — 它的 KV cache 是 Action Expert 的视觉条件源。不是“丢掉 video pipeline”，而是“video 跑 1 step（编码）+ action 跑 20 step（去噪）”。
Skip-imagination ≠ drop the video DiT. The Video DiT still runs 1 prefill pass to seed the KV cache that the Action Expert attends to. The skip is on video denoising, not on video encoding.
infer: 1× prefill only UMT5 text enc Wan2.1 weights prompt → context may offload to CPU WanVAE Wan2.1 weights img → latent fp16, no quant Action Expert (1B) yuanty/fastwam ckpt flow matching head infer: 20× denoise yuanty 同时改写 ActionDiT 和 Video DiT 微调权重 输入 · Input 2 cam front+wrist → 横拼 224×448 + 8-D proprio + prompt 文本 编码 · Encode (once) UMT5 → text context WanVAE → frame latent Video DiT prefill → KV cache (frozen) 去噪 · Denoise (20 step) Action Expert flow-match attends KV cache + ctx 无 video denoise step ~190ms 总耗时 输出 · Output action chunk [24, 7] 7-D = EEF Δxyz3 + axisang3 + grip1 ⚠ SO-101 需 6-DoF joint → cosmetic remap × Δscale

9.1 加载顺序（代码层）· Load order in code

# 1. preprocess（一次性 setup，~10min）— 把 Wan2.2 5B 权重压成 1B ActionDiT init

  --model-config configs/model/fastwam.yaml \

    ...
