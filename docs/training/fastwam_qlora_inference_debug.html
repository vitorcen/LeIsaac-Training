2. Root cause 链路 (Root Cause Chain)

  1. 训练时 model 有多 alias attribute 指向相同 submodule：
    • self.mot ← 主
    • self.dit = self.mot
    • self.action_expert = mot.mixtures.action
    • self.video_expert = mot.mixtures.video
  2. accelerate.save_state → safetensors dedupes shared tensors → 同一权重只在一个 alias 路径下保存。saved state_dict 8324 keys 分布混乱：action_expert 2839 / video_expert 2835 / mot 1226 / dit 1226 / vae 196.
  3. 关键洞察：action_encoder / action_head / proprio_encoder 三个 SO-101 specific 的 6-D 小 Linear 在训练开始时是 random init（Wan2.2 5B base 里不存在这些 6-D 形状），它们的学到的权重 必须从 ckpt 恢复，无法 fresh reproduce。
  4. 当前 inference 路线 A (fresh build + LoRA wrap + NF4 quant + load_state_dict all keys)：missing=5218 unexpected=5517，三个小层的 bnb 量化权重在 alias 错位下没正确 load。
  5. 当前 inference 路线 B (filter out bnb buffers, load only LoRA + bf16 small layers)：三个小层的 bnb-packed uint8 weight 被我误当 bnb base 跳过，结果它们保留 fresh random quantize → garbage output。

3. 修复路线 (Repair Paths) — codex 建议

路线 C：现有 ckpt 救援（立即可做）

Fresh build (bf16, 不量化) → LoRA wrap → 把 ckpt 里的 bnb 4bit packed weight 手动 dequantize 回 bf16 → load_state_dict 进 bf16 model

from bitsandbytes.functional import QuantState, dequantize_4bit
from safetensors.torch import load_file

def dequant_bnb_in_state_dict(sd):
    """Walk sd, find each bnb-saved 4bit weight, dequant to bf16, return clean sd."""
    BNB_META = (".absmax", ".nested_absmax", ".nested_quant_map",
                ".quant_map", ".quant_state.bitsandbytes__nf4")
    # Find base keys (those with quant_state metadata)
    base_keys = set()
    for k in sd:
        if k.endswith(".quant_state.bitsandbytes__nf4"):
            base_keys.add(k[: -len(".quant_state.bitsandbytes__nf4")])

    out = {}
    for k, v in sd.items():
        if any(k.endswith(suf) for suf in BNB_META):
            continue  # consumed below
        if k in base_keys:
            # Reassemble qs_dict for QuantState.from_dict.
            # bnb expects keys WITHOUT the parameter prefix.
            qs_dict = {}
            for suf in BNB_META:
                if k + suf in sd:
                    qs_dict[k + suf] = sd[k + suf]
            qs = QuantState.from_dict(qs_dict, device="cuda")
            packed = v.to("cuda")
            bf16 = dequantize_4bit(packed, qs).to("cpu", dtype=torch.bfloat16)
            out[k] = bf16
        else:
            out[k] = v
    return out

# Usage:
sd = load_file(ckpt_path, device="cpu")
sd = dequant_bnb_in_state_dict(sd)
model = build_fastwam(bf16, no_quant=True)
apply_lora(model)
missing, unexpected = model.load_state_dict(sd, strict=False)
Pitfalls:
  • QuantState.from_dict 期望 qs_dict 的 key 是 剥掉模型 prefix 的 short keys ("absmax", "quant_state.bitsandbytes__nf4" 等)，源码里 qs_dict.pop(first_qs_key) 然后 k.split(".")[-1] 处理 — 注意 key 格式
  • 多 alias dedupe 意味着同一 tensor 在不同 alias 路径下都要恢复（load_state_dict 会自动通过共享内存解决，但 alias 必须都被 model.state_dict() 暴露）
  • 显存：bf16 inference ~13GB（vs NF4 推理 ~5GB），4090 24GB 仍 fit

路线 D：训练时另存干净 LoRA 文件（防范未来）

训练 hook 里另存 adapter_small_layers.safetensors，只包含 LoRA + 小层 bf16，跟 accelerate 的 bnb-毒-state_dict 解耦：

SAVE_PREFIXES = ("action_encoder.", "action_head.", "proprio_encoder.")

def export_adapter_and_small_layers(model, path):
    sd = model.state_dict()
    out = {}
    seen = set()  # dedupe by tensor id (multi-alias)
    for k, v in sd.items():
        if id(v) in seen: continue
        ck = k.split(".", 1)[1] if k.startswith(("mot.", "dit.")) else k  # canonicalize
        is_lora = ".lora_A." in ck or ".lora_B." in ck
        is_small = any(p in ck for p in SAVE_PREFIXES)
        if not (is_lora or is_small): continue
        seen.add(id(v))
        out[ck] = v.detach().cpu().to(torch.bfloat16)
    safetensors.torch.save_file(out, path)

# Wire via accelerator.register_save_state_pre_hook
def hook(models, weights, input_dir):
    export_adapter_and_small_layers(
        accelerator.unwrap_model(models[0]),
        Path(input_dir) / "adapter_small_layers.safetensors"
    )
accelerator.register_save_state_pre_hook(hook)

@torch.no_grad()
def merge_manual_lora(m):
    scale = m.alpha / m.r
    delta = (m.lora_B.weight @ m.lora_A.weight) * scale
    m.base_layer.weight.add_(delta.to(m.base_layer.weight.dtype))
    return m.base_layer

for name, mod in list(model.named_modules()):
    if isinstance(mod, ManualLoraLinear):
        parent_name, leaf = name.rsplit(".", 1)
        setattr(model.get_submodule(parent_name), leaf, merge_manual_lora(mod))
safetensors.torch.save_file(model.state_dict(), "fastwam_merged_bf16.safetensors")

7.2 Root cause (修订版) (Updated Root Cause)

原 §2 root cause 链路（多 alias dedupe + bnb buffer 错位）是次要，真正吃掉权重的是训练侧两个 "好心办坏事" 的 patch：

毒患 A：safetensors.load_file strip bnb buffers
症状：phase 1 干净（loss 2.26→0.05），phase 2 一开始 loss 直接 1e21。
原因：resume 时把 .absmax / .quant_state.bitsandbytes__nf4 等 bnb 元数据当成"加载噪声"过滤掉了，结果 bnb Params4bit 反量化时拿不到 absmax，权重全归 0/inf。
修复：删除 strip 逻辑，让 bnb buffers 原样进 state_dict。
毒患 B：Optimizer.load_state_dict = no-op
症状：resume 后 loss 从 0.5 急速爆到 290000，不在第一步，而是第 ~20 step。
原因：optimizer 没恢复 momentum/variance，相当于 cold-start Adam 但 LR 还是 warmed-up 后的值 → 第一波梯度直接把 LoRA 权重打飞。
修复：恢复 load_state_dict，并加 bf16→fp32 promotion wrapper（torch 2.5 strict dtype 要求 fp32 state）。

8.1 根因 A：Server 端 normalization 完全缺失 (Missing Normalizer)

训练时 fastwam_processor.py 用 LinearNormalizer min/max 把 state/action 归一到 [-1, 1]：

scale = 2 / (max - min)
offset = -1 - 2*min/(max-min)
forward(x) = clamp(x * scale + offset, -5, 5)
backward(x) = (x - offset) / scale

Dataset stats (degrees)：joint 2 范围 [-99, +99]，joint 1 范围 [-100, +64] 等。

但原 server_qlora.py 完全跳过 normalize：直接喂 raw degree state，model 看到 50° = 模型期望的 +0.5 但实际是 50（OOD 50×）；输出当 raw delta 用 arm_delta_scale=0.05 hack 凑数。

修复（commit pending）：
  • 启动时 load dataset_stats.json，建 _state_scale/_state_offset + _action_scale/_action_offset
  • predict_action 第 1 步：state_norm = clip(state * scale + offset, -5, 5) 给模型
  • model 输出后：act_phys = (act_norm - offset) / scale
  • 砍掉 arm_delta_scale 和 cumsum hack（model 输出本来就是绝对关节位置，dataset 验证 action.mean ≈ state.mean）

修了之后 eval 仍失败 → normalize 是必要不充分。

8.2 根因 B：joint video+action loss 在小样本上的 pathology (Loss Imbalance Pathology)

Fast-WAM 是 video + action 联合 diffusion training：

# fastwam_idm.py:222
loss_total = self.loss_lambda_video * loss_video + self.loss_lambda_action * loss_action

默认 lambda_video=lambda_action=1.0。但：

Token typeToken count / sampleLoss MSEEffective gradient norm
Video latents33 frames × ~16 token/frame ≈ 5280.2822×
Action latents240.081×

结果：video diffusion 吃掉 LoRA 表达能力，action head 学不到 visual → action 映射。Smoke test 看模型 raw output ±2 in normalized space (训练 target 是 [-1, 1]) → un-normalize 后超出物理可达 joint range → 机器人撞限位停在 "收起" 姿态。

  • 多个 bug 互相掩盖时，loss 在降是假信号。video loss 即使带着崩坏 LoRA 也照样降，不代表 action 在学。
  • Train 和 serve 必须 byte-exact match preprocessing chain。少一个 normalize step 就是 OOD 50×。一次 audit 5 项才发现 4 个 bug。
  • Joint multi-task loss 必须按 token count 重新计算 effective weight。看 lambda=1.0 ≠ 实际 1:1，要看 grad norm。
  • "LoRA 在小样本失败"是过早结论。是 Fast-WAM 的 joint training 特有 pathology，不能推到 OpenVLA / X-VLA。但 "Fast-WAM 5B + 39M LoRA 不适合 50-demo BC"是站得住的结论。
  • 训练 loss 降 ≠ 模型在学：0.0025 训练 loss 看似收敛，但在 normalized 空间是「在训练数据上的 5° per joint memorize」，eval 时直接 extrapolate 80°+ 出训练 range。看泛化必须 offline action MSE on hold-out 或 Isaac Sim eval，不能信训练 loss。
  • 当多个 fix 都加完仍失败，承认架构不适配，不要无穷调参。本次 6+h 调试 / 训练 6h 后转向 OpenVLA，比继续硬撑 r=64 / freeze video 风险低。
  • 所有 patch 都要留原因（why+how to apply），便于 compact 后接续。否则下次重新研究 6h。
