π0.5 微调记录

SO-101 PickOrange · LeIsaac benchmark baseline #5 · 2026-05-18

Physical Intelligence π0.5 (Black et al. 2024) — PaliGemma-2B VLM with a 300M action expert head and flow-matching action generation. ~3.36B total params, our smallest "true" VLA in the leaderboard.

0. TL;DR

π0.5 是 VLA 家族最小成员（3.36B params，PaliGemma 2B vision-language + 300M action expert + flow-matching）。 本地 LoRA fine-tune（仅 ~5M trainable）+ MLX 部署 strict 表现 0/3 round, 0/9 oranges（leaderboard 末位）， 关键问题：action_horizon=50 是 trap，正确值 35（chunk-execution mismatch）。我们的小数据（50 demo）训出来过拟合明显。
🔴 2026-05-24 后续：PyTorch expert-only FT 也失败（leaderboard 末位）
从 MLX 5M LoRA 改成 PyTorch 693M expert-only FT + freeze VLM（GR00T 范式），跑了 10k step + 13 ckpt sweep + ckpt-2000 strict 20-round = 1/60 (1.7%)，仍 leaderboard 末位低 SmolVLA 15×。 根因锁定为 SigLIP@224 vision bottleneck（PaliGemma 硬编码 224，橙子缩后 ≤1 SigLIP patch）。 详见独立报告 pi05_pytorch_expert_ft_negative.html。

读图要点：

  • π0.5 是 VLM + action expert 双脑设计：VLM 跑 1 次缓存 KV，expert 跑 10 次去噪（同 SmolVLA、FastWAM MoT 范式）
  • 真正的"加速 trick"是 VLM 不参与去噪 loop：去噪每步只跑小的 300M action expert，VLM KV 是只读 cache
  • 训练用 flow-matching loss（不是 DDPM），与 SmolVLA / X-VLA / FastWAM 同家族
  • chunk_size=50 是大 chunk，消费 horizon ≠ chunk_size：v1 配 horizon=50 失败，v3 配 horizon=35 才出第一颗橙子（chunk-execution mismatch）
  • state pad 到 32 维：跟 SmolVLA 一致，所有 lerobot policy 默认 max_state_dim=32

2.5 16-bit 重训资源估算 · bf16 retraining resource budget

Cloud GPU memory + disk budget for retraining π0.5 in bf16 (the "16-bit" path, vs MLX 4-bit / int8 quant deployments).

背景： 第一轮 v1-v3 走的是 MLX (Apple Silicon) 自训 + 远端部署，4-bit / int8 量化 + LoRA on M-series； "16-bit 重训" 是切回 CUDA + PyTorch bf16（与 SmolVLA / X-VLA 同栈），便于纯 PyTorch pipeline + auto-eval watcher (scripts/training/lerobot_finetune.sh AUTO_EVAL=1) 集成。

为何 π0.5 16-bit 比 OpenVLA 16-bit 友善一档：
  • π0.5 参数量 3.36B vs OpenVLA 7.6B → base 占用 6.7G vs 15.2G，差 8.5 GB
  • π0.5 chunk=50 action 序列在 expert 这一侧 activations 会比 OpenVLA AR decode 大 ~2 GB，但 base 节省的 8.5 GB 抵消有余
  • 4090 24G 可装 bf16 LoRA（第一轮 MLX 是因 chunk=50 + cross-attn KV cache 没 offload 才 OOM；切 PyTorch grad-ckpt + batch=1 后能装）
  • full FT 仍需 80G 卡 —— 3.36B 的 fp32 AdamW state 单独就 40 GB

3. Eval 参数 + chunk-execution mismatch

🔴 horizon trap：chunk_size=50 训练，但 policy_action_horizon=50 是错的。
  • v1 (horizon=50): 0/3, 0/9 — arm 漂移
  • v2 (horizon=50, 不同 ckpt): 0/3, 0/9 — 同样问题
  • v3 (horizon=35): 0/3, 1 颗落盘 — 终于第一次能抓到
原因：模型训的是 "50 步轨迹"，但部署消费完 50 步后 obs 已经偏离训练分布，最后 15 步是"垃圾外推"。
