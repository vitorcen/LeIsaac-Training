OpenVLA-OFT QLoRA 微调计划

SO-101 PickOrange · 第 6 个 fine-tune baseline · pivot from vanilla OpenVLA · 2026-05-18

Migration plan: vanilla OpenVLA-7B (discrete token, autoregressive) → OpenVLA-OFT (continuous L1, parallel decode, chunked actions). The goal is to break the step_hz=5 ceiling and the gripper-precision floor we hit on vanilla.

0. TL;DR

Pivot 触发条件：vanilla OpenVLA ckpt-2000 GUI 实测显示"机械臂朝目标移动 + 试图抓握 + gripper 差 ~30° 闭不到位"。
Why pivot：（1）step_hz 错配本质 — vanilla 单步 token AR @ 240ms/action 强制 sim step_hz=5，训练数据是 30Hz；（2）discrete bin floor — gripper 在 256-bin 离散化下夹紧区只有 ~10 bins。
How OFT solves both：parallel decoding + chunked action + continuous L1 loss。
Hardware：QLoRA NF4 on 4090（同 vanilla 链路，零工程成本，注意：这是非官方栈，官方用 bf16 LoRA + DDP，4-bit 只用于推理）。
⚠️ Codex 评审 (2026-05-18) 抓到 3 个真 critical（已修订）+ 1 个误报：
  • 🔴 #1 Architecture：OFT 不是"单 forward 出 (8,6)"。实际：chunk_size × action_dim 个 action positions（chunk=8 + 6D = 48 positions），bidirectional attn 算 hidden，独立 MLP head 按 timestep 聚合。仅改 collator shape + server output shape 不够，model contract 必须照官方实现。
  • 🔴 #2 Embodiment gap：LIBERO ckpt 是 ACTION_DIM=7, PROPRIO_DIM=8 写死全局常量。SO-101 是 6D joint + 6D state。action_head 和 proprio_projector 权重 shape 完全不匹配，必须 random-init 重训。"libero-spatial 是最佳起点"的论断不成立 — 最值钱的两个辅助头反而不能继承。
  • 🔴 #3 QLoRA + HF Trainer 是非官方栈：官方 vla-scripts/finetune.py 是自写 DDP + bf16 LoRA，没有训练时 load_in_4bit。我们继承 vanilla 的 QLoRA-Trainer 路径意味着发明一个 QLoRA-OFT，风险 +1。
  • 🟢 #4 Wire compat broken — 已实测推翻：Pi05ServicePolicyClient 是同步阻塞 ZMQ REQ/REP（不是 lerobot async gRPC），server 慢只会让 client 阻塞等，不会 repeat last_action。policy_inference.py:539 用 for i in range(min(horizon, chunk_size)) 按顺序消费 chunk。wire 兼容性 OK。
Time alignment（codex 提醒）：OFT label 是 [action[t], action[t+1], ..., action[t+7]]（current + next 7），不是"未来 8"。Absolute action 的尾部 padding 必须 repeat last valid action（不是 0）。obs_t = (image[t], state[t]) → 监督 8 个连续 action 起点是 t 不是 t+1。

1. 背景 — 为什么 vanilla 收敛慢且抓不住

问题vanilla OpenVLA 实测（ckpt-2000）OFT 是否解决
Inference latency / action200-240 ms（实测）✅ chunk=8 / forward ≈ 30-50 ms
Eval step_hz5 Hz（被 latency 限制）✅ 可恢复 30 Hz
训练/serve 时间尺度30Hz 学 / 5Hz 服务 → 每步 200ms 撑住 33ms 的 target✅ chunk=8 摊薄到对齐
Action 精度（gripper）256 bin × (q01,q99)=(2.4°,58.3°) → 0.22°/bin，夹紧区仅 ~10 bins✅ 连续 L1 回归无量化损失
训练数据稀缺（grasp transition）1-2% 帧⚠️ 同样存在（chunk 摊薄略缓解）

Vanilla failure mode: arm reaches toward target but gripper closes too slowly because each predicted "Δ ≈ 0.5° / 33ms" gets executed for 200ms in sim — 6× time-stretched. By the end of an episode the arm has only made ~300 sim steps instead of the 1800 the training distribution implicitly assumed.

2. OpenVLA-OFT 架构核心差异

维度vanilla OpenVLAOpenVLA-OFT
Action 表示256-bin discrete token连续 float（直接 regression）
Decode 方式autoregressive，每 token 一个 forwardsingle forward 出整个 chunk
LossCross-entropy on action tokensL1 loss on continuous actions
Action headshared lm_head + bin mapping独立的 MLP action expert
Attention pattern (action)causalbidirectional（chunk 内 actions 互相看到）
Chunk size1 step / forward8 steps / forward（论文默认）
实测 latency~240 ms / action论文报 ~30-50 ms / chunk(8 actions) = ~5 ms / effective action

3.1 重新评估 libero-spatial — 经 codex review 校正

之前的"libero-spatial 最佳"结论站不住脚。真正决定迁移收益的不是任务语义，而是 embodiment gap：
  • LIBERO ckpt 的 action_head：(in=hidden, out=7) × 8 chunks = 56 outputs
  • LIBERO ckpt 的 proprio_projector：(in=8, out=hidden)
  • SO-101 我们的：action 6D, state 6D（甚至不一定带 proprio）
结论：action_head 和 proprio_projector 必须 random-init 重训。能继承的只有 backbone + LoRA 先验，那部分是"通用 VLA pick-place 能力"，跟 spatial / object / goal / combined 子任务关系不大。

5.2 关键代码骨架

Action label（dataset.py 改动）：

# Vanilla: action_tokens (6,) = vocab_size - 1 - bin_idx
# OFT:     action_chunk (chunk=8, 6) = continuous normalized actions in [-1, 1]

def __getitem__(self, idx):
    ep = self.episodes_by_idx[idx]
    f0 = self.frame_idx[idx]
    # Sample 8 consecutive frames (clip at episode end)
    f_range = np.arange(f0, min(f0 + 8, ep.from_idx + ep.length))
    pad = 8 - len(f_range)
    actions = self.actions[f_range]                        # (≤8, 6)
    if pad: actions = np.concatenate([actions, np.tile(actions[-1:], (pad, 1))])
    a_norm = normalize_action(actions, self.stats)         # (8, 6) ∈ [-1, 1]
    img = self._video_cache().get(self._video_path(ep)).read_at(ts0)
    return {"image": img, "action_chunk": a_norm, "prompt": self._prompt}

Custom L1 loss in train.py（替换 HF Trainer 默认 CE）：

class OFTTrainer(Trainer):
    def compute_loss(self, model, inputs, return_outputs=False, num_items_in_batch=None):
        # model.forward returns continuous action predictions, shape (B, chunk, 6)
        pred = model(input_ids=..., pixel_values=...).action_pred
        target = inputs["action_chunk"]                    # (B, chunk, 6)
        loss = torch.nn.functional.l1_loss(pred, target)
        return (loss, pred) if return_outputs else loss

Server output（openvla_oft_leisaac/server.py 改动）：

def predict_action(self, front_img, state6, prompt):
    inputs = self.processor(fmt_prompt, img)
    # OFT returns (chunk, 6) continuous floats in [-1, 1], we unnorm
    a_norm = self.inner.predict_action_chunk(**inputs, unnorm_key="leisaac")  # (8, 6)
    a = unnorm(a_norm, self.stats)                         # (8, 6) joint positions
    return a   # shape (8, 6)
# wire format: action.single_arm (8, 5) + action.gripper (8, 1)

6.1 P0c 决策 matrix

选项优缺
A. Fork moojink/openvla-oftmodel contract 官方实现，bidirectional mask 等细节免错要适配 6D + LeRobot v3 parquet + QLoRA + watchdog；moojink 用 DDP/RLDS 跟我们栈差异大
B. 自写 OFT-style chunk training in vanilla scaffold完全复用 vanilla 工程链路（QLoRA, watchdog, dataset adapter, server）要重新实现 bidirectional mask + action slot logic + action_head/proprio_projector，等于重做 OFT 论文
C. 折中：保留 OpenVLA-7b base（vanilla 已熟），自加 chunked L1 head（不要 OFT bidirectional）最小改动，1-day 可上手不是真 OFT，只是 chunk-L1 patch — 拿不到 OFT 论文的 25× 加速，单 forward 仍是 6-token AR

初评：B 是认真做 OFT 的正确路径，但成本是 fork + 重新对齐两套实现细节。A 看似最简单但 LeRobot v3 parquet → RLDS 转换本身就是大坑。C 是"假 OFT，真 chunk-L1"，可能只解决 gripper 精度但不解决 step_hz。决策待 ckpt-5000 vanilla GUI 结果。

7. 风险清单 / Risk register（codex review 后增强版）

风险概率影响缓解
🔴 OFT 架构理解错误（action slots 不是简单 chunk×head）高致命P0a 必须读官方 5 个文件；P0b contract spec 钉死后再写代码
🔴 action_head / proprio_projector shape mismatch确定致命必须 random-init 重训这两个头，pretrained ckpt 只继承 backbone
🔴 自写 QLoRA + HF Trainer 跟 OFT bf16 + DDP 官方栈不同高中承认这是 QLoRA-OFT 新分支；P0c 选 path B 时要计入"重新实现 bidirectional mask"成本
🟡 Time alignment off-by-one（label = current + next 7 不是 future 8）低高collator 严格按 actions[f0:f0+8]，f0 对应当前 obs；tail pad = repeat last
🟡 No-op 帧污染（30Hz 数据里大量小动作）中中训练前过滤 |Δaction| < ε 的连续段，或单独看 gripper 维 MAE 监控
🟢 LIBERO-spatial 先验对 SO-101 适配差（embodiment 双重 gap）中低反正 heads 重训，pretrained 价值有限；最差也是 vanilla 水平
🟢 QLoRA NF4 对新 action_head MLP 兼容低低新加的 MLP 默认 fp32，不进 bnb 量化范围
🟢 wire compat（chunk=8 在 sim 消费）低低已实测推翻 codex 担忧：Pi05ServicePolicyClient 同步 ZMQ blocking，policy_action_horizon=8 直接消费
🟢 训练 crash 模式跟 vanilla 一样高低watchdog + auto-resume + ckpt validity 全套复用
