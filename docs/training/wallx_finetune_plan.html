Wall-X / WALL-OSS 微调可行性

X Square Robot（跨维智能）2025 开源 · SO-101 PickOrange 适配评估 · 2026-06-03

Feasibility study for fine-tuning X Square Robot's WALL-OSS embodied foundation model to the LeIsaac SO-101 single-arm PickOrange task. Verdict: highest-confidence new candidate.

TL;DR：WALL-OSS 是当前性价比最高的新候选，把握偏高（估 40–65%）。三个理由：(1) Qwen2.5-VL native dynamic resolution 根治了 π0.5/OpenVLA 那个"橙子被压成 ≤1 个 patch"的死穴；(2) LeRobot 原生 + 6-DOF 单臂走 customized_robot_config 直接配（仓库自带 dual-arm 例，单臂自写一段）；(3) zero-shot Fruit Sorting 得分 96，与 PickOrange 高度同构。
WALL-OSS is the most pragmatic new candidate (est. 40–65%). Native dynamic resolution cures the small-object vision bottleneck; LeRobot-native, single-arm via customized_robot_config; near-identical Fruit Sorting task scores 96 zero-shot.
必做前置 1 / Must-do #1：默认配置把图像降到 256，务必先把训练/推理分辨率调到 480+ 吃满 640×480 原生帧，否则会步 π0.5 后尘。
必做前置 2 / Must-do #2：4090 全量 FT 塞不下（官方表 40G / 需 ≥48G），但 trainer 自带 freeze_vlm —— 冻结 3.75B VLM、只训 ~0.47B action expert ≈ 19G 显存进得去（GR00T 同款省显存套路）。用 freeze_vlm，不用全量。
✅ 实战跑通记录（2026-06-05）· Shipped & training. 已在单 4090 跑通，底座 wall-oss-0.5，freeze_vlm 只训 0.47B action expert + action_preprocessor，loss 1.1–2.4 健康，0.357s/step，VRAM 实测仅 14.3G/24G（比预估 19G 还省）。4 epoch ≈ 13h，ckpt → outputs/wallx-oss05-leisaac-pick-orange/。脚手架 workspace/leisaac_pick_orange/{config_qact_leisaac.yml, compute_norm_stats_leisaac.py, run_leisaac.sh, run_leisaac_retry.sh}。
踩平的 7 个坑（修复已落地）/ 7 gotchas fixed：
  1. 🔴 env 必须 python 3.11：本机 python 3.10.20 让 import torch ~40% 间歇 segfault（C 栈溢出，纯 Python 稳）；换 3.11 → 0/15 稳。与 torch/CUDA 版本无关。那些 posixpath/sre/pyc 诡异错全是同一堆腐蚀的表象。
  2. ulimit -s 131072：py3.11 后更重的启动（ast 解析/深模型构建）+ 默认 8M 栈仍偶发 → 加大栈 + 重试器 run_leisaac_retry.sh 兜（反复启动到 "iter 1/" 进训练循环=稳态）。
  3. transformers 4.51.3（requirements 的 4.49 是 stale，代码要 AttentionInterface）；flash-attn 用 cp311 预编译 wheel；csrc 编译要 TORCH_CUDA_ARCH_LIST=8.9。
  4. 加载跳过 shape 不符 key（modeling…from_pretrained）：0.5 预训练 action_dim=26，SO-101 是 6 → action_preprocessor.{w1,action_proj_back} 保持 6-DOF 新初始化。
  5. freeze_vlm 额外放行 action_preprocessor 可训（trainer）：它在 moe.experts.1 之外，不放行会随机初始化被冻死。
  6. 数据集用 v2.1 格式：lerobot 0.3.4 是 CODEBASE_VERSION=v2.1，读不了 v3.0 的 tasks.parquet → 用 leisaac-pick-orange_old（同 60 ep）；KEY_MAPPINGS/ACTION_DATASET_NAMES 注册 repo_id。
  7. 调度 num_epoch 100→4 / 每 epoch 存：num_training_steps 只管 LR cosine 长度、不是停止条件，原配置会跑 100 epoch ≈ 320h、每 5 epoch 才存（首 ckpt 16h）。

1. 什么是 Wall-X / WALL-OSS · What it is

WALL-OSS（仓库名 wall-x）是 X Square Robot（x-square-robot / X²，跨维智能） 开源的具身基础模型。LeRobot v0.5.0 已把它列为官方 policy（--policy.type=wall_x），是同一个模型的官方协作移植，不是第三方山寨。

3. 核心：vision 分辨率为什么是把握的关键 · Why resolution is the crux

本任务已被 8 个数据点反复验证：成败的胜负手是 vision 输入分辨率，橙子在 640×480 帧里仅 10–40px。固定 224 的模型把橙子压到 ≤1 个 patch → 几乎不可见 → 失败。

ModelVision encoder输入分辨率橙子缩放后结果
GR00T-N1.7Eagle-2 ViT44822–34 px（1.5–2.4 patch）68% ✅
SmolVLASigLIP51224–40 px（1.7–2.9 patch）25% ✅
WALL-OSSQwen2.5-VL（动态）可吃满 640×480原生不缩，token 充足估 40–65% ✅
π0.5SigLIP22410–17 px（≤1 patch）1.7% ❌
OpenVLASigLIP+DINOv222410–17 px（≤1 patch）0% ❌
⚠️ 唯一的陷阱 / The one trap：WALL-OSS 官方部署 guide 默认把各 view 降到 resolution: 256。如果照抄默认配置，小橙子又会被压回 ≤1 patch，退化成 π0.5。 但因为底座是动态分辨率，只需在 config 把分辨率调到 480+ 即可保住小橙子的 token，不用换 encoder——这正是它相对固定-224 模型的根本优势。这是第一个要验证的实验。

4. SO-101 单臂适配（customized_robot_config 自写）· Single-arm path

⚠️ 事实纠正（2026-06-05 clone 仓库核对）：仓库 workspace/lerobot_example/config_qact.yml 自带的是 aloha_mobile_cabinet（双臂 + 移动底盘）示例，不是开箱即用的 SO-101 单臂。"官方 SO-101 示例"出处是第三方部署 blog，不在本仓库。但 trainer 有 enable_customized_robot_config 机制，单臂自写一段 6-DOF 即可——比 LingBot（仅双臂、需 swap head）容易得多。

SO-101 单臂走 customized_robot_config（仿 workspace/README 的 action_eef:6 / action_gripper:1 写法），我们的 6 维直接对上数据集 action.names：

enable_customized_robot_config: true
customized_robot_config:
  name: "leisaac/pick-orange-so101"
  customized_dof_config:        # = predict_action_keys，sum=6，max≤20
    action_arm: 5               # shoulder_pan/lift, elbow_flex, wrist_flex/roll
    action_gripper: 1
  customized_agent_pos_config:  # = obs state，sum=6
    state_arm: 5
    state_gripper: 1
# 另需在 wall_x/data/utils.py 注册本数据集的 camera key 映射：
#   observation.images.front -> face_view ; observation.images.wrist -> left_wrist_view
  • ✅ 预训练已含单臂数据（x2_normal）—— 不是纯双臂迁移，先验对单臂友好。
  • ✅ action_dim 灵活（flow pad-to-20 再截回 6 / FAST 直接 6 维）。
  • ✅ LeRobot v3.0 dataset 原生 —— 我们的 60-demo PickOrange（so101_follower, 6-DOF, front+wrist@480×640）直接喂。
  • ⚠️ 需先跑 scripts/compute_norm_stats.py 生成 norm_stats.json（维度要与 customized_dof_config 之和 = 6 对齐）。

5. 4090 24GB 可行性 · Feasibility（2026-06-05 实测重写）

结论：全量 FT 塞不下，但 freeze_vlm（只训 0.47B action expert）≈ 19G 进得去——和 GR00T-N1.7 当年塞进 4090 拿 68% 是同一招。之前"🔴 单 4090 放不下"只针对全量 FT，是误导，已修正。

显存账（4.22B 总参，action expert ≈ 0.47B；权重 2B/参、AdamW 状态 fp32 m+v≈8B + master 4B/可训参、bf16 梯度 2B/可训参、bs=1 + grad-ckpt 激活 ~4–5G）：

路线可训参数常驻权重梯度+Adam激活合计4090
全量 FT4.22B 全部8.4 G~33 G~4 G~46 G🔴 官方表 40G，需 ≥48G
freeze_vlm ⭐~0.47B（moe.experts.1）8.4 G~6.6 G~4 G~19 G🟢 进得去（首选）
LoRA（r=8）~20 M adapter8.4 G~0.5 G~5 G~14 G🟢 很宽裕（备选）
推理 server—8.4 G—~2 G~11 G🟢 官方 30ms/chunk

trainer 自带的省显存开关（wall_x/trainer/qwen_vl_act_trainer.py:662，无需改代码）：

freeze_vlm: true            # 冻结一切，只训 name 含 "moe.experts.1." 的 action expert
# 或分离学习率（VLM 也微调一点）：
# action_expert_learning_rate: 0.00005
# train_action_expert_only: true   # 只优化 action expert，VLM 常驻不更新
风险分层 · Risk tiers：
  • 低风险：推理（官方背书单 4090 30ms）、数据接入（LeRobot v3 原生）、显存（freeze_vlm 实测账 ~19G）。
  • 中风险（需实测）：① 默认 256 分辨率必须调到 480+（实验 1）；② 分辨率↑ → Qwen2.5-VL 动态视觉 token↑ → 激活涨，是要 grad-ckpt 的真正原因（不是参数量）；③ 0.47B expert 在 60 demo 上容量够不够（GR00T 冻 backbone 也只训 action head 拿到 68%，先验乐观）。

6. 实施路径 · Implementation

Phase 0 — 现实检查（~1h）

  • clone X-Square-Robot/wall-x，读 S101 single-arm finetuning 示例 config。
  • 确认 LeRobot --policy.type=wall_x 训练入口是否暴露 LoRA / 分辨率 knob。
  • 定位分辨率配置项（resolution: 256 → 改 480/512）。

Phase 1 — 分辨率验证实验（~0.5 天）⭐ 最高优先

  • 用官方 ckpt 做 zero-shot / 极短微调，对比 resolution=256 vs 480 两组，看橙子可见性 + 抓取倾向。
  • 这一步直接决定后面值不值得投全量微调。

Phase 2 — 显存路径定型（已实测，~0.5 天）

  • 定案：freeze_vlm: true + bs=1 + grad-ckpt ≈ 19G，单 4090 直接训，不必云端 / 多卡。LoRA 作备选（更省但 expert 已够小）。

Phase 3 — 微调（~1 天）

训练入口 = accelerate launch train_qact.py --config config_qact.yml（非 lerobot CLI；lerobot --policy.type=wall_x 是另一条推理移植线）。关键 config 字段：

model_type: wall-oss
pretrained_wallx_path: /x-square-robot/wall-oss-flow
use_fast_tokenizer: False          # 训 Flow（flow-matching 连续动作）
freeze_vlm: true                   # ⭐ 只训 0.47B action expert → 19G
FSDP2: false                       # 单卡不用 FSDP
torch_compile: false
batch_size_per_gpu: 1
gradient_accumulation_steps: 32    # 等效 batch 32
data:
  action_horizon: 32
  resolution:                      # ⚠️ 关键：默认 256 全改 480
    face_view: 480
    left_wrist_view: 480
enable_customized_robot_config: true   # SO-101 6-DOF（见 §4）

run.sh 单卡：CUDA_VISIBLE_DEVICES=0，accelerate launch --num_processes=1。

Phase 4 — Server + client（~1h，复用 pi05/gr00t 套路）

  • server/wallx_leisaac/server.py 类比现有 ZMQ server。
  • policy_inference.py 加 --policy_type=wallx（wire chunk=32）。

Phase 5 — Eval（~2h）

  • 按 scripts/benchmark/run_one.sh 横评；horizon 从 chunk*0.5=16 起 sweep。
  • 5-round → 若 ≥SmolVLA(25%) 再跑 strict 20-round 上 leaderboard。

7. 成功率预估 · Forecast

估 40–65%（把握偏高 / high confidence）。
落点逻辑：vision 用动态分辨率吃满 640×480 → 进入 GR00T(448,68%)/SmolVLA(512,25%) 的 work 区上沿；flow-matching action head 与 GR00T 同制式（sample-efficient）；官方单臂示例 + 同构 Fruit Sorting 96 压实工程可行性。下界 40% 反映 60 demo 小数据 + 4090 降配训练的不确定性；上界 65% 若分辨率与多视角配齐，有望逼近 GR00T。
Est. 40–65%. Dynamic-resolution vision puts it in the GR00T/SmolVLA work zone; flow-matching head is sample-efficient; official single-arm example + analogous Fruit-Sorting-96 de-risk the engineering.
